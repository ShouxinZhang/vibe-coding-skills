# 缓存与短期记忆：任务即时性的核心层

## 核心定位

缓存（Cache）与短期记忆（Short-Term Memory）共同构成 Agent 的"即时层"。

- 缓存负责毫秒级复用，目标是降低读取和计算成本。
- 短期记忆负责任务级连续性，目标是保持执行过程不丢上下文。

对业务而言，这一层直接决定系统响应速度、流程连续性和一次任务的交付稳定性。

## 缓存与短期记忆对比

| 维度 | 缓存 | 短期记忆 |
|------|------|---------|
| 目标 | 提速 | 保持任务连续性 |
| 生命周期 | 秒到分钟，进程级 | 分钟到小时，任务级 |
| 存储介质 | 内存为主 | 文件或轻量数据库 |
| 数据形态 | 热点结果、目录快照、中间计算值 | 任务状态、关键决策、失败上下文 |
| 容错能力 | 低，丢失可重建 | 中，丢失会影响恢复效率 |

## 典型读写策略

### 缓存策略

1. 读优先：先查缓存，未命中再回源。
2. 写控制：高频结果写入缓存，关键结果同步写入短期记忆。
3. 淘汰机制：优先采用 TTL + LRU，兼顾时效与命中率。

### 短期记忆策略

1. 任务启动加载：读取最近状态和未完成步骤。
2. 关键事件写入：在计划变更、错误出现、阶段完成时落盘。
3. 压缩与归档：长任务保留关键节点，避免状态日志无限膨胀。

## 推荐最小结构

短期记忆建议至少包含三类记录：

1. `workspace_state`：当前工作区状态与变更摘要。
2. `task_record`：步骤执行轨迹与中间结论。
3. `known_issues`：已识别问题、影响范围、当前处理状态。

## 业务收益与代价

| 项目 | 收益 | 代价 |
|------|------|------|
| 性能 | 降低重复读取和重复推理 | 需要容量和失效策略治理 |
| 稳定性 | 任务可暂停可恢复 | 写入过频会带来 I/O 压力 |
| 可管理性 | 能复盘每次关键决策 | 记录粒度过粗会降低可用性 |

## 落地建议

1. 首阶段先做"轻缓存 + 关键事件短期记忆"，避免一次做全导致复杂度失控。
2. 用可观测指标评估：缓存命中率、任务恢复成功率、平均恢复时间。
3. 将短期记忆与执行状态机绑定，避免出现"有记录但不可恢复"的伪可用设计。
