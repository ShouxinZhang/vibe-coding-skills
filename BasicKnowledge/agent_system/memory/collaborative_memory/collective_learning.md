# 集体学习：把多 Agent 经验沉淀为组织知识

## 核心定位

集体学习（Collective Learning）关注的不是某一个 Agent 变聪明，而是让多个 Agent 在持续协作中，把局部经验转化为组织可复用的知识资产。

在业务系统里，真正稀缺的不是一次成功执行，而是成功背后的方法能否被复用、失败背后的教训能否被避免。如果每个 Agent 都只在自己的上下文里学习，那么系统规模越大，重复试错越多，边际成本也越高。集体学习的目标，是把个体表现升级为组织能力。

## 经验如何沉淀为组织知识

多 Agent 产生的原始材料通常包括步骤记录、异常处理、策略切换、任务结果和人工反馈。它们本身不是知识，只是事件流。只有经过整理，才能进入组织记忆。

建议将沉淀链路拆成四层：

| 层级 | 内容 | 作用 |
|------|------|------|
| 事件层 | 原始执行日志、调用链、错误信息 | 保留证据，支撑审计 |
| 结论层 | 本次任务成功或失败原因、关键决策 | 提炼可解释结论 |
| 规律层 | 哪类场景适合哪种策略 | 形成跨任务复用能力 |
| 资产层 | 模板、规则、检索片段、操作手册 | 直接服务后续业务 |

这个分层很重要。事件层解决发生了什么，规律层和资产层才解决下次怎么更快更稳地做。

## 噪声过滤为什么是前提

多 Agent 系统天然高产出，但高产出不等于高价值。未经治理的协作记录会迅速变成噪声池，反而污染长期记忆。

常见噪声主要有三类：

1. 临时性噪声：一次性报错、偶发网络波动、无复用价值的中间状态。
2. 角色性噪声：某个 Agent 的表达习惯、冗长推理、重复确认语句。
3. 场景性噪声：只在极窄上下文成立、无法泛化的局部技巧。

因此，进入集体学习管道前，至少要经过三道过滤：

1. 去重：合并重复事件和同义结论，避免同一经验被多次放大。
2. 验证：没有结果支撑的感觉有效策略，不进入长期层。
3. 定界：为知识补充适用范围、前置条件和失效边界，避免误召回。

过滤的本质不是删数据，而是保护组织记忆的信噪比。

## 何时蒸馏

蒸馏（Distillation）不应该持续对每条记录进行，否则系统会把短期波动误认为长期规律。更合理的做法，是在证据足够、收益明确、模式稳定时触发。

适合蒸馏的典型时机有：

1. 同类任务连续命中同一解法，说明已经出现稳定模式。
2. 某种失败反复出现，且修复路径基本一致，说明可以固化为防错规则。
3. 某类流程被业务高频调用，人工解释成本明显偏高，说明值得抽象为模板。
4. 上下游规则发生变化，旧记忆开始失效，说明需要做一次替换式蒸馏。

换句话说，蒸馏不是为了压缩记录，而是为了把高频、高价值、高确定性的经验升级为组织默认能力。

## 如何回写长期记忆与 RAG

蒸馏后的内容不应直接覆盖历史，而应分层回写：

1. 回写事实层：稳定规范、接口约束、明确规则，供高置信检索使用。
2. 回写经验层：成功案例、失败模式、策略适配条件，供任务规划和异常处理参考。
3. 回写索引层：把知识切成适合检索的最小片段，并附上标签，如来源、时间、业务域、置信度、适用范围。

RAG 的关键不是多召回，而是准召回。因此回写时必须保留元数据，让系统在检索阶段能做重排和门禁。高风险任务优先注入事实层，高不确定任务才参考经验层，避免旧经验压过新规则。

## 与协同记忆治理的关系

集体学习并不是一个独立后处理环节，它依赖整个协同记忆体系：

1. 没有共享存储，个体经验就无法被汇总。
2. 没有冲突治理，沉淀结果会混入错误结论。
3. 没有记忆管理角色，蒸馏时机和质量就难以稳定控制。

因此，集体学习是协同记忆的结果层，也是长期记忆与 RAG 的上游供给层。

## 业务复利

集体学习带来的最大收益，不是单次回答更漂亮，而是业务复利。

今天某个 Agent 在故障定位中少走的一次弯路，明天可以变成整个团队的默认排查路径；今天一次被验证有效的流程拆解，未来可以持续降低交付成本；今天被准确过滤掉的一批噪声，可以避免后续大量误召回和错误扩散。

最终，组织获得的不是更多 Agent，而是更厚的可调用知识底座。这会直接体现在三个结果上：重复劳动减少、交付稳定性提升、系统迭代速度加快。对企业而言，这种可持续累积的能力，比单次模型性能提升更有长期价值。
