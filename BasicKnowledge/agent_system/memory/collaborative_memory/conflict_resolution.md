# 协同记忆中的冲突解决

## 核心定位

协同记忆解决的不是能不能写入，而是多个执行体同时写入时，系统还能不能保持一致、可解释、可恢复。当多个 Agent、工具进程或人工操作同时更新同一份任务状态、经验记录或偏好配置时，记忆系统会从单机台账升级为协作系统。

对业务而言，冲突解决的价值不在技术优雅，而在避免三类直接损失：错误决策被写成事实、高优先级规则被低优先级内容覆盖、事故发生后无法还原责任链路。

## 冲突从哪里来

并发写入最常见的冲突有四类：

1. 同键覆盖：两个写入者同时修改同一字段，后写入者直接覆盖前者。
2. 语义冲突：字段不同，但表达的业务结论互相矛盾，例如一个记录可重试，另一个记录必须回退。
3. 顺序冲突：事件先后被写反，导致系统把恢复动作当成原因，把原因当成结果。
4. 权限冲突：低可信角色改写高可信结论，例如执行器覆盖审计器的封禁标记。

因此，协同记忆不能只做最后写入生效，而要引入版本化、优先级和仲裁机制。

## 版本化是第一道控制

每次写入都应附带最小版本信息：

- `record_id`：记录主键
- `version`：单调递增版本号
- `writer`：写入者身份
- `base_version`：本次修改基于哪个旧版本
- `timestamp`：时间戳

核心规则是先比版本，再决定是否合并。如果写入请求引用的 `base_version` 已过期，就不能直接覆盖，而应转入冲突队列。这样做的业务意义很直接：把静默丢数据升级为显式发现分歧。

建议优先采用追加事件流加物化视图的模式。原始事件只追加，不原地改历史；当前状态由聚合器计算得到。这样即使仲裁错误，也可以重放修正。

## 优先级与仲裁机制

优先级用于回答谁更有资格写，仲裁机制用于回答意见不一致时怎么定。

建议采用固定优先级：

1. 合规与安全规则
2. 人工审批结论
3. 监控者或治理 Agent
4. 执行器运行结果
5. 推测性草稿或自动补全

当两个写入冲突时，先按优先级过滤；同级冲突再进入仲裁。仲裁不应由写入方自行完成，否则会形成既当运动员又当裁判的结构性风险。

一个可落地的仲裁流程如下：

1. 检查是否违反硬规则。
2. 比较写入来源优先级与可信度。
3. 判断是否可自动合并，例如不同字段补充。
4. 无法自动合并则冻结当前视图，转人工或专门仲裁 Agent。
5. 输出结构化裁决结果，并写回事件流。

## 审计追踪与业务风险

冲突解决最危险的问题不是冲突本身，而是冲突被悄悄解决了。没有审计链路，系统就无法回答是谁改了、为什么改、依据是什么。

| 风险 | 业务后果 | 控制策略 |
|------|---------|---------|
| 高优规则被覆盖 | 错误执行、合规事故 | 强制优先级校验，低优写入不得越权 |
| 冲突静默丢失 | 决策失真，复盘失败 | 过期版本禁止直写，全部进入冲突队列 |
| 仲裁不可解释 | 团队失去信任 | 裁决结果必须记录依据、角色、时间 |
| 历史不可回放 | 故障无法还原 | 采用追加日志，禁止直接删除原事件 |

审计字段至少应包含：`conflict_id`、`related_versions`、`decision`、`decider`、`reason`、`timestamp`。这套最小链路足以支撑事故追责、质量复盘和规则优化。

## 与同步机制的分工

同步机制解决的是怎么把变化传出去，冲突解决处理的是变化相遇后怎么裁决。两者不能混为一谈：

1. 没有同步，冲突会被延后暴露。
2. 没有冲突治理，同步只会更快地扩散错误。
3. 没有审计，系统即使能自动裁决，也无法建立团队信任。

因此，冲突解决应被视为共享记忆的门禁层，而不是附加功能。

## 最小可落地版本（MVP）

第一阶段不需要追求复杂分布式一致性，MVP 先解决看得见冲突、拦得住覆盖、查得到原因：

1. 所有写入带 `base_version` 与 `writer`。
2. 发现版本过期时拒绝直写，并生成冲突事件。
3. 实现一套固定优先级表，不允许越权覆盖。
4. 提供自动仲裁的最小规则：可合并、不可合并、需人工。
5. 保留完整审计轨迹，支持按任务和记录回放。

这套 MVP 的业务价值很明确：它不能消灭全部冲突，但可以把无声的数据污染变成可治理的显式风险，让协同记忆从可用走向可控。
