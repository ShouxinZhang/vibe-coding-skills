# 协同记忆：多 Agent 的共享知识底座

## 模块定位

当系统从单 Agent 扩展到多个 Agent、多个流程、多个周期同时协作时，记忆问题就不再只是"我记住了什么"，而是"团队现在共同依赖什么、相信什么、复用了什么"。

协同记忆（Collaborative Memory）讨论的是多 Agent 场景下的共享知识库、同步机制、冲突治理与集体学习。它解决的核心问题不是存储容量，而是如何让多个自治执行体在不失控的前提下共享经验、同步状态、沉淀组织知识。

对业务而言，这一层决定的是系统是否真正具备组织复利：新 Agent 加入后能否快速接手，多个 Agent 并行后是否会互相污染，上一次任务的经验能否成为下一次任务的默认能力。

---

## 为什么需要独立子模块

现有记忆模块已经解释了缓存、短期记忆、长期记忆、机械记忆与记忆治理，但这些内容主要站在单 Agent 视角。进入多 Agent 协作后，会出现新的结构性问题：

1. 多个 Agent 同时读写同一知识库，容易产生版本分叉与状态冲突。
2. 不同角色的记忆可信度不同，低可信写入可能污染高价值规则。
3. 多个任务并行推进时，需要共享事实，却不能让临时状态误伤长期经验。
4. 个体经验只有被持续蒸馏和回写，才能形成组织级知识资产。

因此，协同记忆需要从单独的工程与治理视角展开，而不能只是现有 memory 模块的附属补充。

## 协同记忆的演进阶段

**阶段一：记忆孤岛**。每个 Agent 只管理自己的上下文和局部状态，切换角色或接力执行时需要重复解释。

**阶段二：共享状态层**。开始共享任务进度、已知问题和交接信息，降低多人协作的断层成本。

**阶段三：共享知识库与冲突治理**。共享范围扩展到规则、经验和偏好，系统需要引入同步机制、版本控制和冲突仲裁。

**阶段四：集体学习闭环**。多个 Agent 的经验被持续蒸馏并回写到长期记忆和 RAG，形成组织可复用能力。
