# 共享存储与同步：把团队经验变成可复用产能

## 核心定位

当系统从单 Agent 走向多 Agent、多流程、多周期协作后，记忆就不再只是"我记得什么"，而变成"团队此刻共同相信什么、复用了什么、更新了什么"。

共享知识库的业务价值，不是追求一个很大的库，而是让规范、经验、偏好和当前状态能够被多个执行单元稳定复用。这样做的直接收益是三点：减少重复试错，缩短交接成本，降低同一问题被不同 Agent 反复做出不同答案的概率。

## 共享知识库分层

建议把共享存储拆成四层，而不是把所有内容堆在一个库里：

| 层级 | 内容 | 业务作用 |
|------|------|---------|
| 规则层 | 业务规则、接口约束、合规边界 | 保证输出不越线 |
| 经验层 | 成功案例、失败原因、推荐策略 | 缩短定位与决策时间 |
| 状态层 | 当前任务进度、锁定资源、交接信息 | 支撑多人接力与恢复 |
| 偏好层 | 用户偏好、组织风格、风险容忍度 | 提升结果可接受度 |

分层的目的不是为了让架构看起来整齐，而是为了控制写入污染。规则层应当最稳定，经验层允许逐步增长，状态层强调时效，偏好层强调个体适配。这样才能避免临时状态覆盖长期规则，或者旧经验误导当前执行。

## 写入策略

共享库最怕的是谁都能写，最后没人敢信。因此写入要遵循最小原则：

1. 关键事件才写入，例如规则变更、任务结论、失败原因、交接节点。
2. 先写事实，再写判断。事实可复查，判断要带来源与时间。
3. 不直接覆盖旧内容，高价值变更优先采用追加记录加版本标记。
4. 每条共享记忆至少带上来源、时间、适用范围、可信度。

从业务角度看，这套策略的意义是让知识库成为可审计资产，而不是另一个会制造争议的聊天记录。

## 读取与同步策略

读取不应默认全量拿来，而应遵循按任务召回：

1. 先读规则层，确保不违反硬约束。
2. 再读与当前目标最相关的经验层内容。
3. 进入协作或交接场景时补读状态层。
4. 输出前按需注入偏好层，修正表达与取舍。

同步也不应该追求所有副本时时完全一致。对业务更重要的是：关键规则尽快统一，临时状态允许短时间差异，最终都能回到同一可信记录。换句话说，影响合规和决策边界的内容要优先同步，影响局部效率的内容可以延后同步。

## 一致性与可用性的权衡

共享记忆不是越严格一致越好。若每次写入都要求所有协作方立即确认，系统会变慢，交付会卡住；若完全放松一致性，又会出现多人基于不同版本做决策，导致返工。

更实用的做法是分级处理：

1. 规则层优先一致性。宁可慢一点，也不能多个 Agent 按不同规范执行。
2. 经验层优先可用性。允许先被读取，再通过反馈校正价值。
3. 状态层优先时效性。短时间不一致可以接受，但必须能快速收敛。
4. 偏好层优先最近有效。过旧偏好要能自动降权。

这不是在谈抽象术语，而是在控制业务损失：规则错了会出事故，经验旧了最多降低效率，状态不同步会影响交接，偏好过时则影响体验。

## 与现有记忆层的关系

协同记忆不是替代单 Agent 记忆，而是在其之上建立共享层：

1. 缓存与短期记忆继续服务单次执行效率。
2. 机械记忆负责记录单任务中的 todo、plan、draft 和检查点。
3. 协同记忆负责把跨 Agent 需要共享的部分抽出来，形成团队可复用资产。
4. 长期记忆与 RAG 则负责把高价值共享经验进一步沉淀和召回。

因此，协同记忆更像一个桥梁：一头连接执行现场，一头连接长期知识资产。

## 最小可落地版本（MVP）

一个真正能落地的版本，不需要复杂中台，先做到四件事即可：

1. 用一个共享存储保存四层数据，先支持结构化字段与简单标签。
2. 写入只开放给少数治理角色，普通执行器只能提交候选记录。
3. 读取采用规则优先、经验补充、状态按需的固定顺序。
4. 同步先实现两类机制：关键规则立即刷新，经验与状态定时合并。

只要完成这个 MVP，系统就会从每个 Agent 各记各的，升级为团队经验开始沉淀并被稳定复用。后续再增加冲突检测、质量评分和自动归档，才有意义。否则共享知识库只会越做越大，却没有真正带来交付速度、稳定性和组织复利。
