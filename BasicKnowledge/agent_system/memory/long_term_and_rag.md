# 长期记忆与 RAG：把经验变成可调用资产

## 核心定位

长期记忆（Long-Term Memory）解决的是跨任务持续学习问题，RAG（Retrieval-Augmented Generation）解决的是"需要时把对的知识带回现场"的问题。

两者结合后，Agent 才能真正形成组织级复利：今天的经验，成为明天的默认能力。

## 长期记忆分层

建议将长期记忆拆成三个层次：

| 层级 | 内容 | 业务价值 |
|------|------|---------|
| 事实层 | 稳定事实、规范、接口契约 | 降低低级错误，提升一致性 |
| 经验层 | 成功/失败案例、策略效果 | 缩短问题定位时间 |
| 偏好层 | 用户偏好、风格约束、风险偏好 | 提升个性化与可接受度 |

## RAG 的最小闭环

一个可落地的 RAG 流程应至少包含四步：

1. 召回（Recall）：根据任务意图检索候选记忆。
2. 重排（Rerank）：按相关性、时效性、可信度排序。
3. 注入（Inject）：将高价值片段放入当前上下文。
4. 反馈（Feedback）：记录命中效果，反向优化索引与权重。

## 召回质量风险

RAG 的主要风险不在"检索不到"，而在"检索到错误但看起来合理"。这会导致系统自信地做错事。

常见风险包括：

1. 误召回：语义相近但约束不同的历史案例被注入。
2. 过期知识：旧规则覆盖新规则，导致策略倒退。
3. 冗余注入：上下文里塞入过多低价值材料，稀释关键信息。

## 控制策略

1. 给记忆打标签：来源、时间、适用范围、置信度。
2. 对高风险任务启用双门禁：相关性阈值 + 规则白名单。
3. 对注入内容做预算控制，只保留"最小必要证据"。
4. 记录每次 RAG 的命中效果，用在线反馈做权重校准。

## 业务视角下的收益

| 目标 | 作用 |
|------|------|
| 降本 | 减少重复推理和重复人工解释 |
| 提效 | 历史解法可直接复用，缩短交付周期 |
| 控险 | 用可追踪证据支撑决策，提升审计可解释性 |

## 实施建议

1. 先建设小而准的长期记忆库，不追求一开始就覆盖全部历史。
2. 优先接入高复用场景（例如故障处理、代码改造、流程规划）。
3. 把 RAG 当作"决策增强"而不是"真理来源"，始终保留验证步骤。
