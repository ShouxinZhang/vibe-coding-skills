# 机械记忆：让执行过程可追踪、可恢复、可复用

## 为什么叫机械记忆

机械记忆（Mechanical Memory）指的是不依赖模型临场发挥、而是通过结构化状态明确保存的执行信息。它不是知识库，而是 Agent 的工作台账。

核心目的是把"做过什么、做到哪一步、下一步做什么"变成稳定的数据对象，降低长流程任务中的断点损失。

## 三类核心对象

| 对象 | 定义 | 作用 |
|------|------|------|
| todo | 待完成事项（目标级） | 统一需求入口，避免目标漂移 |
| plan | 可执行计划（步骤级） | 将目标拆解为可交付路径 |
| draft | 草稿与中间想法（探索级） | 保存备选方案与推导过程 |

## 推荐状态流转

一个简化且可落地的状态流转如下：

1. `todo -> draft`：先把需求翻译成可讨论的草案。
2. `draft -> plan`：通过评审后转为可执行步骤。
3. `plan -> executing`：执行器按步骤推进并持续写回状态。
4. `executing -> completed|failed`：结束后保留结果与关键原因。
5. `failed -> draft`：失败回到草案层，避免直接重复原错误。

## 与执行器协同的关键点

1. 计划领取时写入 `owner` 与 `start_time`，防止并发重复执行。
2. 每步执行后写入 `checkpoint`，确保中断后可以续跑。
3. 关键决策必须记录 `reason`，便于复盘与责任追踪。
4. 失败事件至少包含：错误类型、影响范围、恢复建议。

## 失败恢复策略（最小版本）

1. 轻失败（临时网络、工具超时）：原步骤重试。
2. 中失败（局部逻辑错误）：回到最近检查点继续。
3. 重失败（计划本身有缺陷）：回退到 `draft` 重写方案。

## 最小可落地版本（MVP）

第一阶段可先实现以下能力：

1. todo/plan/draft 三对象存储。
2. 计划步骤状态与检查点写回。
3. 失败自动分类为"重试/回退草案"两种处理。
4. 一份可检索的执行历史，用于后续经验沉淀。

这套 MVP 足以让 Agent 从"不可恢复流程"升级为"可管理执行系统"。
