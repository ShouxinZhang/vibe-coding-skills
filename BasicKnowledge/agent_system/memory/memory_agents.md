# 记忆管理 Agent：记录者与监控者

## 模块定位

当记忆规模扩大后，单靠执行器顺带写日志会出现质量不稳定、延迟高、冲突难排查等问题。记忆管理 Agent 的目标是把记忆治理职责独立出来。

系统至少需要两个角色：

1. 记录者 Agent（Recorder）：负责记忆写入与归档。
2. 监控者 Agent（Monitor）：负责健康检查与策略触发。

## 职责边界

| 能力 | Recorder | Monitor |
|------|---------|---------|
| 事件落盘 | 主责 | 旁路 |
| 状态完整性校验 | 主责 | 协助 |
| 异常检测 | 旁路 | 主责 |
| 恢复策略触发 | 记录执行结果 | 主责 |
| 指标与告警 | 提供原始数据 | 主责分析 |

边界原则：Recorder 不做复杂决策，Monitor 不直接改写历史记录。

## 触发时机

### Recorder 触发

1. 步骤开始与结束。
2. 计划变更与人格切换。
3. 错误发生与恢复完成。

### Monitor 触发

1. 定时巡检（超时、失败率、记忆膨胀）。
2. 高风险事件即时触发（连续失败、规则冲突）。
3. 阶段复盘触发（任务完成后生成优化建议）。

## 冲突处理策略

常见冲突是"执行器建议重试"但"监控者建议回退"。建议采用统一优先级：

1. 明确业务规则优先。
2. 安全与合规优先。
3. 局部效率优化最后考虑。

同时要求所有冲突都写入同一事件流，避免"决策发生了但没有证据"。

## 最小事件协议

建议采用最小字段集：

- `event_id`：事件唯一标识。
- `event_type`：如 `step.completed`、`plan.failed`、`recovery.triggered`。
- `source`：来源角色。
- `task_id` 与 `plan_id`：关联主键。
- `timestamp`：时间。
- `payload`：业务数据。

这套协议足够支撑审计、重放和故障排查。

## 最小可落地版本（MVP）

1. Recorder 只做追加写入和基础校验。
2. Monitor 只实现三类规则：超时、连续失败、状态缺失。
3. 恢复动作先限定为：重试、回退检查点、回退草案。
4. 每次恢复都要求产生结构化结果，回写到机械记忆。

完成该 MVP 后，系统会从"出了问题靠人工盯"升级为"可自动监测与半自动恢复"。
