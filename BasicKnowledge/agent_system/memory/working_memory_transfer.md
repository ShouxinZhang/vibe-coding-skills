# 工作记忆迁移：让暂停后的工作状态可恢复

## 核心定位

工作记忆迁移（Working Memory Transfer）解决的不是“有没有把历史存下来”，而是“任务暂停、切换、再回来时，能不能快速恢复工作态”。
对业务来说，昂贵的不是一次中断，而是中断后重新理解背景、重新建立判断框架、重新找到下一步动作。迁移机制的价值，是把这些成本压缩成一次可控的预热过程。

## 为什么单靠缓存和机械记忆还不够

[cache_and_short_term.md](./cache_and_short_term.md) 解决的是即时连续性，[mechanical_memory.md](./mechanical_memory.md) 解决的是结构化状态保存，但两者之间还缺一层“恢复机制”。
如果只有缓存，任务一旦冷却就容易失去上下文；如果只有机械记忆，系统虽然知道“做到哪了”，却不一定能重新进入可执行状态。
真正长期运行的系统，还必须回答：旧工作态如何重新升温。

## 最小迁移链路

一个可落地的迁移链路至少包含五层：

| 层级 | 作用 | 如果缺失会怎样 |
|------|------|----------------|
| 热层 | 保持当前任务的即时上下文与注意力焦点 | 一切切换都会变成中断 |
| 机械层 | 固定 todo、plan、draft、checkpoint 等结构状态 | 恢复只能靠猜 |
| 冷层 | 保存可复用但暂不活跃的工作记忆 | 历史经验难以跨周期复用 |
| 检索层 | 按任务、阶段、时间、风险快速定位记忆 | 恢复路径过长，成本线性膨胀 |
| 预热层 | 把召回内容重建成可执行工作态 | 找回信息却接不上执行 |

这条链路的关键不是存更多，而是把“恢复所需的最小闭环”保留下来。

## 什么叫无损迁移

无损迁移不是零损耗保存全部细节，而是关键状态不丢。
至少要保住六类信息：
1. 当前目标是什么。
2. 已经推进到哪一步。
3. 当时为什么这样决策。
4. 哪些风险和约束仍然有效。
5. 下一步动作从哪里接。
6. 失败时应该回退到哪个检查点。
只要这些状态仍然清楚，大量细节都可以在恢复后按需补回。

## 跳级检索与预热

任务恢复时，最差的策略是重读完整历史。更合理的方式是跳级检索（Skip-level Retrieval）：先命中最近一次可靠状态，再补齐最少必要上下文。
预热（Warm-up）不是简单加载文本，而是重建工作态。它至少要完成三件事：
1. 重新确认当前任务意图。
2. 召回与该意图最相关的结构状态和关键证据。
3. 把“下一步动作”重新挂回执行器。
所以，恢复本质上是状态重建，不是资料回放。

## 治理边界

迁移机制最大的风险，不是忘记，而是错误复活。
过期策略、相似但不适用的旧案例、已经失效的工作假设，都可能在恢复时被错误激活。为此，冷层记忆至少要带有时间、来源、适用范围和置信度；恢复时要有预热门槛，而不是“检索到就注入”。
监控侧不应只看“恢复成功几次”，还要看召回命中率、预热成功率、恢复后稳定时长和错误复活比例。没有这组指标，系统只是在反复修补，而不是形成可持续复利。

## 业务价值

工作记忆迁移带来三类直接收益：
1. 降低跨天、跨任务、跨角色的恢复成本。
2. 减少中断后的重复阅读、重复推理和重复返工。
3. 让长期流程从“靠人硬记”升级成“可恢复、可审计、可交接”的连续系统。
因此，工作记忆迁移不是记忆模块的边角能力，而是长期运行 Agent 的运行时基础设施。
