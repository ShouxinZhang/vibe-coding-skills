# 编程：多智能体软件开发系统

## 一、为什么用多 Agent 写代码

单 Agent 编程的局限性是结构性的：上下文窗口有限，无法同时理解大型代码库的全貌；角色单一，既当架构师又当测试员，质量无法自洽；缺乏外部反馈，错误在生成后得不到及时纠正。

软件开发本身就是多角色协作的过程——架构师设计系统、开发者实现功能、测试者验证正确性、审查者把关质量。多 Agent 系统天然映射到软件开发的团队结构，将角色分离的人类实践转化为 Agent 间的结构化协作。

---

## 二、典型角色设计

| 角色 | 输入 | 输出 | 质量标准 |
|------|------|------|---------|
| **产品经理 Agent** | 原始需求描述 | 结构化需求文档、用户故事 | 需求完整、无歧义、可验证 |
| **架构师 Agent** | 需求文档 | 系统设计、模块划分、接口定义 | 模块解耦、扩展性、技术选型合理 |
| **开发者 Agent** | 设计文档 + 接口契约 | 代码实现 | 功能正确、风格一致、通过测试 |
| **测试者 Agent** | 代码 + 需求 | 测试用例、测试报告 | 覆盖率、边界条件、回归测试 |
| **审查者 Agent** | 代码 + 设计 | 审查意见、改进建议 | 代码质量、安全性、可维护性 |

角色之间形成链式依赖：产品经理 → 架构师 → 开发者 → 测试者 → 审查者。每个环节的输出是下一个环节的输入，任何一环的质量下降都会被下游放大。

---

## 三、系统架构

### 3.1 瀑布式流水线

```
需求 → 设计 → 编码 → 测试 → 审查 → 交付
```

最直觉的架构：各角色按序执行，前一步完成后才启动下一步。

**优点**：职责清晰，中间产物有明确的文档形式，易于审计。

**缺点**：反馈延迟大。编码阶段发现设计缺陷时，需要回退到架构师；测试发现 Bug 时，需要回退到开发者。在 Agent 系统中，这种回退的成本是重新生成——可能需要多轮迭代。

### 3.2 迭代式

```
编码 ⇄ 测试 ⇄ 修复（循环直到通过）
```

核心是**编码-测试-修复循环**。开发者 Agent 生成代码后，测试者 Agent 立即运行测试；如果失败，测试报告回传给开发者 Agent 修复；修复后再测试——循环直到所有测试通过。

**优点**：快速收敛，适合局部修复。

**缺点**：只能修复测试能发现的问题，架构级缺陷无法在这个循环内解决。

### 3.3 混合式

```
整体：需求 → 设计 → [编码 ⇄ 测试 ⇄ 修复] → 审查 → 交付
```

整体瀑布 + 局部迭代。设计完成后进入编码-测试循环，循环结束后由审查者做最终把关。这是目前大多数多 Agent 编程系统采用的架构。

### 3.4 通信协议

Agent 间的消息不是自由文本，而是**结构化工件（Artifact）**：

- **需求文档**：Markdown 格式的用户故事和验收标准
- **设计文档**：模块图、接口定义、数据模型
- **代码块**：带文件路径和行号的代码片段
- **测试报告**：通过/失败、错误堆栈、覆盖率统计
- **审查意见**：定位到具体代码行的评论

结构化通信是多 Agent 编程系统优于自由对话系统的关键——它减少了歧义，使每个 Agent 能精确定位上下文。

## 四、核心挑战

### 4.1 一致性（Consistency）

多个 Agent 生成的代码风格和架构决策可能冲突。Agent A 用了回调模式，Agent B 选择了 async/await，集成时互不兼容。

**对策**：
- 架构师 Agent 在设计阶段定义统一的编码规范和模式约束
- 所有代码通过共享的 Linter 和格式化工具强制一致
- 审查者 Agent 在最终环节检查架构一致性

### 4.2 上下文传递

代码量超过单个 Agent 的上下文窗口时，Agent 无法看到完整的代码库。

**对策**：
- **摘要**：将大文件压缩为接口签名 + 关键逻辑摘要
- **代码骨架（Skeleton）**：只保留类定义、方法签名、类型注解，省略实现细节
- **接口契约**：Agent 之间通过接口定义（而非实现）传递依赖信息
- **检索增强**：使用代码索引，Agent 按需检索相关代码片段

### 4.3 幻觉与错误传播

一个 Agent 生成了错误的代码或文档，下游 Agent 在此基础上继续构建，错误被继承和放大。

**对策**：
- 每个环节设置**门禁（Gate）**：输出必须通过验证才能传递给下游
- 测试者 Agent 独立于开发者 Agent，不假设代码正确
- 审查者 Agent 同时检查代码和设计文档的一致性

### 4.4 集成困难

各 Agent 独立生成的模块在集成时可能不兼容——接口不匹配、数据格式冲突、隐式依赖未声明。

**对策**：
- 架构师 Agent 在设计阶段定义清晰的接口契约
- 集成测试作为独立阶段，在模块测试通过后执行
- 使用依赖注入和接口抽象降低模块间耦合

### 4.5 评估标准

| 指标 | 含义 | 衡量方式 |
|------|------|---------|
| 功能正确性 | 代码是否实现了需求 | 测试通过率 |
| 代码质量 | 可读性、可维护性、安全性 | 静态分析 + 审查评分 |
| 开发效率 | 完成任务的时间和迭代次数 | 轮次计数、Token 消耗 |
| 一致性 | 多模块间的风格和架构统一 | 自动化检查 + 人工审计 |

## 五、代表性系统

| 系统 | 架构风格 | Agent 数量 | 通信方式 | 适用场景 |
|------|---------|-----------|---------|---------|
| **MetaGPT** | SOP 驱动 | 5–8 | 标准化文档（PRD、设计稿、代码） | 从需求到代码的全流程 |
| **ChatDev** | 对话驱动 | 4–6 | 自然语言对话 | 小型项目快速原型 |
| **AutoGen** | 对话框架 | 灵活 | 自然语言 + 代码执行 | 通用多 Agent 任务 |
| **Devin/OpenHands** | 全栈自主 | 1（内嵌多角色） | 内部推理链 | 端到端代码修改 |

### MetaGPT

MetaGPT 的核心创新是用**标准化操作流程（SOP）** 驱动 Agent 间通信。产品经理输出 PRD 文档，架构师输出系统设计文档，开发者根据文档编码。每个阶段的输出格式是固定的，减少了自由文本的歧义性。

### ChatDev

ChatDev 模拟一个虚拟软件公司，Agent 之间通过对话协作。CEO Agent 与 CTO Agent 讨论技术方案，CTO 与程序员 Agent 讨论实现细节。对话驱动的方式灵活但容易话题漂移。

### AutoGen

微软的通用多 Agent 对话框架，不限定角色和流程。开发者自行定义 Agent 角色、通信拓扑和终止条件，适合构建定制化的多 Agent 系统。

## 六、设计模式与最佳实践

### 6.1 代码审查-修复循环

有效的反馈闭环设计：

```
开发者 → 代码 → 审查者 → 审查意见 → 开发者 → 修复后代码 → 审查者 → 通过/继续
```

关键点：
- 审查意见必须**定位到具体代码行**，而非泛泛评论
- 每轮审查只聚焦**最严重的 3–5 个问题**，避免信息过载
- 设置最大迭代轮数（通常 3 轮），避免无限循环

### 6.2 测试驱动的 Agent 开发

先让测试者 Agent 根据需求生成测试用例，再让开发者 Agent 实现通过这些测试的代码。

优势：
- 测试用例作为明确的验收标准，减少歧义
- 开发者 Agent 有具体的目标函数（通过测试），而非模糊的"实现需求"
- 自然形成迭代闭环

### 6.3 上下文压缩策略

处理大型代码库时的分层压缩：

| 层级 | 内容 | 适用场景 |
|------|------|---------|
| 全量代码 | 完整源码 | 小文件（< 200 行） |
| 代码骨架 | 类定义 + 方法签名 + docstring | 中等文件（200–1000 行） |
| 接口契约 | 类型定义 + 公开 API | 大型模块 |
| 自然语言摘要 | 模块功能描述 | 跨模块理解 |

### 6.4 人类干预点

在哪些节点引入人类审查最有效：

- **需求确认**：产品经理 Agent 输出需求文档后，人类确认理解正确
- **设计审查**：架构决策对系统有长期影响，值得人类把关
- **最终交付**：代码通过所有自动化检查后，人类做最后审阅

中间环节（编码-测试-修复循环）通常不需要人类介入，由 Agent 自主完成。

## 七、与 Agent Workflow 的关系

编程 Agent 系统是 [agent_workflow](../../agent_workflow/introduction.md) 理念在软件开发领域的具象化。但存在关键区别：

- **agent_workflow** 侧重**单 Agent 流水线**：一个 Agent 端到端完成任务，强调 Gate、Hooks、日志等流程控制
- **编程多 Agent 系统**侧重**角色分工与协作**：多个 Agent 各司其职，强调通信协议、工件传递、质量反馈循环

两者可以嵌套：每个角色 Agent 内部运行自己的 agent_workflow（接收任务 → 分解 → 执行 → 验证 → 输出），而多个 Agent 之间的协作构成更高层级的系统架构。
