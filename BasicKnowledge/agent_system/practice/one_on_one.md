# 1v1：双智能体交互系统

## 一、场景特征

两个 Agent（智能体）之间的直接交互是最小化的多智能体系统形态。双方通过同步通信、交替发言的方式展开协作或对抗。这种模式下每一轮对话都受到严格的上下文窗口限制，要求系统设计必须在有限的历史记录中维持一致的角色设定和推理链条。

典型应用场景包括：辩论（Debate）、谈判（Negotiation）、面试模拟、教学对话、代码审查（Code Review）。每种应用对通信协议、终止条件、评判机制的要求差异显著。

---

## 二、交互模式分类

| 模式 | 特征 | 通信协议 | 目标 | 典型案例 |
|------|------|---------|------|---------|
| **合作式** | 目标一致，信息对称 | 同步响应，渐进补充 | 共同完成任务 | 教师-学生、面试官-候选人 |
| **对抗式** | 目标互斥，零和胜负 | 轮流发言，完整陈述 | 一方获胜 | 辩论、博弈 |
| **协商式** | 目标部分重叠，各有保留 | 信息逐步披露，让步与反馈 | 达成可接受的协议 | 商务谈判、资源分配 |

不同模式下的通信策略差异：合作式强调上下文累积和细节补充；对抗式需要完整的逻辑链和防守反击；协商式涉及信号博弈（Signaling Game），一方可能隐瞒私有信息（Private Information）。

## 三、系统架构

### 3.1 核心模块

```
┌─────────────────────────────────┐
│     轮次管理（Turn Manager）      │
│  - 当前轮数、最大轮数限制         │
│  - 谁发言、谁等待               │
└────────┬────────────────────────┘
         │
    ┌────┴───────┐
    │            │
┌───▼─────┐  ┌──▼──────┐
│ Agent A │  │ Agent B  │
│ 角色:X  │  │ 角色:Y   │
└───┬─────┘  └──┬──────┘
    │           │
┌───┴───────────┴───┐
│ 上下文窗口管理      │
│ - 历史消息缓冲      │
│ - Token 计数       │
│ - 压缩策略         │
└───────────────────┘
         │
    ┌────┴────────┐
    │             │
┌──▼────┐   ┌───▼──┐
│提示工程│   │评判者 │
│角色定义│   │终止条件│
└────────┘   └──────┘
```

### 3.2 轮次管理

| 环节 | 具体操作 | 检查项 |
|------|---------|--------|
| **初始化** | 两个 Agent 加载各自的系统提示 | 角色定义清晰性 |
| **轮次操作** | Agent A 生成回复 → 加入历史 → Agent B 读取 → 生成回复 | 输出长度、响应时间、一致性 |
| **终止检查** | 达到最大轮数？双方共识？评判者裁决？ | 何时停止、是否强制退出 |

### 3.3 上下文窗口与历史管理

对话历史容易超限。标准做法是按时间逆序保留最新的、关键的消息，舍弃冗余的中间过程。对于长对话（20+ 轮），可采用两层策略：
- **第一层**：保留完整的系统提示 + 最近 5 轮对话 + 重要的阶段性总结
- **第二层**：定期生成"中间报告"，记录当前共识点、分歧点、下一步方向

## 四、核心挑战

### 4.1 一致性维护（Consistency Preservation）

长对话中 Agent 容易遗忘早期的表述或自相矛盾。解决方法：
- 在系统提示中明确列出不可更改的立场与约束
- 每隔 3–5 轮要求 Agent 输出一条"立场声明"，作为检查点
- 使用向量数据库存储关键陈述，每轮查询一次"我之前说过什么"

### 4.2 策略深度不足（Shallow Strategy）

简单的 LLM Agent 容易陷入"顶层回应"——仅对最后一句话做出反应，而不进行多步推理。防止方法：
- 在提示中明确要求"列出 3 个可能的策略，选择最优的一个"
- 要求输出"假设对方的反驳是 X，我的预案是 Y"
- 定期生成博弈树或决策链，显式化推理过程

### 4.3 评估困难（Evaluation Difficulty）

何为"更好的回复"或"谁赢了"？定量评估困难。常见方案：
- **LLM-as-Judge**：第三个模型评判哪一方更有理有据
- **规则评分**：按逻辑严密性、论证覆盖度、对对方论点的回应充分度打分
- **基准集**：预先标注优秀回复，微调评判模型

### 4.4 循环对话陷阱（Loop Trap）

两个 Agent 陷入重复轮回，互不让步。特征：同一论点反复出现，对话无进展。
- 检测：连续 3 轮相似度 > 0.85，触发告警
- 打破：引入"提议新角度"的强制指令，或由第三方（仲裁者）插入新信息片段

## 五、博弈论视角

### 5.1 博弈分类

| 博弈类型 | 特征 | 应用 | 解 |
|---------|------|------|-----|
| **零和博弈** | 一方所得即他方所失 | 辩论、竞争性编码 | 纳什均衡（Nash Equilibrium） |
| **非零和博弈** | 合作可提升双方收益 | 谈判、协作 | 帕累托最优（Pareto Optimality） |

### 5.2 谈判中的纳什谈判解

在两人谈判中，若双方都是理性的，最稳定的协议应满足纳什谈判解（Nash Bargaining Solution）：

$$
\max_{(x_1, x_2)} (u_1(x_1) - d_1) \cdot (u_2(x_2) - d_2)
$$

其中 $u_i$ 是参与者 $i$ 的效用函数，$d_i$ 是保留效用（Disagreement Point）。直观理解：在双方都能接受的方案中，选择使"双方收益乘积"最大的方案。

### 5.3 信号博弈（Signaling Game）

一方拥有私有信息（Type），另一方不知道。拥有私有信息的一方通过行动（Signal）试图披露或隐瞒其类型。例如：谈判中一方假装对某个条款"无所谓"（实则很在意），试图让对方让步。

防守策略：要求对方"额外证明"关键主张，或设计激励机制使其主动真实披露。

## 六、典型实现

### 6.1 AI Safety via Debate

Irving et al. 2018 提出的框架：两个 Agent 辩论某个问题的答案，第三方 Judge 模型评判谁说得更有理。优势：
- 通过对抗性思考发现原始模型的错误
- 将复杂判断分解为"哪个论据更有说服力"的简单问题

### 6.2 Constitutional AI 中的自我对弈

模型与自己对弈，一方生成候选回复，另一方评判是否违反"宪法"（Constitution，即一组预定价值观）。通过自我对话不断优化。

### 6.3 LLM-as-Judge 模式

用第三个 LLM 评判两个 Agent 的对话质量。核心问题：Judge 的公正性与稳定性。改进方法：
- 多个 Judge 投票取多数
- 定期用人工标注数据验证 Judge 的评判与人类评价的一致性
- 对 Judge 的提示工程做 A/B 测试

## 七、设计建议

### 7.1 防止循环对话

| 技术手段 | 实施细节 | 成本 |
|---------|---------|------|
| **相似度检测** | 计算连续 N 轮回复的向量相似度，阈值 0.80 以上触发告警 | 低 |
| **强制创新指令** | 在系统提示中添加"提出之前未提及的新角度或证据" | 低 |
| **引入随机扰动** | 偶尔注入"新信息片段"打破僵局 | 中 |
| **第三方仲裁** | Judge 识别僵局并强制切换话题 | 高 |

### 7.2 角色提示的鲁棒性设计

- **显式边界**：逐一列出"你不能做 X""你必须信守 Y"，避免隐式假设
- **例举法**：提供 3–5 个好回复的例子，模型能更好地泛化
- **反向清单**：列举"常见的失败回复"，告诉模型不要这样做
- **递进式约束**：优先级排序，在冲突时明确哪条约束优先

### 7.3 何时引入仲裁者

**引入条件**：
- 对话持续超过 15 轮且无明显进展
- 两方陷入"各执一词"，难以达成共识
- 需要外部客观标准（如"代码质量""论文学术水平"）

**仲裁者职责**：
- 识别虚假僵局（实则可达成共识，只是表述方式不同）
- 提出折中方案或新维度
- 最终做出"谁的论点更有说服力"的判定
