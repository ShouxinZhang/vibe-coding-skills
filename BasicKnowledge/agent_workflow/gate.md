# Gate：门禁，自动化流水线的安全阀

## 一、为什么需要门禁

在完全自动化的工作流里，没有人在中间盯着看。

这意味着：一个小错误，如果没有在产生的地方被拦截，就会随着任务的推进不断放大——错误的代码被测试引用，错误的测试被 CI 通过，错误的产物被部署上线。到发现问题时，已经是系统级故障。

门禁（Gate）是在流水线的关键节点植入的**强制检查点**。

通过，继续。不通过，停下。没有灰色地带。

---

## 二、门禁的本质：pass/fail 信号

门禁不是建议，不是警告，不是"可以忽略的 lint 提示"。

门禁只有两种输出：

| 结果 | 退出码 | 含义 |
|------|--------|------|
| PASS | `0` | 检查通过，流水线继续 |
| FAIL | 非 `0` | 检查失败，流水线阻断 |

这种二值化设计是刻意的。在无人值守的流水线中，"大部分正确"和"完全错误"的危险程度相差无几——因为没有人在场判断"这个警告是否真的重要"。门禁把这个判断前置，在规则设计阶段由人类完成，而不是在执行阶段由 agent 临时决策。

---

## 三、门禁的类型

### 类型一：代码质量门禁

在代码被提交或合并之前，验证其静态正确性。

**典型检查项：**
- **Linter（风格与语法）**：确保代码符合规范，无明显语法错误（如 Ruff for Python）
- **类型检查**：确保类型标注一致，无类型推断失败（如 Pyright、tsc）
- **格式化**：确保代码格式统一，不引入 diff 噪音

**触发时机：** 每次代码修改完成后，提交前必须通过。

---

### 类型二：架构门禁

在系统结构层面验证模块化原则是否被破坏。

**典型检查项：**
- 层级依赖方向是否正确（高层不直接依赖底层实现）
- 是否有跨模块边界的禁止导入
- 是否出现重复实现（代码冗余检测）
- 目录结构是否符合架构规则

**触发时机：** 每次新增文件或模块后运行；PR 合并前必须通过。

---

### 类型三：工作区完整性门禁

验证工作区状态的一致性，防止 agent 操作后留下脏状态。

**典型检查项：**
- 新增文件是否已登记文档
- Workspace Map 是否已更新
- 是否存在孤立文件（无所属模块的裸文件）
- 是否存在禁止目录名

**触发时机：** 每次任务结束时必须运行。这是 agent 的"扫尾检查"。

---

### 类型四：执行日志门禁

验证 agent 的行为是否有完整的可审计记录。

**典型检查项：**
- 当前任务是否已生成日志条目
- 日志是否包含必要字段（prompt、操作摘要、git 状态、可恢复性评估）
- 任务结束后是否导出了日志总览

**触发时机：** 每次任务结束时，与工作区完整性门禁配套运行。

---

## 四、门禁的执行位置

门禁可以部署在以下位置，形成纵深防御：

```
用户 prompt ──→ Agent 执行 ──→ 本地门禁 ──→ CI 门禁 ──→ 部署门禁
                                   │               │
                               最快反馈         最终保障
```

**本地门禁**（agent 自执行）：在任务结束、提交之前由 agent 主动触发，反馈最快，修复成本最低。

**CI 门禁**（流水线自动触发）：在代码推送后由 CI 系统运行，作为本地门禁的兜底。即使 agent 忘记运行本地门禁，CI 也会拦截。

**部署门禁**（上线前最终检查）：对于关键系统，在实际部署前做最后一道验证，通常包含集成测试和冒烟测试。

---

## 五、门禁失败时的处理规则

门禁失败后，agent 的行为必须遵循固定规则，不允许自由裁量：

1. **停止当前任务的后续步骤** — 不得在门禁失败的状态下继续推进
2. **输出明确的失败原因** — 记录哪项检查失败、失败的具体内容
3. **修复失败项，然后重新运行门禁** — 修复后必须重跑，不得假设修复成功
4. **若无法修复，上报人类** — agent 不应为了"完成任务"绕过门禁

**绝对禁止的行为：**
- 使用 `--no-verify`、`--force` 等参数跳过检查
- 把门禁失败的内容标记为"已知问题"后继续
- 修改门禁规则以让当前代码通过（除非有明确的人类授权）

## 六、设计一个门禁的最小要素

如果你要为自己的 agent workflow 设计一个新门禁，它需要包含：

| 要素 | 说明 |
|------|------|
| **名称** | 简短清晰，说明这个门禁检查什么 |
| **触发条件** | 什么时候必须运行这个门禁 |
| **检查逻辑** | 具体验证什么，判断标准是什么 |
| **退出码规范** | 通过返回 `0`，失败返回非 `0`，含义明确 |
| **失败输出格式** | 告诉 agent（和人类）失败的位置和原因 |
| **修复指引** | 失败时 agent 下一步应该做什么 |

门禁脚本应该是独立的、无副作用的——它只读取状态，输出判断，不修改任何文件。修复是 agent 的事，门禁只负责裁判。
