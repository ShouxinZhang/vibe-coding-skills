# Hooks：流水线的生命周期钩子

## 一、什么是 Hook

门禁是检查点，负责判断通过还是阻断。  
Workspace Structure 是约定，负责定义空间秩序。  
Hook，是**事件触发器**——在流水线生命周期的特定节点，自动执行预设逻辑。

Hook 不问"现在对不对"，它问的是"现在发生了什么，我该做什么"。

---

## 二、Hook 的触发模型

一个 Agent Workflow 的生命周期，可以抽象为一系列有序事件：

```
task:start
  ↓
context:load
  ↓
plan:generate
  ↓
step:before  ──→  step:execute  ──→  step:after
  ↓                                       ↓
(循环多步)                            gate:run
                                          ↓
                                    task:complete / task:fail
                                          ↓
                                      log:write
                                          ↓
                                    workspace:sync
```

每一个节点都可以挂载 Hook。Hook 在该节点触发，执行自定义逻辑，然后流水线继续（或根据 Hook 的结果决定是否继续）。

---

## 三、核心 Hook 类型

### `pre-task` — 任务开始前

**触发时机：** agent 接收到用户任务，尚未开始执行任何操作。

**典型用途：**
- 加载 Workspace Map，建立当前工作区全局感知
- 检查是否存在未完成的上次任务（防止任务叠加）
- 快照当前工作区状态（为可恢复性做准备）
- 记录任务开始日志

**示例逻辑（伪代码）：**
```
pre-task:
  1. workspace_docs.query("*")          # 加载全局文件清单
  2. check_incomplete_tasks()           # 检查是否有挂起任务
  3. git.snapshot()                     # 记录当前 git 状态
  4. agent_logs.start(prompt=user_task) # 开启日志条目
```

---

### `pre-step` / `post-step` — 每个子步骤前后

**触发时机：** agent 执行任务拆解后，每个子步骤开始前 / 完成后。

**典型用途（pre-step）：**
- 确认该步骤涉及的文件已有文档记录
- 确认该步骤不会触犯架构边界规则

**典型用途（post-step）：**
- 更新被修改文件的文档记录
- 运行局部代码质量检查（而非等到任务结束才全量检查）

---

### `on-gate-fail` — 门禁失败时

**触发时机：** 任意门禁检查返回非 `0` 退出码。

这是最重要的 Hook 之一。门禁失败后，agent 不应自由发挥，而应由 `on-gate-fail` Hook 接管行为。

**标准处理流程：**
```
on-gate-fail:
  1. 记录失败的门禁名称与失败原因
  2. 停止当前步骤的后续操作
  3. 查找门禁的修复指引（SKILL.md 或错误输出）
  4. 尝试修复（最多 N 次，N 可配置）
  5. 修复后重新运行该门禁
  6. 若超过重试次数仍失败 → 上报人类，暂停任务
```

**关键约束：** `on-gate-fail` Hook 本身不得修改门禁规则，也不得降低门禁严格程度。

---

### `post-task` — 任务完成后

**触发时机：** 所有步骤执行完毕，所有门禁通过。

**典型用途：**
- 导出 Workspace Map（`workspace_docs.export()`）
- 写入完整任务日志并导出日志总览
- 运行全量架构检查（作为最终验证）
- 清理任务过程中产生的临时状态

---

### `on-error` — 非预期错误发生时

**触发时机：** 流水线在任意位置遇到未被门禁覆盖的错误（异常抛出、工具调用失败、超时等）。

与 `on-gate-fail` 不同——门禁失败是预期内的"检查不通过"，`on-error` 处理的是**预期外的崩溃**。

**标准处理流程：**
```
on-error:
  1. 记录错误上下文（发生位置、错误信息、当前任务状态）
  2. 回滚到最近一次 git snapshot（如有）
  3. 标记任务为异常中止
  4. 写入错误日志
  5. 通知人类介入
```

## 四、Hook 的设计原则

**1. Hook 应该是幂等的**  
同一个 Hook 触发两次，结果应该和触发一次相同。流水线可能因重试而多次触发同一 Hook，幂等性防止副作用叠加。

**2. Hook 本身不应有副作用门禁**  
Hook 是响应逻辑，不是判断逻辑。在 Hook 内部运行门禁会产生嵌套触发链，导致行为难以追踪。如需验证，通过门禁系统显式触发。

**3. Hook 失败不得静默**  
Hook 执行失败必须产生可见输出，并决定流水线是继续还是中止。不允许 Hook 失败后流水线假装什么都没发生。

**4. Hook 逻辑应集中注册，不应散落在任务代码中**  
所有 Hook 注册在一个统一的配置入口（如 `hooks.json` 或 `AGENTS.md` 的 Hook 段），而不是散落在各个任务脚本里。这使得 Hook 体系可见、可审计、可修改。

---

## 五、Hook 与门禁的关系

Hook 和门禁是互补的，不是重复的：

| | 门禁（Gate） | 钩子（Hook） |
|---|---|---|
| **问题** | 现在的状态对不对？ | 现在发生了什么事？ |
| **触发方式** | 显式调用，必须通过才能继续 | 事件驱动，自动触发 |
| **输出** | pass / fail | 执行一段逻辑 |
| **可选性** | 强制 | 可配置（但建议必选核心 Hook）|
| **典型例子** | 代码质量检查、架构验证 | 加载上下文、写日志、快照恢复 |

一个成熟的 Agent Workflow，门禁负责守住质量边界，Hook 负责维持流水线的上下文连续性与可观测性。两者缺一，自动化就是在走钢丝。
