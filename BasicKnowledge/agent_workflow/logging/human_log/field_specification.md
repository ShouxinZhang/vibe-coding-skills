# 字段规范

## 设计原则

Human Log 的字段体系遵循一个核心约束：**每个字段回答一个且仅一个问题。**

字段之间不应有语义重叠。如果两个字段经常被填入相同的内容，说明其中一个是多余的。如果一个字段经常被留空，说明它要么不够必要，要么定义不够清晰。

字段的设计不追求"全面覆盖所有可能"。它追求的是：**人类在阅读时不需要猜。** 每个字段的含义从名字就能看出来，不需要查手册。

---

## 字段总览

| 字段 | 类型 | 必选 | 回答的问题 |
|------|------|------|-----------|
| `timestamp` | string | ✅ | 发生在什么时间？ |
| `prompt` | string | ✅ | 用户的原始请求是什么？ |
| `summary` | string | ✅ | Agent 做了什么？为什么？ |
| `files_modified` | list | ✅ | 动了哪些文件？ |
| `git_op` | string | ✅ | 涉及哪些 git 操作？ |
| `gate_results` | object | ✅ | 门禁结果如何？ |
| `isolated` | boolean | ✅ | 变更范围是否受控？ |
| `backup` | boolean | ✅ | 有没有备份？ |
| `integrity_ok` | boolean | ✅ | 任务后工作区完整吗？ |
| `recoverability_ok` | boolean | ✅ | 可回滚吗？ |
| `error` | string/null | ✅ | 出错了吗？ |

注意：**所有字段都是必选的。**

这是一个刻意的设计决定。可选字段是日志退化的温床——"这次不重要所以不填"会迅速扩散为"每次都不填"。如果一个字段确实不适用，填 `null` 或 `N/A`，但必须显式标注。留白不等于跳过。

## 逐字段深入

### timestamp

**回答的问题**：这件事发生在什么时间？

**格式**：ISO 8601，精确到秒，必须包含时区。

```
✅ 2026-03-06T14:32:07+08:00
❌ 2026-03-06 14:32
❌ 1709712727（Unix timestamp）
❌ "今天下午"
```

**为什么精确到秒**：当同一天有多个任务时，秒级精度是区分它们的最低要求。分钟级可能碰撞。

**为什么必须包含时区**：无人值守的 agent 可能跨时区运行。没有时区的时间戳是无意义的——"14:32"在哪个时区？

**边界案例**：
- 任务跨越午夜：使用任务**开始**的时间戳，不是结束时间。日期目录归属以开始时间为准。
- 时钟漂移：如果系统时钟不准确，这条日志的时序就是错的。timestamp 依赖运行环境的时钟可靠性，日志系统无法独立解决这个问题。

**反模式**：
- 使用相对时间（"5 分钟前"）——日志是持久档案，相对时间在写入瞬间就过期了。
- 后填时间戳——任务结束后才补写开始时间，记忆不可靠，应该在任务开始时立即记录。

### prompt

**回答的问题**：用户说了什么？

**核心规则：原文照录，不做任何改写。**

这是日志系统中最硬的规则之一。

prompt 是"意图存证"。它记录的是人类的原始请求，不是 agent 对请求的理解。如果 agent 改写了 prompt，事后审阅时就无法区分"用户要求这样做"和"agent 自己决定这样做"。

```
✅ "在 agent_workflow 模块新增 hooks.md，并更新架构文档"
❌ "用户希望新增一个关于 hooks 的文档"（改写了原文）
❌ "新增 hooks.md"（省略了部分内容）
```

**边界案例**：
- prompt 非常长（超过 500 字）：全文保留在日志中，但 summary 不应复述 prompt。如果 prompt 长到影响日志可读性，可以将完整 prompt 放在日志末尾的附录区域，正文中用一句话概括并标注"完整 prompt 见附录"。
- prompt 包含敏感信息（密码、API key）：**脱敏后标注**。例：`"部署到 [API_KEY_REDACTED] 环境"`。脱敏本身也应该被记录："原始 prompt 包含 API key，已脱敏"。
- 没有明确的 prompt（自动触发的任务）：记录触发条件。例：`"[自动触发] schedule: daily-cleanup at 03:00 UTC"`。
- prompt 包含多轮对话上下文：只记录最终触发任务的那条指令，但标注"本次任务基于多轮对话，前置上下文未包含在此"。

**反模式**：
- 翻译 prompt——用户用中文写的 prompt 被翻译成英文记录，丢失了原始语义。
- 截断 prompt——"太长了所以只记录前 100 字"。截断需要在摘要中说明，但 prompt 字段本身不允许截断。

### summary

**回答的问题**：Agent 做了什么？为什么这样做？

这是整条日志中**信息密度最高**的字段，也是最容易写烂的字段。详细的写作指导见 [写作技艺](./writing_craft.md)。

**核心要求**：
1. 叙述操作事实（做了什么）
2. 交代决策理由（为什么这样而不是那样）
3. 标注异常情况（如果有）

**字数控制**：
- 最短不低于 30 字。低于 30 字的 summary 几乎不可能包含足够的信息。
- 最长不超过 300 字。超过 300 字说明在讲故事而不是写摘要，应该精简。
- 典型长度：80–150 字。

**反模式**（后果严重，必须避免）：
- "完成了用户的请求" — 零信息量。
- "按照要求修改了文件" — 零信息量。
- 复制粘贴 prompt 原文 — 没有 agent 的判断，日志退化为 prompt 的回声。
- "新建了 X，修改了 Y，更新了 Z" — 只有操作列表没有理由，这跟 files_modified 重复。

### files_modified

**回答的问题**：动了哪些文件？

**格式要求**：
- 每个文件独占一行
- 必须标注操作类型：`[新建]`、`[修改]`、`[删除]`、`[重命名]`、`[移动]`
- 路径使用相对于仓库根目录的完整路径

```
✅
- [新建] BasicKnowledge/agent_workflow/hooks.md
- [修改] docs/architecture/repository-structure.md

❌
- hooks.md（路径不完整）
- 修改了架构文档（用自然语言描述代替路径）
- BasicKnowledge/agent_workflow/hooks.md（缺少操作类型）
```

**边界案例**：
- 修改了 50 个文件：全部列出。这不是应该省略的字段——如果一次任务修改了 50 个文件，这本身就是一个值得关注的信号。
- 文件被修改后又回滚了：记录为 `[修改后回滚]`，说明最终状态等同于未修改，但操作确实发生过。
- 仅修改了文件的权限或元数据：记录为 `[元数据修改]`，标注具体变更（如权限从 644 变为 755）。

**与 summary 的关系**：files_modified 是事实清单，summary 是叙事解释。summary 不应重复列出文件列表，而应解释"为什么要动这些文件"。

### git_op

**回答的问题**：是否执行了 git 操作？如果有，具体是什么？

**格式**：

```
# 无 git 操作
git_op: none

# 有 commit 但没 push
git_op: commit (sha: a3f2c1d, msg: "add hooks.md")

# 有 push
git_op: push (branch: main, remote: origin)

# 有多个操作
git_op:
  - commit (sha: a3f2c1d, msg: "add hooks.md")
  - push (branch: main, remote: origin)
```

**为什么这个字段至关重要**：git 操作是不可逆操作的边界。文件修改可以靠 `git checkout` 回滚，但 `push` 一旦执行，影响范围就超出了本地工作区。

**边界案例**：
- agent 执行了 `git add` 但没有 `commit`：记录为 `staged (files: X, Y)`。虽然 add 本身影响不大，但它改变了暂存区状态，下一个 agent 可能受到影响。
- agent 执行了 `commit --amend`：必须特别标注。amend 修改了已存在的 commit，历史被改写。
- agent 执行了 `force push`：这是高危操作，必须用醒目标注（如加粗或 ⚠️ 前缀）。

**反模式**：
- 只记录 "yes" / "no"——没有具体内容等于没记录。
- 遗漏了 push 操作——这是审计盲点中最危险的一种。

### gate_results

**回答的问题**：各门禁的运行结果如何？

**格式**：

```
gate_results:
  architecture-guard: PASS
  workspace-integrity: PASS
  code-quality: FAIL (reason: "eslint 发现 3 个 error")
  type-check: SKIP (reason: "本次任务不涉及 TypeScript 文件")
```

**四种合法状态**：
- `PASS`：门禁通过。
- `FAIL`：门禁未通过，**必须附原因**。
- `SKIP`：门禁被跳过，**必须附原因**。
- `ERROR`：门禁本身运行异常（区别于检查不通过），**必须附错误信息**。

**为什么 SKIP 和 ERROR 必须附原因**：SKIP 是合理的——不是每个门禁都适用于每个任务。但没有理由的 SKIP 和"直接跳过"无法区分。ERROR 意味着门禁系统本身不可靠，这是系统性风险。

**边界案例**：
- 门禁通过但有警告：记录为 `PASS (warnings: 2, details: "...")`。
- 门禁被人类手动豁免：记录为 `OVERRIDE (by: human, reason: "...")`。这必须显式标注——绕过门禁是高权限操作。
- 同一门禁运行了多次（重试）：记录每次的结果，包括中间的失败。

**反模式**：
- 只记录 PASS 的门禁，省略 FAIL 的——选择性报告，直接破坏日志的信任度。
- 没有区分 FAIL 和 ERROR——"没通过"可能是代码有问题（FAIL），也可能是 linter 本身崩溃了（ERROR），两者的应对方式完全不同。

### isolated

**回答的问题**：变更是否限定在预期范围内？

**类型**：boolean + 说明

```
# 变更受控
isolated: yes（变更限定在 BasicKnowledge/agent_workflow/ 和 docs/）

# 变更溢出
isolated: no（预期只修改 hooks.md，但实际还触碰了 gate.md 的导入语句）
```

**为什么需要这个字段**：隔离性是模块化设计的运行时验证。架构规定每个任务应该在有限范围内操作。isolated 字段记录的是"实际做到了吗"。

**判断标准**：变更涉及的所有文件，是否都在任务开始时预期的模块范围内。如果任务是"修改 hooks.md"但实际还修改了 gate.md——即使修改是合理的，isolated 也应标记为 no，并在 summary 中解释原因。

**反模式**：
- 总是填 yes——如果变更从来不溢出，要么 agent 的自律性很强（好事），要么这个字段没有被认真填写（坏事）。定期审查 isolated 全是 yes 的日志序列，确认是否属实。

### backup

**回答的问题**：操作前是否有备份？

**类型**：boolean + 说明

```
backup: yes（git snapshot at HEAD: a3f2c1d）
backup: yes（文件备份至 .backup/2026-03-06/）
backup: no（原因：只读操作，未修改任何文件）
```

**为什么 no 也必须附原因**：缺少备份是一种风险暴露。如果确实不需要备份（比如只读操作），那么说明原因是安全的。如果是因为遗忘或技术限制导致没有备份——这是一个需要被标记的风险点。

**边界案例**：
- git 工作区有未提交的修改：这时 `git stash` 或手动备份变成了必要操作。单纯指向 git HEAD 不够——HEAD 只保护了已提交的状态，未提交的修改不受保护。
- 备份机制本身不可靠（如磁盘空间不足）：记录为 `backup: partial (reason: "...")`。

---

### integrity_ok

**回答的问题**：任务完成后，工作区的整体状态是否正常？

**类型**：boolean

检查内容包括但不限于：
- 文件结构是否符合 repository-structure.md 的定义
- architecture 文档是否与实际结构同步
- 是否有残留的临时文件
- 是否有被意外修改的配置文件

**与 gate_results 的关系**：gate_results 记录的是门禁在运行时的判断。integrity_ok 是更宏观的检查——即使所有门禁都 PASS，工作区整体上也可能存在不一致（比如某个门禁没有覆盖到的检查项）。

---

### recoverability_ok

**回答的问题**：如果这次操作的结论有误，能否安全回滚？

**类型**：boolean + 说明

```
recoverability_ok: yes（git reset --hard HEAD~1 可完全还原）
recoverability_ok: no（已 push 到 remote，需要 force push 才能回滚）
```

**与 backup 的区别**：backup 是"操作前有没有保存现场"，recoverability_ok 是"操作后能不能回到之前的状态"。有备份不一定能恢复——如果操作涉及外部系统（如 API 调用、数据库写入），即使本地有备份也无法回滚外部影响。

---

### error

**回答的问题**：任务过程中是否出现了错误？

**格式**：

```
# 无错误
error: null

# 有错误
error:
  location: "gate.md 第 42 行引用检查"
  message: "找不到引用的文件路径 hooks_old.md"
  handling: "修正引用路径为 hooks.md，门禁重新通过"
  resolved: yes
```

**三个必须回答的子问题**：
1. 错误在哪里发生的（location）
2. 错误是什么（message）
3. 如何处理的（handling）

**边界案例**：
- 错误被自动重试后消亡：仍然记录。"重试后成功"是有价值的信息——它暗示系统中存在间歇性故障。
- 多个错误：每个单独记录。不要合并为"遇到了一些问题"。
- 错误未被解决：标记 `resolved: no`，并在 summary 中标注为遗留问题。

**反模式**：
- 只记录"有错误"不记录具体内容。
- 错误已经影响了结果，但 error 字段填了 null——这是最危险的一种日志失真。

## 字段之间的依赖关系

字段不是独立存在的。某些字段的值会约束其他字段的合法取值。

| 条件 | 约束 |
|------|------|
| `git_op` 包含 push | `recoverability_ok` 不应为 yes（除非有明确的回滚方案） |
| `gate_results` 有任何 FAIL | `summary` 必须说明失败原因及后续处理 |
| `error` 不为 null 且 `resolved: no` | `summary` 必须标注遗留问题 |
| `isolated: no` | `summary` 必须解释为什么变更超出预期范围 |
| `files_modified` 为空 | `summary` 必须说明为什么没有修改任何文件 |
| `backup: no` 且 `files_modified` 非空 | 这是一个风险信号，必须在 summary 中解释原因 |

这些约束不是建议，是**硬规则**。门禁应自动检查这些依赖关系，不一致即报错。

## 不同任务类型的填写差异

### 创建型任务（新建文件 / 模块）

重点字段：`files_modified`（列出全部新建文件）、`isolated`（新文件是否放在了正确的模块里）、`integrity_ok`（架构文档是否同步更新）。

summary 应说明：创建了什么、为什么需要创建、结构设计的关键决策。

### 修改型任务（修改现有代码 / 文档）

重点字段：`backup`（修改前有没有保存现场）、`files_modified`（标注修改的具体范围）、`recoverability_ok`（能否安全回滚）。

summary 应说明：修改了什么内容、为什么需要修改、修改前后的差异概述。

### 删除型任务（删除文件 / 代码）

重点字段：`backup`（**删除型任务中必须为 yes**——删除操作不可逆）、`files_modified`（用 `[删除]` 标注）、`recoverability_ok`（从备份还原的具体路径）。

summary 应说明：为什么需要删除、删除后是否影响其他模块、删除的内容是否有替代方案。

### 重构型任务（移动 / 重命名 / 结构调整）

重点字段：`isolated`（重构往往涉及多个模块，溢出风险最高）、`files_modified`（需要完整记录移动的源路径和目标路径）、`integrity_ok`（重构后引用链是否完整）。

summary 应说明：重构的动机、新旧结构的对比、是否有遗留的引用需要清理。
