# Human Log：写给人类审阅的任务日志

## 为什么人类日志值得单独设计

在一条全自动化的 Agent Workflow 流水线中，人类不在场。

Agent 自行接收任务、规划路径、操作文件、运行门禁、提交结果。整条链路可以在午夜运行，可以在假期运行。人类不需要盯着它执行。

但人类需要**事后知道发生了什么**。

这不是可选的。当一个无人值守的系统连续运行了 50 个任务，第 37 个任务引入了一个隐患，在第 48 个任务爆发为错误——人类必须能够回溯到第 37 个任务，理解当时的决策过程，才能判断问题的根因。

**Human Log 的存在理由：让不在场的人类，拥有在场的判断力。**

---

## 人类日志不是 Agent 日志

这两者经常被混淆，但它们服务于完全不同的读者。

**Agent Log** 是机器之间的工作记忆传递。它结构严格、字段精确，下一个 agent 读取后可以直接用于决策。它追求的是**可解析性**。

**Human Log** 是写给人类的审计档案。它叙事清晰、逻辑完整，人类阅读后应该能还原全貌。它追求的是**可理解性**。

一句话概括：Agent Log 像数据库查询结果，Human Log 像调查报告。

两者不能合并。格式冲突（叙事性 vs 结构化），更新频率冲突（实时 vs 关账式），读取方式冲突（程序解析 vs 人眼阅读）。详见 [logging/introduction.md](../introduction.md)。

## 一条好日志的标准

贯穿本章节的核心判断标准：

> **一个从未接触过这个项目的人，仅凭这条日志，就能理解当时发生了什么、为什么这样做、以及结果如何。**

如果读完一条日志还需要去翻代码、看 diff、问同事——这条日志就是失败的。
