# 存储架构

## 日志不只是写出来的，还是找出来的

写日志是为了事后阅读。事后阅读的核心场景是**搜索**——人类带着一个问题来翻日志，需要快速定位到相关的那一条。

如果日志写得再好，但存储结构让人找不到想要的那条——日志系统就是失败的。

存储架构要解决三个问题：
1. 单条日志放在哪
2. 多条日志怎么索引
3. 日志量增长后怎么扩展

---

## 整体目录结构

```
docs/agent-logs/
├── YYYY-MM-DD/
│     └── log-<序号>-<HHMMSS>.md       ← 单条任务日志
├── INDEX.md                            ← 日期索引
└── AGENT_LOGBOOK.md                    ← 全局日志册
```

三层结构，每层服务于不同的检索粒度。

---

## 第一层：单条日志

### 命名规范

```
log-<序号>-<HHMMSS>.md
```

- **序号**：当天的任务编号，从 `001` 开始，三位数，零填充。
- **HHMMSS**：任务**开始**时间，24 小时制。

示例：
```
log-001-143207.md    ← 当天第 1 个任务，开始于 14:32:07
log-002-145501.md    ← 当天第 2 个任务，开始于 14:55:01
log-017-032200.md    ← 当天第 17 个任务，开始于 03:22:00
```

### 为什么用序号 + 时间双重标识

单独用序号：看不出时间。单独用时间：当需要"第几个"这种序数引用时不方便。两者并存，信息冗余但使用方便。

### 序号重置规则

每天从 `001` 重新开始。不使用全局递增编号——全局编号在数百天后会到达四位数甚至五位数，失去了快速识别的意义。

### 序号上限

三位数 = 999。如果一天内执行了超过 999 个任务，扩展到四位数（`0001`）。但这种情况本身是一个告警信号——说明任务粒度可能太细。

### 文件内容结构

每条日志是一个独立的 Markdown 文件，标题格式固定：

```markdown
# Task Log — YYYY-MM-DD · #<全局编号>

## 基本信息
- **时间**: <timestamp>
- **Prompt**: <prompt>

## 操作摘要
<summary>

## 修改文件
<files_modified>

## Git 操作
<git_op>

## 门禁结果
<gate_results>

## 安全性评估
- isolated: <isolated>
- backup: <backup>
- integrity_ok: <integrity_ok>
- recoverability_ok: <recoverability_ok>

## 错误记录
<error>
```

**全局编号**（标题中的 `#0042`）与文件名里的序号不同。全局编号是跨天的累计编号，从项目开始以来持续递增，用于唯一标识一条日志。文件名中的序号是当天内的编号。

### 全局编号管理

全局编号由 AGENT_LOGBOOK.md 维护。每次写入新日志时：
1. 读取 AGENT_LOGBOOK.md 中最后一条记录的编号
2. +1 作为新编号
3. 写入新日志的标题和 AGENT_LOGBOOK.md

不依赖文件系统计数——文件可能被手动增删，文件系统计数不可靠。

## 第二层：日期索引（INDEX.md）

### 作用

日期索引是时间维度的入口。人类的第一个检索动作往往是"那天发生了什么"。

### 格式

```markdown
# Agent Logs — 日期索引

## 2026-03-06（5 个任务）
| # | 时间 | 摘要 | 文件 |
|---|------|------|------|
| 0042 | 14:32 | 新增 hooks.md，Hook 触发模型与类型 | [log](2026-03-06/log-001-143207.md) |
| 0043 | 14:55 | 修正 gate.md 引用路径 | [log](2026-03-06/log-002-145501.md) |
| ... | ... | ... | ... |

## 2026-03-05（3 个任务）
| # | 时间 | 摘要 | 文件 |
|---|------|------|------|
| ... | ... | ... | ... |
```

### 设计要点

- 最新日期在最上面。
- 摘要列不超过 30 字——这是导航摘要而非完整 summary。人类用它判断"是不是这条"，然后点进去看完整日志。
- 每个日期区块标注当天任务数量。

### 更新时机

每次写入新日志时，同步更新 INDEX.md。这是 post-task Hook 的一部分，不是独立操作。

---

## 第三层：全局日志册（AGENT_LOGBOOK.md）

### 作用

全局日志册是"一张表看全部"的入口。它不提供详情，只提供概览，用于快速扫描历史。

### 格式

```markdown
# Agent Logbook

| # | 日期 | 时间 | 摘要 | 状态 | 文件 |
|---|------|------|------|------|------|
| 0043 | 2026-03-06 | 14:55 | 修正引用路径 | ✅ | [→](2026-03-06/log-002-145501.md) |
| 0042 | 2026-03-06 | 14:32 | 新增 hooks.md | ✅ | [→](2026-03-06/log-001-143207.md) |
| 0041 | 2026-03-05 | 16:20 | 重构日志模块结构 | ⚠️ | [→](2026-03-05/log-003-162000.md) |
| ... | ... | ... | ... | ... | ... |
```

### 状态列

- `✅`：任务成功完成，无遗留问题。
- `⚠️`：任务完成但有妥协或遗留问题。
- `❌`：任务失败或中途终止。

三种状态足够。不需要更细的分级——日志册是导航入口，不是分析工具。

### 与 INDEX.md 的区别

INDEX.md 按日期分组，适合"查看某天的日志"。
AGENT_LOGBOOK.md 是扁平列表，适合"从最新往回翻"和"搜索关键词"。

两者信息有重叠，但服务的检索模式不同。维护成本可以接受，因为两者的更新都是自动化的。

---

## 命名规范总结

| 元素 | 规范 | 示例 |
|------|------|------|
| 日期目录 | `YYYY-MM-DD` | `2026-03-06/` |
| 日志文件 | `log-<NNN>-<HHMMSS>.md` | `log-001-143207.md` |
| 日期索引 | `INDEX.md` 固定名称 | — |
| 全局日志册 | `AGENT_LOGBOOK.md` 固定名称 | — |
| 全局编号 | 四位数零填充 | `#0042` |
| 当天序号 | 三位数零填充 | `001` |

所有文件名使用小写字母和连字符。不使用空格、下划线或中文字符。

## 归档与轮转策略

### 为什么需要归档

日志是只增不删的。运行半年后，`docs/agent-logs/` 目录可能有 180 个日期目录，INDEX.md 可能有数千行。文件系统性能不会有问题，但人类的阅读体验会急剧恶化。

### 归档规则

```
docs/agent-logs/
├── archive/
│     ├── 2026-Q1/                   ← 按季度归档
│     │     ├── 2026-01-15/
│     │     ├── 2026-01-16/
│     │     └── ...
│     └── 2026-Q1-INDEX.md           ← 归档后的季度索引
├── 2026-03-05/                       ← 当前季度保留在顶层
├── 2026-03-06/
├── INDEX.md                          ← 仅包含当前季度
└── AGENT_LOGBOOK.md                  ← 始终包含全部（最近 200 条）
```

**归档触发条件**：新季度开始时，将上一季度的日志移入 `archive/`。

**归档后 INDEX.md 的处理**：
- 主 INDEX.md 仅保留当前季度。
- 归档季度生成独立的 `YYYY-Qn-INDEX.md`。
- 主 INDEX.md 底部保留归档链接列表。

**归档后 AGENT_LOGBOOK.md 的处理**：
- 始终保留最近 200 条记录。
- 超过 200 条的旧记录移入 `archive/LOGBOOK-ARCHIVE.md`。
- AGENT_LOGBOOK.md 底部标注"更早记录见归档"。

### 为什么是按季度而非按月

按月归档导致频繁的目录操作。按年归档则在年中时积累太多未归档日志。季度是平衡操作频率和积累量的折中。

---

## 当日志量级达到数百条 / 数千条

### 文件系统层面

1000 条日志 ≈ 100-200 个日期目录 × 5-10 个文件/目录。现代文件系统完全能处理。

但人类的认知不能。1000 条日志意味着 INDEX.md 有上千行，打开就要滚屏几十次。归档策略解决的就是这个问题：当前季度通常不超过 100 条，这是可管理的规模。

### AGENT_LOGBOOK.md 的增长

AGENT_LOGBOOK.md 是唯一一个无限增长的文件（归档只转移旧条目，全局编号持续增加）。

当总量超过 200 条时，AGENT_LOGBOOK.md 拆分为：
- 主文件：最近 200 条
- 归档文件：其余

人类几乎不会翻看 200 条以前的记录。如果需要，通过归档文件查找或使用 `grep`。

### 全局编号的上限

四位数编号最大 9999。按每天平均 5 个任务计算，够用约 5.5 年。到达上限时扩展到五位数。

## 搜索与检索场景

日志的价值在事后审阅中兑现。人类带着问题来搜索时，通常属于以下几种模式：

### 场景一："那天发生了什么？"

**检索路径**：INDEX.md → 找到日期 → 浏览当天所有任务的导航摘要 → 点进目标日志。

**设计支撑**：INDEX.md 按日期分组，每个条目有 30 字以内的导航摘要。

### 场景二："某个文件最近被谁改过？"

**检索路径**：`grep -r "target_file.md" docs/agent-logs/` → 找到包含该文件名的日志 → 按时间排序取最近的。

**设计支撑**：每条日志的 `files_modified` 字段使用完整路径，`grep` 可以精准匹配。

### 场景三："最近出过什么错？"

**检索路径**：AGENT_LOGBOOK.md → 过滤状态列为 ⚠️ 或 ❌ 的条目 → 点进去看详情。

**设计支撑**：AGENT_LOGBOOK.md 的状态列用 emoji 标记，肉眼扫描即可筛选。

### 场景四："某个决策是什么时候做的？"

**检索路径**：`grep -r "关键词" docs/agent-logs/` → 在 summary 中搜索决策相关的关键词。

**设计支撑**：summary 中的决策点使用具体名词（如 "Hook 类型按触发时机分类"），关键词可被搜索命中。这就是为什么 summary 中不应使用模糊表述（如"做了合理的设计"）——模糊表述无法被搜索。

### 场景五："编号 #0042 的日志在哪？"

**检索路径**：`grep "#0042" docs/agent-logs/AGENT_LOGBOOK.md` → 找到链接 → 点进去。

**设计支撑**：全局编号在日志标题和 AGENT_LOGBOOK.md 中同时出现，双重索引。

## 自动化写入流程

存储架构的所有操作都应由 `post-task` Hook 自动完成。人类不参与日志的日常写入和索引维护。

```
post-task Hook 执行流程：
1. 生成日志文件名（读取当天序号 + 1）
2. 读取 AGENT_LOGBOOK.md 最后一条编号 + 1 → 全局编号
3. 创建日期目录（如果不存在）
4. 写入日志文件
5. 更新 INDEX.md（追加当天条目）
6. 更新 AGENT_LOGBOOK.md（在顶部插入新行）
```

步骤 5 和 6 的顺序不重要，但必须在步骤 4 之后——日志文件必须先存在，索引中的链接才有效。

**关键约束**：步骤 2 和步骤 6 必须是原子操作——读取编号和写入新编号之间不能有其他任务插入，否则会出现编号冲突。在单 agent 顺序执行的场景下这自然满足；在多 agent 并发场景下需要锁机制。
