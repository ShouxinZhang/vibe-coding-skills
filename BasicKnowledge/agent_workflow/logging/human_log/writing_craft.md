# 写作技艺

## 日志不是表格填写，是叙事工程

字段规范定义了"日志包含什么"。但填满了所有字段的日志，不等于好日志。

差日志的特征：字段齐全，但每个字段都是最低信息量的填充物。timestamp 有了，prompt 有了，summary 写了"完成了任务"——技术上合规，实际上无用。

**好日志的本质是一篇微型调查报告。** 它有时间线，有因果链，有决策点，有结论。读者不需要额外信息就能还原全貌。

本章讨论的不是"怎么填 summary 字段"，而是**怎么把整条日志写成一份人类真正能用的记录**。

---

## 信息密度控制

信息密度是日志质量的第一指标。

**密度过低**：读了一整段，没有获取到新信息。

```
❌ "根据用户的要求，我对指定的文件进行了相应的修改和更新，
    确保了修改后的内容符合项目的整体规范和要求。"
```

这段话有 50 个字，信息量为零。删掉它，日志没有任何损失。

**密度过高**：一句话塞进了太多信息，读者需要反复阅读。

```
❌ "重构了 hooks.md 的触发模型从事件驱动改为状态机并修正了
    gate.md 中 on-gate-fail 的重试逻辑同时更新了架构文档
    中的 Hook 章节描述和交叉引用。"
```

这句话有 4 个独立操作，压缩在一个没有断点的长句里。读者需要自己拆解。

**正确的密度**：每句话传递一个清晰的信息单元，句间有逻辑连接。

```
✅ "重构了 hooks.md 的触发模型，从事件驱动改为状态机——
    状态机在多步任务中更容易追踪当前阶段。
    同步修正了 gate.md 中 on-gate-fail 的重试逻辑，
    因为旧逻辑在新模型下会导致死循环。
    架构文档已更新。"
```

**经验法则：一句话不超过 25 个字。一段不超过 4 句。如果一段写到第 5 句，说明应该拆段落了。**

## 因果链还原技术

人类审阅日志时，80% 的时间在做一件事：**还原因果链**。他们不只是想知道"发生了什么"，还想知道"为什么会这样"。

因果链的写法有一个固定结构：**现象 → 判断 → 行动 → 结果**。

### 结构示例

```
现象：architecture-guard 报错，提示 hooks.md 的位置违反了模块隔离规则。
判断：hooks.md 放在了 agent_workflow/ 根目录，规则要求功能文件放在子目录中。
行动：将 hooks.md 移入新建的 hooks/ 子目录，并更新所有引用路径。
结果：architecture-guard 通过，无其他门禁报错。
```

这四步不一定要标注关键词，但信息必须完整。缺少任何一步，因果链就断了。

### 常见断链模式

**缺少"判断"**：
```
❌ "architecture-guard 报错了，所以我把 hooks.md 移到了子目录里。"
```
为什么移到子目录就能解决？读者不知道报错的原因是什么。

**缺少"行动"**：
```
❌ "architecture-guard 报错，提示位置违规。现在已经修复了。"
```
修复了什么？怎么修的？不知道。

**缺少"结果"**：
```
❌ "把 hooks.md 移到了子目录里。"
```
移了之后呢？问题解决了吗？有没有引发新问题？不知道。

## 决策点标注方法

决策点是日志中最有价值的信息。

**决策点 = agent 在两个或以上合理选项中做出了选择的地方。** 它标注的不是"做了什么"，而是"为什么这样做而不是那样做"。

### 标注格式

在 summary 中，决策点用短句明确标出：

```
✅ "Hook 的类型分类按触发时机而非功能划分——因为流水线视角下，
    时机比功能更直观。"
```

这一句包含：选择（按时机分类）、放弃的选项（按功能分类）、理由（时机更直观）。

```
❌ "Hook 的类型做了合理的分类。"
```

"合理的"是主观评价，不是决策记录。读者不知道有哪些选项、选了哪个、为什么。

### 决策点的密度

不是每个操作都是决策点。打开文件、修改拼写错误、更新日期——这些不需要标注。

**判断标准：如果换一个 agent 来做，有可能做出不同的选择，那这就是一个决策点。**

一个典型任务通常有 1-3 个决策点。如果一条日志标注了 10 个决策点，说明要么任务太复杂（应该拆分），要么标注太琐碎（降低了真正重要决策的可见性）。

## 如何处理"做了但没做成"

Agent 尝试了一个操作，失败了，然后换了一个方案。

很多日志会跳过失败的尝试，只记录最终成功的路径。**这是错误的做法。**

失败的尝试是极其有价值的信息。它告诉未来的读者：这条路走不通，别再试了。

### 标注方法

```
✅ "首先尝试用正则表达式批量替换所有引用路径，
    但发现 Markdown 文件中的相对路径格式不统一（有的带 ./，有的不带），
    正则方案无法覆盖所有情况。
    改为逐文件手动修正引用，共修改 7 处。"
```

关键要素：
1. 明确说明尝试了什么
2. 明确说明为什么失败
3. 明确说明最终采用的替代方案

```
❌ "修正了所有引用路径。"
```

过程中的失败尝试被完全抹除。下一个 agent 碰到类似问题时，可能会重蹈覆辙。

## 如何记录妥协和遗留问题

完美的任务执行几乎不存在。大多数任务结束时，都会留下一些不完美的地方：

- 一个已知但未修复的小问题
- 一个不够优雅但能工作的临时方案
- 一个检测到但不在本次任务范围内的风险

这些信息**必须在日志中显式记录**。

### 标注格式

在 summary 末尾，用独立段落标注妥协和遗留：

```
✅ "妥协：hooks.md 的 ASCII 流程图在窄终端下会换行变形，
    暂未找到跨宽度的兼容方案，保留当前格式。

    遗留：gate.md 中有一处引用了已废弃的 hook 名称 on-pre-check，
    应更新为 pre-task，但本次任务范围不包含 gate.md 修改。"
```

### 妥协 vs 遗留 的区别

- **妥协**：agent 知道有更好的方案，但因为某种限制选择了当前方案。强调"为什么不用更好的方案"。
- **遗留**：agent 发现了一个问题，但它不在本次任务的范围内。强调"这个问题存在，等待后续处理"。

两者都不是错误。它们是对现实的诚实记录。**隐藏妥协和遗留，比妥协和遗留本身更危险。**

## 好坏对比示例

### 对比一：简单创建任务

**坏：**
```
## 操作摘要
按照要求新建了 hooks.md 文件。内容包括 Hook 的定义和类型。
同步更新了架构文档。
```

**好：**
```
## 操作摘要
新建 hooks.md，内容涵盖 Hook 触发模型、五种核心 Hook 类型
（pre-task / post-task / on-error / on-gate-fail / on-context-load）
及其与门禁的对比。

决策点：Hook 类型按触发时机分类而非功能分类，
因为在流水线的阅读语境中，"什么时候触发"比"做什么"更自然。

同步更新 docs/architecture/repository-structure.md，
新增 hooks.md 的条目和说明。
```

**差距在哪**：坏例子只有操作列表。好例子有具体内容概述、决策点和理由。

---

### 对比二：遇到错误的任务

**坏：**
```
## 操作摘要
修改了 gate.md 的门禁配置。过程中遇到了一些问题，已解决。

## 错误记录
- 有一个错误，已修复
```

**好：**
```
## 操作摘要
修改 gate.md，将 architecture-guard 的检查范围从"仅 .ts 文件"
扩展为"所有文件类型"。

首次运行门禁时，architecture-guard 本身报错（非检查不通过，
是脚本异常退出），原因是新增的 .md 文件匹配逻辑中
正则表达式的贪婪模式导致回溯超时。
将正则改为非贪婪模式后，门禁正常通过。

遗留：当前正则方案在文件名包含特殊字符时可能仍有问题，
暂未验证，因为项目中目前没有这类文件。

## 错误记录
- location: gate.md 第 87 行，正则匹配逻辑
- message: "ReDoS: regex backtracking exceeded 10s timeout"
- handling: 将 `.*` 改为 `.*?`（非贪婪），门禁重新运行通过
- resolved: yes
```

**差距在哪**：坏例子把错误一笔带过，好例子完整还原了错误发生的现象、原因、处理方式和遗留风险。

---

### 对比三：删除型任务

**坏：**
```
## 操作摘要
删除了不再需要的旧文件。

## 修改文件
- [删除] old_config.md
```

**好：**
```
## 操作摘要
删除 BasicKnowledge/agent_workflow/old_config.md。
该文件在 2026-02-15 的重构中被 workspace_structure.md 取代，
内容已完全迁移，不再有任何文件引用它。

删除前已确认：
1. 全局搜索无任何指向 old_config.md 的引用
2. git log 确认最后修改者是本项目的 agent
3. 文件内容与 workspace_structure.md 对应章节无差异

备份位于 git HEAD（sha: b4e5f6a），可通过 git checkout 还原。

## 修改文件
- [删除] BasicKnowledge/agent_workflow/old_config.md

## 安全性评估
- backup: yes（git HEAD: b4e5f6a）
- recoverability_ok: yes（git checkout b4e5f6a -- BasicKnowledge/agent_workflow/old_config.md）
```

**差距在哪**：坏例子不解释为什么删除，好例子交代了删除的理由、确认过程和恢复路径。删除是不可逆操作——日志中的详细程度应该与风险程度成正比。

## 叙事结构模板

对于大多数任务，以下叙事结构可以作为起点：

```
第一段：做了什么（操作事实，2-3句）
第二段：为什么这样做（决策点和理由，1-2句）
第三段（可选）：遇到了什么问题（失败尝试、错误处理，2-3句）
第四段（可选）：妥协和遗留（如果有，1-2句）
```

这不是必须遵循的模板。复杂任务可能需要更多段落，简单任务可能只需要一段。但信息的流向应始终是：**事实 → 理由 → 问题 → 遗留**。
