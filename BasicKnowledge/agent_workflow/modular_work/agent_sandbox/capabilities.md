# 沙箱内无所不能——Agent 的全权限操作模型

边界之内，agent 是这片领地的王。

这不是一句口号，而是架构设计的必然结论：如果你给了 agent 一个任务，又在任务内部处处设卡，它就会绕路——留下隐患、留下脆弱逻辑、留下你根本不知道的副作用。**半吊子授权比全授权更危险。**

全权限是授权的完整性，不是放任。

---

## 一、全权限的含义

Sandbox 是一个有边界的工作空间。边界之外，agent 没有权限。边界之内，agent 拥有**完整的所有权**。

"完整的所有权"意味着：

- 它可以读任何文件，写任何文件，删任何文件
- 它可以启动进程，也可以杀掉进程
- 它可以安装依赖，修改配置，破坏一个结构再把它重建
- 它不需要在每一步回来问人："我可以这样做吗？"

这是它的工作台。工人在车间里不需要每拿一把锤子就申请一次许可。

真正重要的权限控制发生在**边界上**，而不是工作台内部。

---

## 二、能力清单

### 文件系统操作

Sandbox 内的文件系统对 agent 完全开放。无论是源代码、配置文件、临时产物还是构建输出，agent 都有**读、写、删除、重命名、修改权限**的能力。

这包括：创建新目录结构、重构模块布局、清理中间产物、原地编辑文件而不生成副本。它不需要一个"建议修改"的中间状态——修改就是修改。

### 进程管理

agent 可以 **fork 子进程**、执行外部命令（exec）、向运行中的进程发送信号（kill）、以及将长期服务以 **daemon** 方式挂在后台。

构建工具、测试框架、代码分析器、语言服务器——这些都是 agent 可以随时拉起、随时终止的工具，不需要人类的操作系统界面介入。

### 网络（Sandbox 内）

在 sandbox 的网络边界内，agent 可以**监听本地端口**、在内部服务之间互相通信、调用白名单范围内的外部 API。

这允许 agent 自己拉起一个临时 HTTP 服务做冒烟测试，或者通过内部消息总线协调多个子 agent。网络不是障碍，是工具。

### 计算资源

Sandbox 分配了明确的 **CPU 和内存配额**。在配额内，agent 可以自由支配——并行运行多个编译任务、加载大型模型权重、执行密集的静态分析。

配额本身是边界，agent 不需要在配额内还要额外申请"可以跑这个进程吗"。

### 工具调用

注册到 sandbox 的**工具和插件**，agent 可以直接调用，无需每次等待人类确认。

工具注册是一次性的授权决策，由人类在 sandbox 配置阶段完成。之后 agent 调用 linter、测试运行器、格式化工具、文档生成器，都是它的内部操作，和它直接写文件没有本质区别。

---

## 三、全权限的前提：可逆性设计

全权限不等于鲁莽。

**负责任的全权限**意味着：在做一件破坏性操作之前，agent 应优先建立可撤销路径。

- **快照**：在大规模重构前保存当前状态
- **事务性写入**：先写临时文件，验证通过后再原子替换
- **备份**：删除之前先复制

这不是约束，是工程素养。agent 拥有破坏再重建的能力，但明智的 agent 总是让"撤销"成为可能——不是因为规则要求，而是因为它理解**可逆性是长期可靠性的基础**。

全权限 + 可逆性设计 = 强大且可信的操作模型。

---

## 四、与 Gate 的关系

有人会问：全权限了，Gate 还有意义吗？

Gate 的意义从未改变，只是它的本质不同于你想象的。

**Gate 不是能力限制器。** 它不是"你无权做这件事，所以我拦你"。Gate 是**质量检查点**——它检查的是：这件事做得对不对，而不是能不能做。

agent 在 Gate 处停下，不是因为没有权限继续，而是因为它**自己知道这里需要外部验证**。它做完了一个阶段的工作，主动提交检查，等待 pass/fail 信号，然后继续或回滚。

这是成熟系统的标志：**能力无限，自律有度**。Gate 是 agent 对外宣布"我在这里对齐"的时刻，不是系统对 agent 说"你到此为止"的铁栏。

两者的关系是协作，不是对抗。
