# Agent Sandbox：边界是自由的前提

## 一个反直觉的命题

大多数人认为 sandbox 是对 agent 的**限制**。这个认知是错的。

没有边界的 agent 不是"更自由"的 agent，而是一个没人敢授权、没人敢相信的 agent。Sandbox 的真正作用是**解放**——只有当运行边界足够清晰，人类才能放心地把完整的决策权交给 agent，而不是在每一步都插手干预。

---

## 为什么边界是必须的

没有 sandbox 的 agent 在真实环境中会发生什么？

**场景一**：agent 在清理临时文件时路径判断出错，`rm -rf` 误删了生产环境的配置目录。没有文件系统隔离，这个操作无法被拦截，也无法被回滚。

**场景二**：agent 调用外部 API 时遇到速率限制，决定自动重试。在没有网络边界的情况下，它触发了付费接口的超额调用，账单在几分钟内累积到数千美元。

**场景三**：agent 读取工作目录时，意外扫描到了上层目录中的 `.env` 文件，并将其内容写入了一个公开可访问的输出文件。secrets 就这样泄露了。

这三个场景的共同点：**agent 不是在作恶，它只是在"合理地"完成任务**。问题出在它的能力范围超出了人类的预期，而没有任何边界机制在"预期之外"触发阻断。

边界不是枷锁。边界是**防止意外成为灾难的缓冲层**。

---

## 什么是 Agent Sandbox

Agent Sandbox 是一个具有**明确资源边界**的运行环境。它的核心设计原则只有一条：

> **在边界内，agent 拥有完整权限；在边界外，agent 没有任何权限。**

这个"边界"是多维的：

- **文件系统边界**：agent 只能读写指定目录，无法访问宿主机的其他路径
- **网络边界**：只允许访问白名单内的域名或 IP，出站流量受控
- **计算资源边界**：CPU 使用率、内存占用、执行时长均有上限，防止失控的递归或死循环耗尽系统资源
- **API 调用边界**：对外部服务的调用频率和总量设有硬性限制，避免意外的费用爆炸

Sandbox 不关心 agent **如何**使用这些资源——那是 agent 的内部决策。它只关心 agent 的行为是否**超出了约定范围**。

## 结尾洞见

Sandbox 的本质是一份**信任合约**。

人类在合约中写明边界：这块资源你可以用，那块资源你不得碰。Agent 在合约范围内全权负责：怎么做是你的事，结果由你承担。

这个合约一旦成立，双方都获得了自由——人类不需要盯着每一步操作，agent 不需要在每个决策点等待审批。这才是 Agent Workflow 里"无人值守"真正得以实现的前提。

没有 sandbox，就没有信任。没有信任，就没有真正的自动化。
