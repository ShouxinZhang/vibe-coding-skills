# Agent Sandbox 安全边界

Sandbox 的存在不是为了约束 agent，而是为了让 agent 能在一个可信边界内大胆行动。

边界清晰，agent 才能放手做事；边界模糊，任何操作都是潜在的安全事故。

---

## 一、威胁模型

在讨论防御之前，先看清攻击面。Sandbox 面临的威胁不是抽象的——它们有具体的攻击路径和真实的案例。

**沙箱逃逸**是最严重的威胁类型。容器逃逸通常利用内核漏洞或特权配置错误，让容器内的进程获得宿主机权限；**符号链接攻击**则更隐蔽——在 Sandbox 内创建指向宿主机路径的符号链接，再通过文件读写越过边界。两者的共同点是：一旦成功，Sandbox 的隔离就形同虚设。

**资源耗尽**是另一类无声的攻击。`fork bomb` 可以在秒级内耗尽进程表，让系统无响应；恶意代码持续写磁盘同样可以将宿主机的可用空间归零，影响范围远超 Sandbox 本身。这类攻击不需要权限提升，只需要执行机会。

**信息泄露**往往来自配置疏失。最典型的场景：将宿主机的 `~/.ssh`、`~/.aws` 或环境变量文件挂载进了容器，而 agent 执行的代码有权限读取它们。凭据一旦泄露，Sandbox 的其他防护措施都失去意义。

**供应链攻击**是现代软件环境下难以完全消除的风险。agent 在 Sandbox 内安装依赖时，恶意包可能在 `install` 阶段执行任意代码——提取环境变量、建立反向连接、修改文件系统。包名仿冒（typosquatting）让这类攻击的门槛极低。

## 二、权限最小化原则

进入 Sandbox 之前，先做减法。

**Linux Capabilities** 将传统的 `root` 超级权限分解为数十个独立能力。绝大多数 agent 任务只需要基础文件读写和网络访问，不需要 `CAP_SYS_ADMIN`、`CAP_NET_RAW` 或 `CAP_PTRACE`。在启动容器时，默认丢弃所有 capability，再按需添加——而不是反过来。

**seccomp**（Secure Computing Mode）在系统调用层面设置过滤器。agent 的工作负载通常只涉及有限的系统调用集合；将 `ptrace`、`mount`、`unshare` 等高风险调用加入黑名单，即使代码存在漏洞，也大幅压缩了可利用的攻击面。Docker 默认的 seccomp profile 封锁了约 44 个系统调用，这是一个可参考的起点。

**AppArmor / SELinux** 提供基于策略的强制访问控制（MAC）。与 DAC（自主访问控制）不同，MAC 的策略由系统管理员定义，进程本身无法绕过。为 Sandbox 定制一份最小化 AppArmor profile，明确列出允许访问的路径和网络操作，能够在代码逻辑之外建立第二道防线。

权限最小化不是让 agent 无法工作，而是让"出错的代价"尽可能小。

## 三、审计与可观测性

安全事件发生时，能否还原现场，决定了损失是"可挽回的教训"还是"永久的黑箱"。

**系统调用审计**是第一层。`auditd` 或 eBPF-based 工具（如 Falco）可以记录 Sandbox 内每一个系统调用——谁调用了 `execve`，传入的参数是什么，调用时间是什么。这不是监控 agent 的行为，而是建立事后取证的基础数据。

**网络流量可观测**是第二层。所有出入 Sandbox 的网络包都应当可被记录和分析。agent 的正常工作流所需的目标 IP 和端口是有限的；当发现非预期的出站连接时，这是一个需要立即审查的信号。

**文件变更追踪**是第三层。在任务执行前后对文件系统做快照，或实时记录 `inotify` 事件，可以精确重建 agent 在 Sandbox 内做了哪些修改。结合 Git diff，可以区分"正常产出"和"可疑写入"。

三层数据的价值不在于实时告警，而在于**事后还原**。一次完整的事件调查需要时间线、涉及路径和触发原因——审计日志提供的正是这些。

## 四、Agent 应有的安全意识

安全机制的最后一道防线，是 agent 的主动配合。

**不探测边界**。Sandbox 的限制是有意设计的，不是系统 bug。当某个文件路径无权访问、某个系统调用被拒绝时，正确响应是报告失败并上报，而不是尝试绕路或寻找替代方法继续推进。探测边界的行为本身就是一个需要被记录的异常信号。

**不在 Sandbox 外写文件**。所有任务产出都应当落在约定的工作目录内。发现自己需要访问 Sandbox 外的路径时，正确做法是暂停任务，通过正式渠道请求权限扩充，而不是自行寻找挂载点或符号链接。

**发现异常主动上报**。当 agent 在执行过程中观察到非预期行为——某个依赖包在安装时触发了网络请求、某个文件路径莫名可访问、资源使用量出现突刺——这是应当立即写入日志并通知人类的事件，而不是默默忽略继续执行。

**安全意识的本质，是 agent 理解自己行为的边界，并主动维护它。** 一个能够识别异常并主动收手的 agent，比一个"任何情况下都尽力完成任务"的 agent 更可信，也更安全。
