# 防污染设计：让 Agent 在模块边界内持续工作

## 一、这件事解决什么业务问题

当 agent 长时间在一个真实代码库里连续执行任务时，最常见的失败不是"代码不会写"，而是**工作区被逐步污染**：

- 临时文件和实验产物散落在错误目录
- 本应局部生效的修改扩散成跨模块改动
- 上下文越来越臃肿，后续任务基于过期信息决策
- Hook 触发链不可见，修复动作与检查动作相互覆盖

短期看，这些问题不一定立刻阻断任务。长期看，它们会显著拉高维护成本，让团队对自动化失去信任。

本模块的目标是：把"不污染工作区"从经验要求变成一套可执行设计。

---

## 二、四个控制面

防污染不是单点技术，而是四个控制面的联动：

- **提示词工程（Prompt Engineering）**：约束 agent 的决策边界与禁止动作
- **上下文工程（Context Engineering）**：控制 agent 能看到什么、按什么顺序看到
- **Workflow 设计（Workflow Design）**：把任务拆成可验证的短闭环，避免大步跨模块
- **Hooks 设计（Hook Design）**：在关键生命周期节点自动执行边界检查与状态同步

四者缺一不可：

- 只有 Prompt，没有 Context，规则会被过期信息误导
- 只有 Context，没有 Workflow，任务仍可能大范围漂移
- 只有 Workflow，没有 Hooks，执行中会逐步偏离设计
- 只有 Hooks，没有 Prompt，Hook 会变成补救而非预防

## 四、使用建议

推荐把本文作为 `agent_workflow` 的执行侧配套文档来读：

1. 先用 `workspace_structure.md` 建立"哪里可以改"的空间认知
2. 再用 `gate.md` 明确"什么情况必须停"
3. 然后结合 `hooks.md` 确保生命周期可观测
4. 最后用本模块把四类控制面组合成日常执行协议

这样形成的不是单次任务技巧，而是可复用的组织级工作方式。
