# 用提示词、上下文、Workflow 与 Hooks 防止工作区污染

## 一、什么叫工作区污染

在模块化工程里，污染不是语法错误，而是**边界失真**：

- 修改落点偏离目标模块
- 无关文件被带入提交范围
- 上下游模块被隐式耦合
- 任务结束后留下不可解释的状态

污染一旦发生，会直接影响交付速度：回归测试范围扩大、代码评审成本上升、定位故障时间增加。

---

## 二、控制面一：提示词工程（Prompt Engineering）

提示词不是"让 agent 更聪明"，而是把组织规则转成可执行约束。

### 1. 提示词必须包含的边界条款

- **目标模块条款**：本次任务允许修改的模块路径
- **禁止动作条款**：禁止新建根目录文件、禁止跨模块搬运逻辑、禁止修改无关配置
- **验证条款**：每个步骤结束必须说明影响文件及理由
- **失败处理条款**：触发门禁失败后必须停止推进并进入修复链

### 2. 推荐结构

```text
任务目标
允许修改范围
禁止修改范围
执行步骤约束
验证与回报格式
失败升级策略
```

### 3. 业务价值

把隐性经验改成显性规则后，团队可以稳定复用同一执行质量，不再依赖个人习惯。

---

## 三、控制面二：上下文工程（Context Engineering）

上下文工程的目标是：**让 agent 在正确的信息边界内决策**。

### 1. 上下文分层

- **L1 架构上下文**：仓库结构、模块边界、放置规则
- **L2 任务上下文**：当前需求、目标文件、验收标准
- **L3 运行上下文**：最新门禁结果、变更清单、临时状态

加载顺序固定为 L1 -> L2 -> L3。先看边界，再看任务，再看执行细节。

### 2. 上下文减脂规则

- 不把历史长对话原样注入后续任务
- 只保留当前步骤必需的信息
- 每完成一个子步骤就刷新一次"当前事实"

### 3. 业务价值

上下文越干净，决策越稳定，返工率越低。

---

## 四、控制面三：Workflow 设计（Workflow Design）

防污染的 Workflow 不追求一步到位，而是追求**小步可验证**。

### 1. 四段式工作流

1. **Locate（定位）**：确认目标模块和允许触达的文件集合
2. **Change（变更）**：只在目标集合内改动
3. **Check（检查）**：运行局部门禁，验证边界未破坏
4. **Sync（同步）**：更新架构索引或文档映射

### 2. 关键约束

- 任一阶段失败，禁止直接跳到下一阶段
- 每个阶段输出结构化结果（文件列表、风险点、下一步）
- 单次循环尽量不跨两个以上模块

### 3. 业务价值

任务被拆成短闭环后，问题会在更早节点暴露，修复成本更低。

---

## 五、控制面四：Hooks 设计（Hook Design）

Hook 负责把上述规则嵌入生命周期，避免执行中漂移。

### 1. 最小 Hook 组合

- `pre-task`：加载架构规则与允许修改路径
- `pre-step`：检查当前步骤目标文件是否在白名单内
- `post-step`：记录变更摘要并触发局部检查
- `on-gate-fail`：冻结后续步骤，进入固定修复链
- `post-task`：清理临时状态并执行完整性检查

### 2. 防污染重点

- Hook 结果必须可观测，不能静默失败
- Hook 逻辑不能在不同脚本中重复实现
- Hook 只能执行约定职责，不能临时扩权

### 3. 业务价值

把防污染从人工提醒变成系统行为，能显著降低人为波动。

---

## 六、四个控制面的协同关系

```text
Prompt 定义规则
    ↓
Context 提供事实
    ↓
Workflow 组织动作
    ↓
Hooks 保证执行不偏航
```

这不是理论组合，而是生产环境里的最小稳定单元。任一环节缺失，污染风险都会明显上升。

---

## 七、落地检查清单

在任务开始前和结束后，至少确认以下事项：

- 是否明确了允许修改路径
- 是否只加载了当前任务必要上下文
- 是否按 Locate -> Change -> Check -> Sync 执行
- 是否启用了 `pre-task`、`pre-step`、`post-step`、`on-gate-fail`、`post-task`
- 是否更新了受影响模块的结构文档

若以上任一项无法满足，应视为流程设计不完整，而不是"先做完再说"。
