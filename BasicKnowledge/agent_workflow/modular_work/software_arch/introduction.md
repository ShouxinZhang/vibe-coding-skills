# 软件工程架构基础：模块化工作的底层支撑

## 一、为什么 agent 需要懂架构

Agent 做的不是玩具任务。它修改真实代码库、调用真实服务、生成真实产物。

一个不理解架构的 agent，面对一个有分层结构的代码库时，会做出这样的决策：在 UI 层直接调数据库、把业务逻辑塞进工具函数、为了"快速实现"把两个模块的职责合并。每一步看起来都在推进任务，但每一步都在悄悄降低系统的可维护性——而且 agent 自己不会察觉，因为它看到的是"代码能跑"。

**能跑不等于正确。** 软件工程架构讲的正是这件事：什么样的结构，能在时间维度上保持正确。

对 agent 来说，理解架构原则不是为了"写出漂亮的代码"，而是为了**不做破坏系统稳定性的事**。这是无人值守工作的基本安全线。

---

## 二、核心概念：架构是什么

### 分层架构（Layered Architecture）

分层架构把系统切成水平的层：**表现层 → 业务逻辑层 → 数据访问层 → 基础设施层**。

每层只做自己职责范围内的事，只向相邻的下层发出请求，不跨层直接对话。业务逻辑层不知道数据库是 PostgreSQL 还是 SQLite，表现层不知道业务规则的细节。

这种隔离的意义是：底层换实现，上层不用改。换了数据库，只要接口不变，业务逻辑层感知不到变化。

### 模块化与职责隔离

模块化是把系统切成垂直的块：每个模块有自己的边界，对外暴露接口，对内隐藏实现。

**高内聚**指模块内部的元素高度相关，彼此服务同一个职责。**低耦合**指模块之间的依赖尽量少、尽量浅，通过接口而不是直接依赖具体实现来通信。

这两个原则的联合效果是：修改一个模块，不会引发连锁的跨模块副作用。这对 agent 来说极其重要——agent 操作的影响范围要可预测，可预测才能可验证。

### 接口契约（Interface Contract）

接口是两个模块之间的协议：我暴露什么、你调用什么、输入输出是什么格式、边界条件如何处理。

接口契约一旦确定，调用方不需要了解被调方的内部实现。它只需要知道"调这个方法、传这个参数、会得到这个结果"。

这对 agent 的意义是：**agent 可以在不读懂整个模块的情况下正确使用它**，只要接口文档是准确的。

### 依赖方向

在健康的系统里，依赖是有方向的。高层模块定义策略，低层模块提供实现，依赖从高层指向低层，**不能反向**。

如果底层工具函数开始引用业务逻辑类，依赖方向就倒了——这意味着一个底层的修改会波及上层，整个系统变成一张互相牵连的网。

## 三、Agent 如何把这些原则用起来

### 任务分解时，沿着模块边界切

当 agent 接到一个复杂任务，需要把它分解成子任务时，最安全的切法是**沿着已有的模块边界分解**，而不是按实现步骤分解。

"修改订单处理流程"不应该被分解成"先改数据库查询、再改 API 响应格式、再改前端展示"——这是按技术层拆，但每步都横跨了多个模块。正确的分解是：先在业务逻辑模块内完成逻辑修改，通过测试；再在接口层做适配；最后在表现层消费新数据。每个子任务的影响范围都被限定在最小的模块边界内。

### 修改代码前，看懂依赖图

在动手修改一个文件之前，agent 需要知道：这个文件被谁依赖？它依赖谁？如果我改了它的接口，影响链会延伸到哪里？

这不是谨慎，这是必要的。盲改代码、没有依赖图意识的 agent，是一个不受控的 agent。

### 不越界动不该动的层

Agent 在一个任务中被允许修改特定模块时，**只动那一层、那一块**。即使它发现相邻层有"可以顺手优化"的点，也不应该动。

这不是因为那个优化不好，而是因为：未经授权的修改超出了任务边界，打破了变更的可追溯性，也可能触发它没有意识到的连锁影响。多做不如不做——在自动化流水线里，**克制是一种能力**。

## 四、本文件夹的定位

`modular_work/` 文件夹记录的是 agent 独立工作的两个基础支柱：

**软件架构基础**（本文所在的 `software_arch/`）讲的是 agent 需要懂的系统结构知识——分层、模块边界、接口、依赖方向。这些知识让 agent 知道"系统是怎么组织的"，从而做出符合架构意图的操作，而不是破坏它。

**Agent Sandbox**（`agent_sandbox/`，同级目录）讲的是 agent 工作的运行时环境——隔离、权限边界、可回滚的操作空间。这些机制让 agent 在不影响生产环境的情况下安全试错。

两者共同构成"agent 有纪律地独立工作"的基础：一个给认知边界，一个给操作边界。缺少任何一个，agent 要么做出架构上错误的决策，要么在错误的环境里做出不可撤销的操作。
