# 隔离机制：模块化架构的物理边界

模块化不只是代码组织的问题。代码层面的分层设计如果没有底层的物理边界支撑，就只是一种君子协定——任何一个 agent 的错误操作都可以绕过它。

**隔离机制**把模块之间的边界从"约定"变成"强制约束"。它是 agent 沙箱（sandbox）的工程实现基础。

---

## 一、进程隔离

### 是什么

操作系统以**进程**为单位分配内存空间。每个进程有独立的虚拟地址空间，无法直接读写其他进程的内存。进程之间若要通信，必须经过 IPC（信号、管道、套接字等）——这些通道本身就是可审计的边界。

在容器化环境中，进程隔离进一步叠加了 Linux namespace，使进程不仅无法读取其他进程的内存，也看不到其他进程的存在。

### agent 不遵守会发生什么

Agent 在同一进程里运行多个相互独立的任务，是最常见的隔离失效场景。一个任务的全局变量污染了另一个任务的执行状态；一个任务崩溃导致整个进程退出，拉倒了无辜的其他任务；资源回收的粒度失去控制——你无法 `kill` 一个"卡住的任务"，只能 `kill` 整个 agent。

### agent 应具备的意识

每个独立任务单元应当运行在独立的进程（或至少独立的子进程）中。**Kill/restart 的粒度等于隔离的粒度**。如果你不能单独重启某个失败的任务而不影响其他任务，说明进程边界没有划好。

## 二、文件系统隔离

### 是什么

文件系统隔离通过**工作目录约束**和**挂载点隔离**实现。每个 agent 任务只能读写被明确授权的路径；容器或 chroot 环境使 agent 无法感知宿主机的真实文件系统结构；**临时文件有明确的生命周期**，任务结束时由运行时负责清理，而不是留给 agent 自我判断。

### agent 不遵守会发生什么

没有工作目录约束的 agent 会在整个文件系统里留下痕迹：临时文件散落在 `/tmp` 以外的位置，任务产物写到了不属于自己的模块目录，日志文件无限增长直到磁盘耗尽。更危险的是：一个任务可以读取另一个任务尚未提交的状态，造成隐式的跨任务数据依赖，这种依赖不会体现在任何接口定义里，只会在某次任务顺序变化后突然爆发。

### agent 应具备的意识

每次操作文件前，先确认路径是否在自己的**授权工作区**内。任何写出目标目录之外的操作，都应当被视为越界行为而非"顺手整理"。临时文件的路径应当使用运行时提供的 `tempdir`，而不是硬编码或自行推断。

---

## 三、网络隔离

### 是什么

网络隔离控制 agent 可以建立的出入连接。**出流量控制**（egress）限制 agent 能访问哪些外部端点，防止数据泄露或意外调用生产接口；**入流量控制**（ingress）限制哪些调用者可以触达 agent 暴露的服务；**服务发现边界**决定 agent 能"看到"哪些其他服务——不在边界内的服务，对 agent 而言不存在。

### agent 不遵守会发生什么

未受约束的 agent 在测试环境里调用了生产环境的 API，写入了真实数据。或者反过来：生产环境的 agent 因为配置漂移，错误地将流量发向了测试服务。端口随意暴露意味着任何有网络访问权限的进程都可以和 agent 通信，接口的身份验证形同虚设。

### agent 应具备的意识

网络访问的**默认策略是拒绝**，而不是允许。Agent 使用的外部端点应当在任务启动时通过环境变量或配置注入，而不是硬编码在代码里。暴露的端口应当明确声明，并且只对声明的调用者开放。

---

## 与 agent_sandbox 的关系

以上三种隔离维度——进程、文件系统、网络——是构建 **agent sandbox** 的三块基石。

Sandbox 不是一个单一的技术组件，它是这三层隔离叠加之后形成的执行环境：agent 在里面可以自由运行，但所有操作的副作用都被限制在预定义的边界内。出了边界，什么都做不了。

| 隔离维度 | sandbox 中的对应保障 |
|---|---|
| 进程隔离 | 任务崩溃不传播，资源回收可精确 |
| 文件系统隔离 | 产物可追踪，临时状态自动清理 |
| 网络隔离 | 外部副作用受控，环境之间互不污染 |

没有这三层的工程实现，"sandbox"只是一个概念标签。把它说清楚，是 agent 架构设计的起点，而不是终点。
