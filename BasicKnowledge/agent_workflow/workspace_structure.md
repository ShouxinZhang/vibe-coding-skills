# Workspace Structure：让 Agent 看懂工作区

## 一、Agent 眼中的工作区

人类打开一个项目，可以通过 README、目录命名、文件注释快速建立直觉。Agent 也能读这些东西，但它缺乏人类的模糊推断能力——当目录结构混乱、文件命名随意、职责边界模糊时，agent 的决策质量会急剧下降。

一个为 agent 设计的工作区，需要做到一件事：**让 agent 在不问人的情况下，就能准确知道每个文件是什么、归属哪个模块、修改后该通知谁**。

---

## 二、工作区结构的三个层次

### 层次一：目录契约（Directory Contract）

目录结构是最强的信号。每个顶层目录都应该有明确的语义，并通过 `AGENTS.md` 或架构文档写死规则。

**原则：**
- 目录命名反映职责，不反映技术实现（`business_logic/` 优于 `utils/`）
- 每个叶模块是自治的——它的代码、测试、文档都在自己内部
- 任何"我不知道放哪里"的文件，说明缺少一个模块，而不是应该随便放

**反模式：**
- 根目录直接堆文件
- `misc/`、`temp/`、`old/` 此类目录存在
- 同类职责分散在多个目录

---

### 层次二：文件级文档（File-level Documentation）

目录契约告诉 agent 宏观结构，文件级文档告诉 agent 每一个具体文件的意图。

推荐做法是在 agent 每次创建或修改文件后，同步更新一份机器可读的元数据，记录：

| 字段 | 内容 |
|------|------|
| `path` | 文件相对路径 |
| `description` | 一句话说明这个文件做什么 |
| `notes` | agent 专用备注（注意事项、依赖关系、禁忌操作） |
| `last_modified` | 最后修改时间 |

这份元数据可以是 SQLite 数据库、JSON 文件，或结构化的 Markdown。关键是 agent 在每次任务开始时能查询，在每次任务结束时能更新。

**典型工作流：**

```
任务开始
  ↓
agent 查询目标文件的文档记录
  ↓
了解该文件的职责与注意事项后再动手
  ↓
任务完成，更新文档记录
  ↓
导出全局 Workspace Map（给下一个 agent 或人类审阅）
```

---

### 层次三：全局地图（Workspace Map）

单个文件的元数据是点，全局地图是面。

Workspace Map 是一份聚合文档（通常是一个 Markdown 文件），展示工作区的全貌：哪些模块存在、每个模块下有哪些关键文件、它们的职责是什么。

这份文档有两个读者：
- **人类**：快速了解项目全貌，做审阅和决策
- **Agent**：在任务开始时加载，建立对工作区的整体感知，避免"视野盲区"

Workspace Map 应该由工具自动生成，不应该手动维护。手动维护的文档会腐烂。

---

## 三、结构稳定性原则

当多个 agent（或多轮任务）操作同一个工作区时，最常见的问题是**结构漂移**——目录结构随任务累积而逐渐偏离设计初衷。

防止结构漂移的核心手段是**架构守卫（Architecture Guard）**：

- 在每次任务结束后，运行一个检查脚本，验证当前目录结构是否符合既定规则
- 发现违规时，以非 0 退出码阻断后续操作，强制人类或 agent 修复
- 规则以机器可读格式（JSON/YAML）存储，不以"大家都知道"的隐性惯例存储

**架构守卫的典型规则示例：**

```json
{
  "forbidden_at_root": ["*.py", "*.ts", "*.js"],
  "required_in_each_module": ["README.md"],
  "forbidden_dirs": ["misc", "temp", "old"],
  "max_depth": 4
}
```

---

## 四、给 Agent 的工作区操作守则

任何运行在此类工作区中的 agent，都应遵守以下规则：

1. **修改文件前，先查文档** — 确认该文件的职责和禁忌操作
2. **新建文件后，立即登记** — 更新文件级文档，说明用途和注意事项
3. **任务结束后，导出地图** — 更新 Workspace Map，保持全局视图最新
4. **不得在模块外创建文件** — 所有文件必须归属某个有明确语义的模块
5. **不得创建无名临时目录** — `temp/`、`bak/`、`old/` 是禁止词，用版本控制替代

这五条规则，是工作区契约的最小集合。它们不依赖 agent 的"良知"，而是通过门禁机制强制执行。
