# 元数据与索引：让日志和文档在长期积累后仍然可找

> 内容越多，真正稀缺的越不是存储空间，而是定位能力。没有元数据和索引，知识库会从资产退化成仓库。

---

## 一、为什么仅靠全文搜索不够

全文搜索很重要，但它只能回答“哪些文本包含这些词”。它无法稳定回答更复杂的问题：

- 过去一年里哪些文档仍然有效
- 哪些记录来自同一个项目、同一个人、同一个主题
- 哪些日志对应同一次发布、同一条规则变更、同一轮复盘
- 哪些内容已经废弃，但仍被高频引用

这些问题需要的不只是文本，而是文本之外的**结构化描述信息**，也就是元数据。

---

## 二、元数据到底在补什么

元数据本质上是在回答四类问题：

- 这条内容是什么类型
- 它属于谁、关于谁、服务谁
- 它处于什么生命周期状态
- 它和别的内容有什么关系

对于日志和文档系统，常见的元数据包括：

| 元数据 | 用途 |
|------|------|
| `type` | 区分日志、文档、摘要、规则、观察记录 |
| `status` | 标识草稿、有效、已废弃、已归档 |
| `created_at` / `updated_at` | 支持时间过滤与排序 |
| `source` | 区分手写、同步导入、自动生成、外部抓取 |
| `owner` / `subject` | 指向项目、人、主题或对象 |
| `tags` | 提供轻量分类与交叉组织 |
| `version` | 说明当前内容处于哪个版本节点 |
| `references` | 标识引用关系、依赖关系、上游来源 |

没有这些字段，系统只能把知识当作一堆文本处理。文本可以阅读，但很难治理。

---

## 三、索引不是一张表，而是一层能力

很多人把索引理解成数据库里的某个技术细节。对知识系统而言，索引更像一层中介能力：它决定哪些问题能快速回答，哪些问题只能靠人工翻找。

常见的索引维度有三类：

### 3.1 标识索引

用于快速定位某个明确对象，例如：

- 文档 ID
- 版本号
- 事件序列号

这类索引服务的是“精确找回”。

### 3.2 属性索引

用于按字段过滤和组合，例如：

- 按时间范围筛选
- 按主题、标签、状态筛选
- 按来源、作者、对象筛选

这类索引服务的是“结构化定位”。

### 3.3 语义索引

用于处理“我说不准关键词，但我知道大概想找什么”的问题。向量检索属于这一类。

它适合补全文本召回能力，但不能替代标识索引和属性索引。

---

## 四、日志和文档各自更依赖什么索引

日志系统更依赖：

- 时间索引
- 顺序索引
- trace / session / transaction 关联索引
- 对象 ID 与事件类型索引

文档系统更依赖：

- 版本索引
- 状态索引
- 主题、标签、引用关系索引
- 语义检索索引

两者混合后，知识系统才既能回答“我现在想找什么”，也能回答“这条内容属于哪段历史”。

---

## 五、元数据设计的三个常见错误

### 5.1 字段太少，最后什么都筛不出来

系统刚启动时，大家总觉得只要有标题和正文就够了。等内容多了，才发现无法按状态、来源、时间、主题组织。

### 5.2 字段太多，最后谁都不愿意填

反过来，如果元数据表单太重，记录成本会上升，系统会快速腐烂。好的元数据设计，不是把一切可能性都预埋进去，而是先保住最常用的几个维度。

### 5.3 标签泛滥，分类失控

标签很灵活，但太灵活也容易失控：同义词、近义词、大小写差异、含糊标签会迅速侵蚀索引质量。

因此，标签适合做轻量补充，不适合承担全部分类责任。

---

## 六、一个实用判断

当系统里的内容超过某个阈值后，决定检索体验的往往不再是算法本身，而是前期有没有把元数据和索引层设计清楚。

换句话说，搜索做得再聪明，也无法完全弥补结构信息的缺失。
