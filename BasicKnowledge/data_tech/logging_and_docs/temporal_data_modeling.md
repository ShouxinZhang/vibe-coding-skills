# 时间数据建模：让记录真正成为历史

> 没有时间，数据只是静止的碎片；有了时间，数据才开始具备历史。

---

## 一、为什么时间是日志和文档系统的第一维度

日志库、文档库、知识库看起来像三套不同系统，但它们有同一个前提：所有内容都发生在时间中。

同一条记录，在不同时间语境下意义完全不同：

- 一条待办，在创建当天是意图，在两个月后可能是拖延证据
- 一段会议纪要，在刚写完时是事实记录，在半年后可能变成决策依据
- 一条情绪日志，在当下是表达，在一年后可能是模式分析样本

因此，系统不能只记录“内容是什么”，还要记录“它何时出现、何时生效、何时被替代、何时失效”。

---

## 二、三种必须区分的时间

### 2.1 发生时间

事实在现实世界真正发生的时间。例如“下午三点开会”“上周开始失眠”。

### 2.2 记录时间

内容被系统写入的时间。很多内容是补录、迁移、同步后才进入系统，因此记录时间并不等于发生时间。

### 2.3 生效时间

某个状态、规则、判断从何时开始成立。例如“从今天起采用新的标签体系”。

这三者如果混为一谈，历史就会失真。系统至少要意识到：

- 发生时间回答“世界何时如此”
- 记录时间回答“系统何时知道”
- 生效时间回答“结论何时成立”

## 三、四类基础时间对象

面向日志和文档系统，一个实用起点是把对象拆成四类：

| 对象类型 | 核心问题 | 典型例子 | 时间形态 |
|------|------|------|------|
| 事件（Event） | 发生了什么 | 发布、回滚、同步、修改 | 时间点 |
| 状态（State） | 某段时间内是什么样子 | 草稿、生效、归档、废弃 | 时间区间 |
| 文档（Document） | 某个时刻写下了什么 | 说明书、纪要、规则文档 | 版本序列 |
| 观察（Observation） | 在某时刻测量到了什么 | 评分、统计、摘要结论 | 时间点或时间窗 |

这种划分的价值在于：后续查询、检索、回放和治理的方式会因此变得清晰。

---

## 四、为什么追加式记录通常优于覆盖式记录

覆盖很省事，但会直接抹掉变化过程。对日志与知识系统来说，过程往往比结果更有价值。

更稳妥的原则通常是：

- 能追加就不要覆盖
- 更正建模为新事实，而不是抹掉旧事实
- 允许不确定时间，而不是强行伪精确

追加式记录的好处是：

- 系统知道原来记录过什么
- 系统知道为什么被修正
- 系统知道现在应该采用哪个版本

## 五、一个最小时间模型

如果目标是做一个能长期使用的个人数据或知识系统，一个足够实用的最小模型通常包括：

| 字段 | 用途 |
|------|------|
| `id` | 稳定标识 |
| `type` | 对象类型 |
| `occurred_at` | 事实发生时间 |
| `recorded_at` | 系统写入时间 |
| `valid_from` | 生效开始时间 |
| `valid_to` | 生效结束时间 |
| `source` | 数据来源 |
| `subject` | 数据主体 |
| `status` | 生命周期状态 |
| `supersedes` | 指向被替代对象 |

这个模型不追求一步到位，但能保住最关键的三件事：

- 数据不是只有当前值
- 时间事实不会因为修改而丢失
- 同一条记录能被重新放回时间线理解

## 六、时间粒度决定系统能回答什么问题

不同粒度，支持的问题完全不同：

| 粒度 | 适合内容 | 典型问题 |
|------|------|------|
| 秒/分钟 | 操作日志、设备事件 | 某个触发器什么时候发生 |
| 小时 | 会话、活动块 | 下午做了哪些事 |
| 天 | 日记、复盘、日报 | 最近一周状态如何 |
| 周/月 | 阶段总结、项目进展 | 这个月的主线是什么 |
| 季度/年 | 战略变化、身份迁移 | 我这两年最核心的变化是什么 |

好的时间建模不追求“越细越好”，而追求**与问题匹配的粒度层次**。

---

## 七、一个核心判断

时间字段不是给报表凑数用的。对日志、文档和知识系统来说，时间建模决定了系统究竟只能保存信息，还是能够保存历史。
