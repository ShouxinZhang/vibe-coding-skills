# RAG：给知识装上大脑

> 存储信息是一回事，理解它是另一回事。RAG 做的事情，是在你庞大的个人知识库和真正智能的回答之间，架起一道桥梁。

---

## 一、向量：语义的数学表达

要理解 RAG，首先要理解**向量嵌入（Embedding）**。

想象你把所有词语、句子、段落，都投射到一个巨大的多维空间里。在这个空间中：

- "孤独"和"寂寞"距离很近
- "焦虑"和"紧张"距离很近
- "开心"和"焦虑"距离很远
- "苹果（水果）"和"苹果（公司）"在不同的上下文中，可能落在空间的不同位置

这个"距离"，是通过训练语言模型学到的——它见过海量的人类文字，学会了哪些词语在哪些语境下倾向于一起出现，并把这种共现关系压缩成一串数字（向量）。

```python
# 概念示意（非实际代码）
embed("我今天很开心") → [0.12, -0.34, 0.89, ..., 0.07]  # 1536 维向量
embed("今天心情很好") → [0.11, -0.31, 0.91, ..., 0.09]  # 非常接近！
embed("今天很焦虑")   → [0.45,  0.72, 0.03, ..., 0.56]  # 距离较远
```

**向量嵌入把"语义相似"变成了可计算的"空间距离"。** 这是 RAG 一切的基础。

## 二、RAG 的完整流程

RAG 系统分为两个阶段：**构建阶段**和**查询阶段**。

### 构建阶段（离线）

```
原始文档（日记、笔记、对话……）
         ↓
    文本切分（Chunking）
    把长文档切成适当大小的片段
         ↓
    向量化（Embedding）
    每个片段 → 一串数字
         ↓
    存入向量数据库
    （如 pgvector、Chroma、Qdrant）
```

切分策略很关键：切太短，每段失去上下文；切太长，相似度计算不精准。常见做法是按段落切分，并在相邻片段之间保留一定的**重叠（overlap）**，保证上下文连贯。

### 查询阶段（在线）

```
你的问题："我去年为什么那么迷茫？"
         ↓
    问题向量化
         ↓
    向量数据库中检索最近邻
    返回 Top-K 个最相似片段
         ↓
    构建 Prompt：
    "根据以下背景信息回答问题：
     [片段1] [片段2] [片段3]
     问题：我去年为什么那么迷茫？"
         ↓
    LLM 生成回答
```

## 三、检索质量：RAG 系统的核心难题

RAG 系统最容易被忽视、但最影响结果质量的环节，是**检索（Retrieval）**本身。

**问题1：向量相似 ≠ 语义相关**

"苹果公司的发布会"和"苹果树的生长周期"向量未必很近，但"失眠"和"焦虑"可能非常接近，即使在你的问题里，这两件事是不相关的。

**问题2：Top-K 的困境**

检索 5 个片段还是 20 个片段？片段太少可能遗漏关键信息，太多则会稀释重要内容，还有可能超出 LLM 的上下文窗口。

**问题3：时间衰减**

同样的语义，三年前写的和昨天写的，在你的问题语境下重要性不同。纯粹的语义检索不区分新旧。

**常见改进策略**：

| 策略 | 做法 | 效果 |
|------|------|------|
| 混合检索 | 向量检索 + 关键词检索（BM25）结合 | 兼顾语义和精确匹配 |
| 重排序（Reranking） | 用第二个模型对检索结果重新打分 | 提升精准度 |
| 元数据过滤 | 先按时间/标签过滤，再做向量检索 | 减少噪声 |
| 查询改写 | 让 LLM 先把问题改写成更适合检索的形式 | 提升召回率 |

## 四、个人知识库的 RAG 实践

对于个人数据场景，RAG 最有价值的应用是打通**碎片化的个人知识**。

**场景一：跨时间的自我对话**

你的日记、笔记、想法散落在不同时期、不同工具里。通过 RAG，你可以问：

> "我两年前对创业是什么态度？和现在有什么不同？"

系统会找出相关片段，让 LLM 做对比分析。

**场景二：知识图谱的自动生成**

你读过的书、做过的笔记、收藏的文章——RAG 可以帮你找出它们之间的隐性联系：

> "我关于'注意力'这个话题有哪些散落的思考？"

**场景三：决策辅助**

> "我上次面对类似的两难选择时，最后是怎么决定的？结果如何？"

RAG 让你的过去成为可查询的顾问，而不只是封存的档案。

## 五、RAG 的边界与风险

RAG 非常强大，但它不是万能的，甚至存在一些需要警惕的风险：

**幻觉（Hallucination）**  
当检索结果不够充分时，LLM 会"补全"——它会生成听起来合理、但实际上并不存在于你原始记录中的内容。对于个人数据，这尤其危险：**虚假的记忆比遗忘更有害**。

缓解方式：要求 LLM 在回答时标注引用来源，并提供原始片段供核对。

**选择性失忆**  
RAG 只能找到它索引了的内容。如果某段重要的记忆从未被数字化，或者切分方式恰好割裂了关键上下文，它就永远不会出现在检索结果里。

**回音壁效应**  
如果你的数据库主要是你自己的想法，RAG 会不断强化你已有的认知，而不是给你提供新的视角。它是一面镜子，不是一扇窗。

## 六、向量数据库：RAG 的基础设施

向量数据库是专门设计用于存储和检索向量的数据库，它的核心操作是**近似最近邻搜索（ANN）**。

几个常见选择：

| 工具 | 特点 | 适合场景 |
|------|------|---------|
| **pgvector** | PostgreSQL 扩展，SQL + 向量一体 | 个人项目，结合 SQL 使用 |
| **Chroma** | 轻量，本地优先，易上手 | 快速原型，个人知识库 |
| **Qdrant** | 高性能，支持元数据过滤 | 数据量较大，需要精细控制 |
| **Weaviate** | 内置多模态支持 | 混合文字、图像的场景 |

对于个人数据管理场景，**pgvector** 是一个很好的起点——它让你用同一套 PostgreSQL 数据库同时处理 SQL 查询和向量检索，正好对应第二章描述的"SQL 负责骨架，RAG 负责血肉"的协同架构。
