# 漏斗与留存：把用户旅程翻译成 SQL

> 大多数业务问题，本质上都不是“用户有没有来”，而是“用户来了之后走到了哪一步，又在哪一步离开了”。漏斗和留存，是理解这件事的两把基础尺子。

---

## 一、漏斗回答的是过程流失

一个典型产品流程可能是：访问首页 → 注册 → 激活 → 首单 → 复购。漏斗分析的目标，是看每一步有多少人继续往前走，又有多少人在这一层流失。

如果某一步骤转化明显异常，管理动作就会变得清晰：

- 注册转化低，优先优化注册流程
- 激活转化低，优先优化新手引导
- 首单转化低，优先检查价格、支付和信任机制

---

## 二、漏斗依赖事件建模

要做漏斗，前提不是 SQL，而是事件记录足够清楚。至少要知道：

- 谁触发了事件
- 事件发生在什么时候
- 事件属于哪一步

```sql
SELECT user_id,
       event_name,
       occurred_at
FROM user_events;
```

如果连步骤定义都混乱，漏斗只会把混乱可视化，而不会帮你解释问题。

---

## 三、留存回答的是关系是否持续

留存关注的是：用户第一次出现之后，是否还会在后续周期再次回来。

例如一个月留存分析，通常会看：

- 本月新增用户里，有多少人在下个月再次活跃
- 不同渠道的新用户，留存率有何差异
- 某次改版后，新用户首月留存是否改善

这类问题的本质，是衡量业务关系有没有持续建立，而不只是完成一次交易。

---

## 四、漏斗和留存为什么常被一起看

因为它们回答的是同一条用户旅程的两个阶段：

- 漏斗看首次转化能否发生
- 留存看关系建立后能否延续

一个产品即使漏斗做得很好，如果留存很差，增长仍然不可持续；反之，留存很好但入口转化差，增长规模也起不来。

---

## 五、一个实践提醒

漏斗和留存的误区，通常不在 SQL 技术，而在业务定义：

1. 每一步是否真的有清晰完成标准？
2. 同一个用户能否重复进入漏斗？
3. 留存口径是自然周、自然月，还是注册后第 N 天？

先把定义讲清，再写 SQL，结论才值得被用于管理。
