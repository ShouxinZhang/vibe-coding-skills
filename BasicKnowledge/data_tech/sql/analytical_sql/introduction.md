# 分析型 SQL：从记录走向决策

> 业务系统先要能记录事实，随后才会面对一个更难的问题：怎样从这些事实里提炼出趋势、风险和机会。分析型 SQL 解决的，正是这一步。

---

## 一、本模块的定位

基础查询关注“把数据取出来”，分析型 SQL 关注“把结论提炼出来”。二者的差异，不在于语法是否更复杂，而在于目标是否更靠近决策。

分析型 SQL 最常服务三类业务动作：

- **监控趋势**：收入、活跃、转化是否在上升或下滑
- **识别结构**：头部用户、长尾商品、异常群体分别是谁
- **还原路径**：用户从看到、点击、注册到付费，卡在了哪一步

## 三、为什么单靠 GROUP BY 不够

基础聚合当然重要，但业务问题很快会超出“按某个维度统计一次”的范围。例如：

- 这个用户比上周增长了多少？
- 这个渠道在同类渠道里排第几？
- 哪一步流程流失最严重？

这些问题都要求同时保留明细和上下文。也正因为如此，窗口函数、时间分析和漏斗分析才值得单独成章。

## 五、一个提醒

分析型 SQL 很强，但它只能帮你看见“发生了什么”。至于“为什么发生”和“应该怎么做”，仍然需要业务理解、实验设计和管理判断来补全。
