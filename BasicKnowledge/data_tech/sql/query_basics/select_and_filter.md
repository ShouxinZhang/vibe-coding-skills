# 观察的边界：SELECT、WHERE、ORDER BY

> 查询的第一步，是先决定自己到底想看什么。数据从来不会主动告诉你重点，重点永远来自你的选择。

---

## 一、SELECT 不是“取列”，而是定义视角

```sql
SELECT date, mood, note
FROM daily_log;
```

这条语句看似只是“取三列”，实际上是在声明：此刻我关心的是时间、情绪和解释文本，而不是地点、设备、天气、标签或其他字段。SELECT 决定了你用什么视角看现实。

如果你写的是：

```sql
SELECT *
FROM daily_log;
```

它当然方便，但也意味着你没有真正完成选择。业务上，这通常会带来三类问题：

- 返回了根本不需要的字段，浪费性能
- 把注意力分散在次要信息上
- 在接口或报表里暴露不该公开的数据

---

## 二、WHERE 定义了“这次讨论的世界”

```sql
SELECT date, mood
FROM daily_log
WHERE date >= CURRENT_DATE - INTERVAL '30 days'
  AND mood IS NOT NULL;
```

WHERE 的作用不是“多加几个条件”，而是明确这次分析到底讨论哪个世界。是全部历史，还是最近 30 天？是全量用户，还是新注册用户？是成功订单，还是所有订单？

当 WHERE 写得模糊，分析结果往往也会模糊。

### 一个典型例子

```sql
SELECT COUNT(*)
FROM orders
WHERE status = 'paid';
```

如果业务定义里“退款后仍然保留 paid 状态”，那这条 SQL 算出的就不是“有效成交订单”。问题不在 SQL 引擎，而在你是否真的理解业务口径。

---

## 三、NULL 不是空字符串，而是未知

SQL 里最容易制造误判的概念之一，就是 `NULL`。

```sql
SELECT COUNT(note)
FROM daily_log;
```

这并不是在统计总记录数，而是在统计 `note` 不为 `NULL` 的记录数。如果你想统计总数，应该写：

```sql
SELECT COUNT(*)
FROM daily_log;
```

业务上，`NULL` 往往不是技术细节，而是流程信号：

- 没填地址，是用户没提供，还是系统没采集到？
- 没有退款时间，是因为没退款，还是流程失败了？
- 没有备注，是因为事件不重要，还是记录习惯中断了？

一个成熟的数据系统，会把 `NULL` 当成需要解释的状态，而不是随手忽略的空值。

---

## 四、ORDER BY 与 LIMIT 决定谁先被看见

```sql
SELECT title, like_count
FROM posts
ORDER BY like_count DESC
LIMIT 5;
```

排序和截断本质上是在做优先级分配。你是先看最新的，还是先看最重要的？先看增长最快的，还是先看最异常的？

不同排序会塑造不同管理动作：

- 按时间倒序：适合监控和巡检
- 按金额倒序：适合风险识别和头部分析
- 按差值倒序：适合发现波动与异常

LIMIT 则是在现实约束下承认：人类不可能同时处理全部结果，所以系统必须帮助人先看最值得看的部分。

---

## 五、最小实践准则

写任何一条查询前，都先回答这 4 个问题：

1. 我到底想看哪几个字段？
2. 这次分析的边界是什么？
3. 有没有 `NULL`、状态口径或时间口径的问题？
4. 输出结果应该按什么顺序呈现？

如果这 4 个问题答不清，SQL 很可能只是“能跑”，而不是“有用”。
