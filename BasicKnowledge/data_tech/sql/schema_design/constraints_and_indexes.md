# 约束与索引：一手管正确，一手管速度

> 任何数据库系统都要同时面对两个现实：一方面，数据不能乱；另一方面，查询不能慢。约束和索引，正是解决这两个现实的基础机制。

---

## 一、约束是业务规则的最低执行层

应用代码当然可以做校验，但如果数据库本身没有约束，系统总会在某个边缘路径里把脏数据写进去。约束的价值，在于它提供了最后一道防线。

常见约束包括：

| 约束 | 业务含义 |
|------|---------|
| `NOT NULL` | 这个字段不能缺失 |
| `UNIQUE` | 这个值不能重复 |
| `PRIMARY KEY` | 这是主身份标识 |
| `FOREIGN KEY` | 这条记录必须依附于另一条合法记录 |
| `CHECK` | 这个值必须满足某个规则 |

例如：

```sql
CREATE TABLE users (
    id BIGINT PRIMARY KEY,
    email TEXT NOT NULL UNIQUE,
    age INT CHECK (age >= 0)
);
```

这段定义的意义不只是“建表成功”，而是在明确系统底线：邮箱不能为空、不能重复、年龄不能是负数。

---

## 二、索引不是越多越好

索引能显著加快查询，但每加一个索引，写入成本和存储成本也会跟着上升。因此，索引设计不是“能加就加”，而是要围绕高频访问路径做取舍。

```sql
CREATE INDEX idx_orders_user_id_created_at
ON orders (user_id, created_at DESC);
```

这类复合索引适合这样的业务场景：经常查询“某用户最近的订单”。它服务的是明确场景，而不是抽象的“以后可能有用”。

---

## 三、什么时候该加索引

通常可以优先考虑这些列：

- 高频 `WHERE` 条件列
- 高频 `JOIN` 键
- 高频排序列
- 高频范围查询列

反过来，这些场景就要谨慎：

- 基数极低的布尔字段
- 很少被查、但高频更新的字段
- 只为了“感觉安全”而提前加的索引

索引应该由访问模式驱动，而不是由焦虑驱动。

---

## 四、约束和索引是两套不同逻辑

一个常见误区，是把两者混为一谈。它们的关注点并不相同：

- **约束** 在问：这条数据合法吗？
- **索引** 在问：这条数据能被多快找到？

在业务系统里，两者必须同时存在。只有约束没有索引，系统会越来越慢；只有索引没有约束，系统会越来越不可信。

---

## 五、一个实用原则

每建一张核心表，至少应该主动想清楚三件事：

1. 哪些字段绝对不能缺失？
2. 哪些关系绝对不能断裂？
3. 哪些查询会成为未来的高频路径？

想清这三件事，约束和索引的 80% 决策就已经有了依据。
