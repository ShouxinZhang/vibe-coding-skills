# 从业务对象到数据表：实体与关系

> 在你写第一条 `CREATE TABLE` 之前，真正需要先回答的问题是：这个系统里到底有哪些独立存在的东西，它们之间又是什么关系。

---

## 一、实体是什么

实体（Entity）可以理解为业务里的稳定对象。它通常满足三个条件：

1. 可以被唯一标识
2. 有自己的生命周期
3. 有一组相对稳定的属性

例如：用户、订单、商品、课程、评论，通常都属于实体。因为它们都需要被独立创建、更新、追踪和引用。

---

## 二、关系是什么

关系描述的是实体之间如何连接。最常见的是三种：

| 关系类型 | 例子 | 常见落表方式 |
|---------|------|-------------|
| 一对一 | 用户与实名认证 | 一方持有唯一外键 |
| 一对多 | 用户与订单 | 多方持有外键 |
| 多对多 | 用户与标签、学生与课程 | 通过中间表连接 |

例如，一个用户有多个订单：

```sql
CREATE TABLE users (
    id BIGINT PRIMARY KEY,
    name TEXT NOT NULL
);

CREATE TABLE orders (
    id BIGINT PRIMARY KEY,
    user_id BIGINT NOT NULL REFERENCES users(id),
    total_amount NUMERIC NOT NULL
);
```

这里的关键不是语法，而是业务判断：订单是独立实体，但它归属于某个用户，因此外键放在订单表里。

---

## 三、多对多关系为什么不能偷懒

一个常见错误，是把多对多关系塞成逗号分隔字符串：

```text
user.tags = 'vip,newsletter,trial'
```

这样短期看似方便，长期会带来严重问题：

- 无法可靠检索
- 无法建立约束
- 无法高效统计标签分布
- 很难扩展关系上的额外信息

更合理的做法是使用中间表：

```sql
CREATE TABLE user_tags (
    user_id BIGINT NOT NULL REFERENCES users(id),
    tag_id BIGINT NOT NULL REFERENCES tags(id),
    created_at TIMESTAMP NOT NULL,
    PRIMARY KEY (user_id, tag_id)
);
```

如果关系本身还有属性，例如加入时间、角色、权重，那么中间表就更加必要。

---

## 四、识别实体的一个简单方法

判断某个概念是不是独立实体时，可以连续问这 3 个问题：

1. 它是否需要被单独查询？
2. 它是否需要被其他表引用？
3. 它是否有自己的状态变化过程？

如果答案大多是“是”，就不应该把它仅仅当成某张表里的附属字段。

---

## 五、表设计其实是在划业务边界

一个成熟的 schema 从来不只是数据库问题，它其实是在替业务做边界声明：

- 什么是系统真正关心的对象
- 哪些对象是核心，哪些只是附属属性
- 哪些关系值得被长期追踪

这一步做得越清晰，后面的查询和分析成本就越低。
