# Schema Design：先定义世界，再查询世界

> 一套业务系统的长期成本，往往不是由接口数量决定的，而是由底层表结构决定的。表设计一旦含混，后面的查询、分析、迁移、治理都会越来越昂贵。

---

## 一、本模块回答什么问题

很多人第一次接触 SQL 时，会把重心放在查询语法上。但在真实系统里，影响更深远的往往不是“查询怎么写”，而是“表最初怎么设计”。

好的 schema 设计会带来三种业务收益：

- **正确性**：系统能尽量阻止不合法的数据进入
- **可扩展性**：新需求到来时，不至于牵一发而动全身
- **可分析性**：后续查询和报表可以自然地建立在清晰结构上

## 三、一个常见误解

很多团队在业务早期喜欢把字段不断往一张表里塞，因为这样“上线快”。短期看确实如此，但当系统需要支持更多角色、更多流程、更多统计口径时，这种便宜会迅速变成债务。

Schema 设计真正影响的是未来：

- 你是否能可靠对账
- 你是否能追溯一条记录为什么出现
- 你是否能在不重写系统的前提下支持新业务

## 五、一个判断标准

如果你面对一张表时，无法用一句话说清它到底在记录什么，那它大概率设计得有问题。因为清晰的表，本质上是清晰的业务边界。
