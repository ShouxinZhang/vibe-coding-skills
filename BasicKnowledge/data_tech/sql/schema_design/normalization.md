# 规范化：减少冗余，不让数据自相矛盾

> 规范化的目标不是追求教科书上的优雅，而是尽量避免同一份事实被重复保存、重复修改，最后悄悄互相冲突。

---

## 一、为什么冗余会成为系统隐患

假设你把用户姓名和地址直接复制到每一条订单记录里。刚开始似乎很方便，但很快就会出现问题：

- 用户改地址时，历史订单要不要同步改？
- 同一用户的不同订单里，姓名拼写不一致怎么办？
- 统计用户最新信息时，到底信哪一条？

这种问题的根源，不在查询，而在于同一事实被重复保存了太多次。

---

## 二、规范化到底在做什么

规范化（Normalization）的核心，是把“不同层次的事实”拆到不同表里，让每个事实只在最合适的地方出现一次。

最常见的直觉可以概括为：

1. 一列里不要塞多个值
2. 一张表尽量只描述一个清晰主题
3. 非主键字段应该依赖主键，而不是依赖别的非主键字段

例如，把订单和用户拆开：

```sql
CREATE TABLE users (
    id BIGINT PRIMARY KEY,
    name TEXT NOT NULL,
    address TEXT
);

CREATE TABLE orders (
    id BIGINT PRIMARY KEY,
    user_id BIGINT NOT NULL REFERENCES users(id),
    total_amount NUMERIC NOT NULL
);
```

这样，用户地址只维护一次，订单只关心交易事实。

---

## 三、规范化带来的三种收益

| 收益 | 说明 |
|------|------|
| 更新一致性 | 同一事实只改一处，不容易冲突 |
| 存储清晰 | 每张表的职责边界更明确 |
| 分析可信 | 查询不容易被重复字段污染 |

但规范化不是免费午餐。表拆得越细，查询时越依赖 JOIN，写查询和优化查询的难度都会上升。

---

## 四、什么时候需要适度反规范化

如果某些字段几乎每次查询都要联表取出，而且变化极少，可以考虑适度反规范化，例如冗余存一份快照字段、汇总表或宽表。

典型场景包括：

- BI 报表需要稳定读取聚合结果
- OLAP 分析更重视查询速度而非事务更新
- 某些展示页面要求极低延迟

但反规范化必须是**有意识的工程取舍**，而不是因为一开始没想清楚。最糟糕的系统，不是规范化不足，而是团队根本不知道哪些冗余是故意保留的，哪些只是历史遗留。

---

## 五、实践中的判断标准

面对一份 schema，如果你发现下面这些现象，就应该重新审视规范化程度：

- 同一个字段语义在多张表反复出现
- 改一个业务规则，需要批量修 3 张以上的表
- 同一指标在不同查询里频繁算出不同结果

规范化的本质不是拆表，而是减少未来的混乱。
