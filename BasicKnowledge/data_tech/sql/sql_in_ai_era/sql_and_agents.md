# SQL 与 Agent：自动化系统为什么仍然需要关系数据库

> Agent 擅长规划、调用工具和生成文本，但只要任务一长、步骤一多，它就会暴露出一个弱点：状态容易漂移。SQL 的价值，正是在这里提供一层不会随上下文波动而失真的工作记忆。

---

## 一、Agent 为什么需要状态层

一个真实 Agent 工作流，通常会经历这些阶段：

- 接收目标
- 拆分步骤
- 调用工具
- 记录中间结论
- 失败后恢复
- 最终审计

如果这些状态只存在上下文窗口里，任务一长就会遗失、混淆或重复执行。SQL 提供的，是一种持久、结构化、可回放的状态存储。

---

## 二、什么样的数据适合交给 SQL

适合放进 SQL 的，通常是这些精确状态：

- 任务表：任务 ID、状态、优先级、负责人
- 步骤表：步骤顺序、输入、输出、耗时、重试次数
- 工具调用表：调用了什么工具、结果如何、是否成功
- 审计日志表：什么时间做了什么决定

```sql
CREATE TABLE agent_tasks (
    id BIGINT PRIMARY KEY,
    status TEXT NOT NULL,
    created_at TIMESTAMP NOT NULL,
    updated_at TIMESTAMP NOT NULL
);
```

这类表的意义不是复杂，而是稳定。它们让 Agent 可以在长流程里始终知道自己处在什么位置。

---

## 三、SQL 给 Agent 带来的三项关键能力

### 可恢复

任务中断后，可以根据数据库状态继续执行，而不是完全重新开始。

### 可审计

出了问题，可以追溯是哪个步骤、哪次工具调用、哪条中间结论导致了偏差。

### 可并发协调

多个 Agent 或多个 worker 协作时，需要共享同一套事实状态，避免重复处理和冲突写入。

---

## 四、为什么不能只靠向量库或纯文本日志

向量库擅长语义检索，纯文本日志擅长叙述，但它们都不适合承担精确状态管理：

- 任务是否完成，需要明确布尔或状态枚举
- 某步骤是否已经执行过，需要严格去重
- 某次更新是否成功，需要事务保障

这些问题天然更适合 SQL，而不是概率式检索系统。

---

## 五、一个现实结论

越是想把 Agent 做成真正可用的生产系统，就越不能只依赖模型本身。模型负责推理，SQL 负责记账。二者分工明确，系统才可能长期稳定。
