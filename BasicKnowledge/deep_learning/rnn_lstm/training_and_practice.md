# RNN 系列的训练与实践

> 原理听懂以后，真正落地时最容易出问题的，通常不是“公式忘了”，而是训练不稳、序列填充处理错、推理表现和训练表现脱节。这篇专门讲这些工程问题。

## 一、梯度裁剪：先把最常见的爆炸问题压住

即便用了 LSTM 或 GRU，梯度爆炸依然可能发生，尤其是在：

- 序列较长
- 学习率偏大
- 网络堆叠较深

最常见的止损办法是梯度裁剪（Gradient Clipping）：

```python
torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=5.0)
```

它的意思不是改变梯度方向，而是给梯度大小设一个上限，避免某一步更新过猛把训练直接打坏。

经验上，`max_norm` 常见取值在 `1.0` 到 `5.0` 之间。

## 二、Dropout：别把时序信息自己打断了

很多人学完前馈网络后，会自然地想：过拟合了，那就上 Dropout。

思路没错，但在循环网络里要更谨慎。

如果你在每个时间步都随机丢不同的神经元，等于自己把时间上的连续信息流打碎了。模型本来就要靠状态一路往前传，你再每一步都换一套随机掩码，时序稳定性会明显变差。

因此在 RNN 系列里，更常见的是：

- 对层与层之间使用 dropout
- 或使用同一序列内共享掩码的变体做法

PyTorch 的 `nn.LSTM` / `nn.GRU` 中，`dropout` 参数默认作用在层间连接上，而不是单层内部每个时间步都乱丢。

## 三、初始化：让模型一开始别走得太歪

循环网络对初始化比很多普通网络更敏感，因为它的状态和梯度都要沿时间反复传播。

实践里常见两个经验：

### 1. 循环权重适合用正交初始化

正交初始化（Orthogonal Initialization）通常更有利于保持数值稳定，减少一开始就出现严重梯度衰减或爆炸的概率。

### 2. LSTM 的遗忘门偏置常设为正值

这是一个非常经典的小技巧。把遗忘门偏置初始化成正数，相当于让模型在训练初期更倾向于“先别急着忘”，这样通常更容易学到长期依赖。

## 四、Padding 与 Packing：别让模型把补齐位也当真数据学进去

真实数据里的序列长度往往不一样。为了批量训练，通常会先把短序列补齐到同样长度，也就是 padding。

问题在于：

如果模型把这些补齐出来的 `<PAD>` 位置也当成正常时间步继续更新状态，就会产生两类问题：

- 无意义计算变多
- 模型学到很多本不该存在的模式

因此在工程里通常会用：

- 掩码（mask）
- 或 `pack_padded_sequence` / `pad_packed_sequence`

来告诉框架：哪些位置是真实数据，哪些只是为了凑批次形状。

## 五、Teacher Forcing：训练时很顺，推理时未必顺

在序列生成任务里，训练时有一个非常常见的技巧叫 Teacher Forcing。

它的做法是：

- 训练时，解码器每一步都喂入“真实上一个词”
- 而不是喂入“模型自己刚刚生成的词”

这样训练通常会快很多，因为模型不至于在一开始就被自己的错误输出带偏。

但问题也随之而来：推理时你拿不到真实上一个词，只能喂模型自己刚生成的结果。

这就造成训练场景和推理场景不一致。

## 六、Exposure Bias：为什么推理时错误会滚雪球？

这种“训练时总有人扶着，推理时得自己走”的差异，通常被称为暴露偏差（Exposure Bias）。

它带来的典型现象是：

- 前几步一旦错了
- 后面就会持续基于错误上下文继续生成
- 结果错误像滚雪球一样放大

这也是很多早期 Seq2Seq 模型在长生成任务上表现不稳的重要原因。

工程里常见的缓解方式包括：

- 逐步降低 teacher forcing 比例
- 让模型在训练中部分暴露于自己的预测结果
- 用更贴近推理目标的训练策略做补偿

## 七、一些非常实用的排查思路

如果一个 RNN / LSTM / GRU 项目训练得不对劲，通常优先看这几件事：

### 1. 损失突然发散

优先检查：

- 学习率是不是太高
- 有没有做梯度裁剪
- 初始化是不是太激进

### 2. 训练集效果还行，推理一长就崩

优先检查：

- teacher forcing 比例是不是过高
- 是否存在明显 exposure bias
- 序列过长而模型容量不足

### 3. 模型学得很慢，几乎没进展

优先检查：

- padding 处理是否正确
- 数据预处理是否让大量有效信息被截断
- 循环层规模是否太小，或者正则过强

## 八、今天为什么 RNN 不再常做主力，但仍然值得学？

从工业主流看，RNN 系列现在已经很少担当最核心的大模型骨架，原因很明确：

- 难并行
- 长距离依赖能力有限
- 在大规模训练上不如 Transformer 系列高效

但它依然值得学，原因也很明确：

### 1. 它是现代序列建模的起点

很多后来更强的结构，都是在修补 RNN 暴露出来的问题。

### 2. 它把“状态”“记忆”“时间展开”这些概念讲得最直接

这些概念在更现代的模型里虽然换了形式，但没有消失。

### 3. 轻量场景里它仍有实用价值

在资源受限、小模型、流式处理或某些传统业务系统里，GRU / LSTM 仍然是可用方案。

## 九、小结

RNN 系列的工程实践，核心不是炫技巧，而是守住几条底线：

- 梯度别炸
- 时序信息别被自己打断
- padding 别污染真实训练
- 训练和推理差异要有意识地处理

如果把这些基本面守住，RNN、LSTM、GRU 就能在它们擅长的任务范围里稳定工作；如果守不住，即使原理背得很熟，模型也可能在训练时表现得一团糟。
