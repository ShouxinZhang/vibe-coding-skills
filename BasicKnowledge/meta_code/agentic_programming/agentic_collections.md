# Agentic Collections：把数组、栈、队列、堆、图翻译成 Agent 世界

## 一、为什么 Agent Programming 需要集合抽象

单个 agent 被定义清楚之后，真正的复杂性才刚刚开始。企业系统很少只运行一个 agent。它们真正管理的是很多 agent、很多任务、很多中间状态、很多等待中的动作。

这时，语言层面就不能只讨论“一个 agent 是什么”，还必须讨论“很多 agent 如何被组织”。这正是集合抽象的作用。

传统编程里，数组、栈、队列、堆、图并不是语法装饰，而是组织复杂性的最小工具。Agent Programming 也一样。如果没有集合抽象，系统只能靠散乱的 prompt、脚本和 workflow 节点临时拼接，最后演化成不可治理的流水线。

## 二、数组与列表：Agent 池和有序工作集

传统数组或列表的核心价值，是把多个同类元素按顺序组织起来，支持遍历、批处理和局部更新。

在 agent 世界里，它首先对应两类东西：

| 经典结构 | Agentic 对应物 | 业务作用 |
|------|----------------|---------|
| Array / List | Agent Pool | 管理一组同类 agent 实例，例如一组研究 agent 或一组审计 agent |
| Array / List | Ordered Worklist | 管理一组待处理任务、待审片段、候选方案 |

这层抽象的价值很直接：一旦 agent 成为“池”而不是“单点角色”，组织就能开始做批量分发、并行评估、投票聚合和冗余校验。

因此，agentic list 不再只是“元素排成一排”，而是“一组可被统一调度的执行单元”。

## 三、栈：任务分解栈、反思栈与回退栈

栈的本质是后进先出。它特别适合表达局部上下文和最近决策。

在 agent 系统里，栈至少有三种高价值用法：

| 栈类型 | 含义 | 解决的问题 |
|------|------|-----------|
| 任务分解栈 | 当前子任务如何一层层展开 | 让复杂问题可以递归拆解 |
| 反思栈 | 最近几轮判断与修正链条 | 让 agent 的自我修正有局部记忆 |
| 回退栈 | handoff 或执行失败后的回退路径 | 让恢复不是整条链路重来 |

这说明 stack 在 agent 编程里不只是数据访问方式，更接近“局部推理控制面”。

## 四、队列：任务队列、事件队列与消息收件箱

队列的核心价值是先进先出，它天然适合表达等待、排队和异步流转。

在 agent 系统里，最常见的是三种队列：

| 队列类型 | 含义 | 业务作用 |
|------|------|---------|
| Task Queue | 待领取和待执行任务 | 支撑批量自动化和 worker 式处理 |
| Event Queue | 外部事件、工具回调、定时触发 | 支撑连续运行和事件驱动 |
| Inbox Queue | agent 之间的消息收件箱 | 支撑异步协作而不强制同步对话 |

如果没有 queue，很多系统看上去像多 agent，实际上只是多个同步函数挤在同一轮里。真正的 queue 会把时间差、等待状态和消费顺序正式纳入语言语义。

## 五、堆：优先级调度器

堆在经典编程里常用于优先级队列。在 agent 世界里，这个意义更重要。

因为 agent 系统真正稀缺的从来不是“任务数量”，而是“执行预算”。模型调用、上下文窗口、人工审核、工具配额都不是无限的。所以系统必须持续回答一个问题：下一步最值得执行的是什么。

这就让 heap 对应到：

- 紧急度优先队列
- 风险优先队列
- 价值优先队列
- 恢复优先队列

换句话说，agentic heap 本质上就是调度器的语言级前身。它让系统不是按到达顺序盲跑，而是按收益、风险和时效去决定行动顺序。

## 六、图：依赖图、Handoff 图、状态转移图

图是 Agent Programming 里最自然的集合结构，因为 agent 之间最重要的关系本来就不是“挨在一起”，而是“连在一起”。

在业务系统里，最常见的图结构包括：

| 图类型 | 含义 | 业务价值 |
|------|------|---------|
| Dependency Graph | 哪些任务依赖哪些前置结果 | 控制顺序与阻塞关系 |
| Handoff Graph | 任务如何在多个 agent 之间流转 | 让协作路径可审计、可重放 |
| State Transition Graph | 系统在什么条件下从一个阶段进入下一阶段 | 让流程可治理、可恢复 |
| Capability Graph | 哪些 agent 能调用哪些工具、访问哪些资源 | 让权限与能力关系可视化 |

这也是为什么图结构会成为当前开源框架最先成熟的一层。

## 七、LangGraph 说明了什么，没说明什么

以 `2026-03-12` 拉取的 LangGraph 仓库为例，当前提交是 `93a0dfec082da2deeee3878983f2535e34ec4e7e`。从它的实现可以看到几件很清楚的事：

1. 它已经把 `StateGraph`、条件分支、共享状态、Pregel 风格执行、checkpoint 持久化做成了稳定框架。
2. 它很强地证明了“图 + 状态 + 运行时”这条路是成立的。
3. 但它主要还是 graph orchestration framework，而不是完整的 agentic language。

更具体地说，LangGraph 现在最成熟的是图：

- `StateGraph` 负责定义节点和边
- branching 负责条件路由
- checkpoint 负责状态快照与 thread 级恢复
- runtime/context 负责运行期上下文注入

但它并没有把数组、栈、队列、堆这些集合抽象系统性提升为一套 agentic standard collections。今天这些结构大多仍然由开发者在 Python 或 TypeScript 宿主语言里自行组织。

这不是缺点，而是阶段信号。它说明 open source 已经把“图”做成了第一类结构，接下来该补的是更广义的 collections 层。

## 八、除了五大经典结构，还会出现什么

Agent Programming 很可能还会额外长出几类比传统语言更重要的集合：

| 新集合 | 作用 |
|------|------|
| Agent Registry | 按身份、类型、版本管理 agent 实例 |
| Memory Index | 组织可检索记忆与经验条目 |
| Consensus Set | 管理多 agent 投票和一致性判断 |
| Snapshot Table | 记录不同阶段的状态快照与恢复点 |

这些集合之所以重要，是因为 agent 世界的复杂性不只来自数据量，更来自执行体之间的关系和时间维度。

## 九、为什么这层值得单独研究

从业务角度看，agentic collections 的意义是把“很多 agent 在一起工作”从一种现象，升级成一种可编程能力。

有了这层抽象之后，团队才能真正回答下面这些问题：

- 这一组 agent 是池、队列，还是图
- 什么时候应该批处理，什么时候应该排队
- 什么时候该按优先级调度，而不是按到达顺序执行
- 多个 agent 的协作关系应该被当作图保存，还是当作消息流瞬时处理

当这些问题开始被语言层面表达出来时，agent 编程才会真正像一门语言，而不是一组框架习惯。
