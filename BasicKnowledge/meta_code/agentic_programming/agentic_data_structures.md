# Agentic Data Structures：Agent 编程的基础结构

## 一、为什么 Agent 编程首先需要数据结构

每一代编程范式成熟之前，都会先出现一批“能跑但难治理”的系统。今天的 agent 工程正处在这个阶段。很多团队已经能做出会拆任务、会调工具、会写代码的 agent，但一旦进入长期运行、多任务继承、多人协作、失败恢复，就会迅速暴露结构问题。

根本原因并不神秘：大家在操作 agent，却还没有稳定的数据结构来描述 agent。

没有数据结构，就没有统一序列化方式；没有统一序列化方式，就没有可靠保存、迁移、恢复、比较、审计与组合。所谓“agentic data structures”，本质上是在回答一个最朴素的问题：一个 agent 作为程序单元，到底由哪些字段组成。

## 二、一个最小 Agent 结构体

下面这个结构不是标准答案，但足够作为最小研究骨架：

```text
Agent {
  identity
  objective
  state
  working_context
  memory
  tools
  permissions
  plan_stack
  event_log
  guardrails
  relations
  lifecycle_status
}
```

它不是为了追求形式好看，而是为了让系统知道：什么东西属于 agent 本体，什么东西只是外部输入。

## 三、每个字段解决什么问题

| 字段 | 业务作用 | 缺失后的后果 |
|------|---------|------------|
| `identity` | 给 agent 一个稳定可引用的身份 | 无法恢复、追踪、交接 |
| `objective` | 定义当前职责与目标边界 | agent 容易漂移，产出不可审计 |
| `state` | 保存当前内部状态与进度 | 无法断点续跑，行为只剩瞬时输出 |
| `working_context` | 定义当前轮执行可见的信息切片 | 输入过载或信息缺口，导致判断失真 |
| `memory` | 提供跨轮次和跨任务的可回读经验 | 每次都从零开始，无法积累 |
| `tools` | 描述可调用能力集合 | 能力边界不清，编排成本高 |
| `permissions` | 控制能做什么、不能做什么 | 自动化风险无法隔离 |
| `plan_stack` | 记录当前任务分解与下一步意图 | 中断后难以继续，也难以人工接手 |
| `event_log` | 留下过程证据，支撑回放与审计 | 出错后只能猜，不能定位 |
| `guardrails` | 定义校验、门禁和停止条件 | 小错误会扩散成系统级事故 |
| `relations` | 描述与其他 agent、任务、资源的关系 | 多 agent 协作会失去结构 |
| `lifecycle_status` | 表示创建、激活、挂起、退役等阶段 | 运行时管理无法规范化 |

这张表的真正意义，是把“agent 会做很多事”压缩成“agent 有哪些稳定字段”。一旦字段稳定，接口、存储、迁移、审计、训练、治理都能围绕它建立起来。

## 四、从单体 Agent 到组合结构

单个 agent 只是第一层。真正的 Agent Programming 还会出现更高层结构：

| 组合结构 | 作用 | 对应的业务场景 |
|------|------|---------|
| Agent Registry | 管理所有 agent 的身份、版本、能力标签 | 组织级 agent 资产管理 |
| Task Queue | 管理待执行任务与领取关系 | 长流程自动化、批量任务处理 |
| Shared Blackboard | 让多个 agent 在同一工作面协作 | 研究、写作、分析等并行协作 |
| Handoff Graph | 记录任务在多个 agent 之间如何流转 | 多角色流水线、审批链路 |
| Memory Index | 给记忆建立检索与回写入口 | 跨项目经验复用 |

这说明 agentic data structures 不只是“定义一个结构体”，而是会逐步长出整套集合、索引、图和队列结构。它和传统数据结构的区别在于，这些结构承载的不只是静态数据，还承载“谁能行动、何时行动、依据什么行动”。

## 五、为什么这件事有研究价值

如果没有这一层结构化抽象，未来的大多数 agent 系统都会面临同一条天花板：

1. 能 demo，但很难成为基础设施。
2. 能完成一次任务，但不能稳定继承上一轮成果。
3. 能做单体实验，但无法支撑组织级协作和审计。

从业务角度看，agentic data structures 的价值，是把 agent 从“临时劳动力”升级为“可管理资产”。只有当 agent 能像数据库记录、对象实例、任务节点一样被结构化描述时，企业才可能真正把它们纳入长期生产系统。

## 六、边界与下一步

这篇文章只回答“agent 里应该有哪些字段”，没有完全回答“这些字段如何变化”。字段变化涉及两个更深的问题：

- 状态和上下文如何区分
- 身份和引用如何稳定存在
