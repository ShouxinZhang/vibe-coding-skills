# Agentic Type System：给 Agent 编程加上边界与契约

## 一、为什么 Agent Programming 迟早需要类型系统

把 agent 当作新的程序单元之后，下一步问题就不再是“能不能跑”，而是“能不能安全地组合、传递和复用”。

传统编程语言里，类型系统承担的是秩序功能。它不是为了让程序更学术，而是为了在执行之前就回答几个关键问题：这个东西是什么、它能和谁连接、它不能做什么、如果连错了会在哪里出问题。

Agent Programming 也会遇到同样的问题，而且更严重。因为 agent 不只是数据，还带着权限、记忆、目标、工具和行为倾向。没有类型系统，所有 agent 都会退化成“一个大而模糊的角色名”，最终带来三类业务后果：

1. 任务能分发，但分给谁并不可靠。
2. 能力能复用，但边界不清，错误传播更快。
3. 系统能扩容，但一旦 agent 数量增长，治理成本会超过收益。

## 二、Agent 的“类型”到底在约束什么

Agent 类型系统不是简单照搬 `int`、`string` 那种数据分类。它约束的是一个 agent 的多维边界：

| 类型维度 | 要回答的问题 | 业务意义 |
|------|-------------|---------|
| 角色类型 | 这是研究型、执行型、审计型还是协调型 agent | 决定它适合承担什么职责 |
| 能力类型 | 它会检索、规划、编码、审查还是调用外部系统 | 决定能接什么任务 |
| 权限类型 | 它可读什么、可写什么、可调用哪些工具 | 决定风险边界 |
| 状态类型 | 它需要怎样的状态字段与记忆结构 | 决定能否恢复、迁移、复用 |
| 输入输出类型 | 它接受什么上下文，产出什么结果或句柄 | 决定能否稳定接入流水线 |
| 生命周期类型 | 它是一次性、会话型、长期驻留型还是可派生型 | 决定运行成本和运维方式 |

也就是说，Agent 类型系统真正约束的不是“它长什么样”，而是“它在组织里应该如何被安全使用”。

## 三、为什么角色名远远不够

今天很多团队的做法是给 agent 起名字：researcher、coder、reviewer、planner。这个阶段有用，但很快就会失效。

因为名称不是类型。名称只是标签，类型才是约束。一个叫 `researcher` 的 agent，可能有只读权限，也可能能直接写数据库；可能只会检索总结，也可能会发起外部 API 调用。仅靠名字，系统并不知道它究竟能不能接某个任务。

如果没有类型系统，就会出现一种典型伪装：两个 agent 看上去名字相近，实际能力边界完全不同；或者名字不同，实际却在做大量重复工作。组织会误以为自己在复用 agent，实际上只是在复制混乱。

## 四、一个最小类型系统骨架

Agent 类型系统至少需要四层：

| 层级 | 作用 | 示例 |
|------|------|------|
| 基础类型 | 描述 agent 的角色与核心职责 | `ResearchAgent`、`ExecutionAgent`、`AuditAgent` |
| 能力接口 | 描述可提供的能力集合 | `CanPlan`、`CanSearch`、`CanWriteCode` |
| 权限修饰符 | 描述可访问资源和操作强度 | `ReadOnly`、`RepoWrite`、`ExternalCall` |
| 生命周期修饰符 | 描述实例的存在方式 | `Ephemeral`、`SessionScoped`、`Persistent` |

把它组合起来，一个 agent 才会从“名称”变成“合同”。例如：

```text
ResearchAgent
  implements CanSearch, CanSummarize
  with ReadOnly
  and SessionScoped
```

这个表达的价值，不在于形式，而在于它让调度器、运行时、审计系统和人工管理者都能用同一套语言理解这个 agent。

## 五、类型系统能提前阻断什么错误

从业务角度看，类型系统最大的价值不是表达优雅，而是提前拦错。

| 错误类型 | 没有类型系统时会怎样 | 有类型系统后能做什么 |
|------|--------------------|-------------------|
| 任务错配 | 把需要写权限的任务交给只读 agent，流程运行到中途才失败 | 在调度前直接拦截 |
| 权限越界 | 把敏感工具误分配给不该拥有的 agent | 在声明层就隔离 |
| 输出不兼容 | 上游产出的是摘要，下游需要的是可执行计划 | 在接口层提前发现 |
| 生命周期错用 | 把一次性 agent 当作长期代理，导致状态丢失 | 在实例管理前明确限制 |
| 组合失控 | 多个 agent 表面可连，实际边界冲突 | 在编排时做兼容性检查 |

这说明 Agent 类型系统的商业意义很清楚：它不是锦上添花，而是组织规模化用 agent 的前置门槛。

## 六、类型系统不会是纯静态的

与传统编程不同，agent 的很多边界具有动态性。权限可能升级，记忆可能增长，能力可能通过工具装配临时扩展。因此，Agent 类型系统更可能是“静态约束 + 运行时校验”的混合体。

这意味着它会有两类职责：

1. 在执行前给出尽量多的结构保证。
2. 在执行中持续验证 agent 是否仍符合其声明边界。

所以，Agent 类型系统最终不会孤立存在，它一定要和运行时绑在一起。类型系统负责“声明你是谁”，运行时负责“确保你按声明活着”。

## 七、对未来语言设计意味着什么

如果未来真的出现 agent 编程语言，那么类型系统至少要支持以下能力：

- 声明 agent 的角色类型与能力接口
- 为权限和生命周期添加修饰符
- 检查不同 agent 之间的连接兼容性
- 区分“返回数据”与“返回 agent 句柄”这两类不同结果
- 支持对派生、代理、组合后的新 agent 重新赋型

这会让 agent 语言从“会描述角色”升级为“能校验组织结构”。而这恰好是企业真正需要的东西：不是更多会说话的 agent，而是更多可预测、可审计、可安全调度的 agent。
