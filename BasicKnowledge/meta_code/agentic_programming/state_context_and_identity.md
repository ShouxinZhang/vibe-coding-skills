# 状态、上下文与身份：Agent 的值与地址

## 一、为什么必须把这三个概念拆开

很多 agent 系统一出问题，团队第一反应是“上下文不够”或“记忆坏了”。这往往不是真正的问题。真正的问题是，系统从一开始就没有把状态（state）、上下文（context）和身份（identity）分开。

当这三者混在一起时，系统表面上还能运行，但会出现一连串业务后果：

- 失败后不知道该恢复哪个部分
- handoff 时不知道该交接哪些内容
- 审计时不知道某次行为属于哪个 agent
- 迁移时不知道哪些东西该复制，哪些东西只能引用

所以，想把 agent 当成新的程序单元，第一步不是让它更聪明，而是让它的三层语义更清楚。

## 二、三者分别是什么

| 概念              | 定义                                     | 更接近传统编程里的什么 |
| ----------------- | ---------------------------------------- | ---------------------- |
| 状态（State）     | agent 当前内部持有、会随执行变化的数据   | 当前值                 |
| 上下文（Context） | 当前轮执行时被加载进可读工作面的信息切片 | 活跃工作内存           |
| 身份（Identity）  | 让系统持续找到同一个 agent 的稳定句柄    | 地址 / 引用锚点        |

这三者的关系可以这样理解：

- 状态回答“它现在是什么样”。
- 上下文回答“它这一轮看到了什么”。
- 身份回答“系统怎么确认还是它”。

用户提出的“value 和 address”类比在这里就有了更精确的落点：`state` 更接近值，`identity` 更接近地址或引用，而 `context` 是运行时为了当前决策临时映射进来的工作集。它既不是全部状态，也不是永久身份。

## 三、为什么上下文不等于状态

这是今天很多系统最容易犯的错误。

上下文是“当前可见”，状态是“当前持有”。一个 agent 的长期记忆、历史计划、角色边界、权限配置都可能属于状态，但不会在每一轮都完整加载到上下文窗口中。相反，一次临时检索结果、外部文档片段、用户刚追加的说明可能进入上下文，却不应直接写回长期状态。

如果不区分两者，就会出现两种极端：

1. 把所有状态都塞进上下文，导致窗口臃肿、成本飙升、判断变慢。
2. 把所有上下文都写回状态，导致系统长期污染、记忆失真、错误永久化。

因此，Agent Programming 的一个关键能力不是“存更多”，而是“知道什么该持久，什么该临时激活”。

## 四、为什么身份不是可有可无的元数据

很多人把身份当作日志字段，好像只用于排查问题。实际上，身份是 agent 可编程性的根。

没有身份，就没有稳定引用；没有稳定引用，就没有真正的变量语义。你不能可靠地说“把这个任务交给刚才那个研究 agent 继续做”，因为系统根本不确定“刚才那个”是谁。

身份至少承担四个业务职能：

| 身份职能     | 价值                                  |
| ------------ | ------------------------------------- |
| 恢复锚点     | 让系统在中断后找到同一个 agent        |
| 权责归属     | 让行为、结果、错误能被追责和审计      |
| 协作引用     | 让其他 agent 或流程节点可以显式引用它 |
| 生命周期管理 | 让创建、复制、挂起、退役有明确对象    |

这也是为什么 agent 编程迟早会需要自己的“引用语义”。未来的系统可能不仅传递数据，还会传递 agent handle、snapshot、fork、lease 等更接近运行时的概念。

## 五、生命周期：让值与地址真的动起来

一旦把状态、上下文和身份拆开，生命周期就能被正式描述：

1. 创建：分配身份，初始化最小状态。
2. 激活：为某轮任务装载上下文。
3. 执行：在上下文中读取信息，并更新状态。
4. 挂起：保存必要状态，释放当前上下文。
5. 恢复：基于身份重新找到该 agent，并重新装载工作面。
6. 派生：复制部分状态或共享部分引用，形成新的 agent 分支。
7. 退役：冻结或归档状态，停止继续调度。

这个过程与传统变量生命周期不同，因为 agent 不只是被读写，它还会行动、暂停、恢复、协作和分叉。但底层逻辑是一致的：一个成熟的编程体系，必须说明一个单元何时存在、如何变化、何时结束。

## 六、对未来语言设计意味着什么

如果未来真的出现 agent 编程语言，那么它至少要显式支持下面几类能力：

- 声明一个 agent 实例及其身份
- 区分持久状态与临时上下文
- 在不同任务之间传递 agent 引用而不是重复造一个新实例
- 对 agent 做暂停、恢复、派生、合并和退役
- 为身份、状态和上下文分别定义校验与治理规则

这正是“agentic data structure”研究真正指向的方向。它不是为了给现有术语换个名字，而是为了把今天分散在 prompt、memory、workflow、logging、tool use 里的碎片，收敛成一套能被语言、运行时和组织流程共同理解的统一结构。
