# 面向 Agent 的上下文设计

## 一、为什么"写得清楚"仍然可能不适合 Agent

很多文档对人类很友好，因为人类擅长在模糊处自行补全背景、从散落段落里提取主线、用经验弥合上下文缺口。但 agent 的工作方式不同。它并不是在"慢慢读懂"一篇文章，而是在有限上下文窗口和任务目标下，寻找可立即用于决策与执行的信号。

所以，对 agent 来说，真正重要的不是文笔是否顺滑，而是结构是否显式、边界是否清楚、信息是否按任务模式压缩。一个让人类觉得自然的写法，如果把前提、限制、步骤、例外都混在叙述里，agent 的读取成本就会迅速上升。

上下文设计的业务价值，是把 agent 的理解成本从"临场推理"转为"按结构抓取"。前者依赖模型状态，后者依赖工程设计。只有后者才能规模化。

## 二、应该让 Agent 读什么

给 agent 的上下文不应该是资料全集，而应该是完成当前动作所需的最小闭环。这个闭环通常由五类信息组成。

第一类是任务目标。agent 必须知道当前要交付什么，而不是只知道一个模糊主题。

第二类是边界条件。哪些事情不能做，哪些目录不能碰，哪些约束不可违背，这些比补充背景更重要。

第三类是局部结构。也就是当前任务所依赖的模块关系、入口文件、相邻文件、已有接口，而不是整个世界的全景图。

第四类是证据材料。只有与当前决策直接相关的文档、日志、状态记录才应该进入本轮上下文。证据越贴近动作，推理越稳定。

第五类是输出格式。agent 不只是要思考，还要产出。没有明确输出形态，很多推理都会因为落点不清而漂移。

## 三、Agent 的三种高效阅读模式

同一份材料，在不同任务模式下应该以不同方式被读取。真正高效的上下文工程，不是只有一种万能写法，而是让材料适配不同阅读模式。

### 1. 路由模式

路由模式的目标不是深度理解，而是快速判断"该去哪里"。在这种模式下，agent 需要的是模块地图、索引、入口关系、文件定位、主题边界。这里最重要的是覆盖面和导航效率，而不是论证细节。

### 2. 执行模式

执行模式的目标是完成具体动作，例如改文件、写模块、生成产物。此时上下文必须更窄、更硬：明确输入输出、限制条件、依赖关系、最近状态。执行模式最怕的是背景过多，因为那会把决策焦点冲散。

### 3. 校验模式

校验模式的目标是判断结果是否可信。此时 agent 需要的不是完整叙事，而是证据优先的结构：预期结果、检查项、失败样例、风险提示、对照依据。没有证据化结构，校验很容易退化成风格判断。

一份成熟的上下文体系，应该允许同一主题被这三种模式分别消费，而不是强迫所有任务都用同一种长文读取方式。

## 四、让上下文对 Agent 友好的四个设计原则

第一，层次显式。入口负责导航，叶子文件负责展开，不把总览和细节混在一个平面里。

第二，字段显式。目标、约束、输入、输出、证据、状态这些字段最好能被 agent 直接定位，而不是藏在叙述深处。

第三，负载受控。上下文不是越多越安全。真正安全的是让每一轮只看到足够完成当前动作的材料。

第四，模式匹配。路由、执行、校验三种任务，对材料组织方式的需求不同。好的上下文不是一稿通吃，而是对不同模式有不同压缩方式。

## 五、失效模式：为什么很多上下文会越补越差

最常见的失效模式，是把上下文当作资料堆。每次出错就多塞一份说明、多贴一段历史、多加一条规则，结果不是更稳，而是让真正重要的信息被淹没。

第二种失效模式，是把人类写作习惯直接迁移给 agent。人类喜欢自然过渡和留白，但 agent 在执行时更需要显式约束和结构骨架。

第三种失效模式，是忽略任务模式差异。把适合学习的长篇说明直接拿去支撑执行，把适合执行的硬约束直接拿去做启发式探索，都会让效率下降。

上下文设计的成熟标志，不是文档越来越长，而是 agent 越来越少需要在无关信息里自行打捞重点。
