# 从工作记忆到 Skill 的蒸馏

## 一、为什么自动化系统不能一直依赖工作记忆

工作记忆的价值，在于它能保存当前任务最贴近现场的状态：做过什么、卡在哪里、哪些尝试失败了、哪些判断暂时成立。但工作记忆也有天然上限。它通常噪音高、重复多、时间敏感强，适合支撑当下动作，不适合长期复用。

如果系统每次都重新读取大量工作记忆，再由 agent 临场总结经验，那么自动化看似在学习，实际上只是重复消化。真正有业务价值的不是让 agent 每次都重新聪明一次，而是让它把重复出现的经验压缩成稳定结构，形成可复用的 skill、模版、规则和契约。

蒸馏的意义就在这里：把一次性痕迹转化为长期资产。

## 二、蒸馏不是压缩字数，而是提升抽象层

很多人以为蒸馏只是摘要，把十页缩成一页，把一百条日志缩成十条结论。但真正有效的蒸馏，核心不是更短，而是更稳。它要求系统把材料从事件层提升到模式层，从局部经验提升到可迁移结构。

这通常经历三个层次。

第一层是去冗余。把重复表达、偶发噪音、与结果无关的过程信息剔除掉。

第二层是提炼核心。把那些真正影响成败的约束、判断路径、失败原因、成功条件抽出来。

第三层是结构升级。把核心经验写成可复用的形式，例如检查清单、任务模版、节点契约、meta skill、参数化 prompt，而不是停留在一段"下次记得这样做"的自然语言提醒。

所以，蒸馏的本质不是删，而是升维。

## 三、从工作记忆到 Skill 的四步闭环

### 1. 收集原始痕迹

原始材料包括任务日志、计划、失败记录、修改轨迹、人工反馈和最终结果。这个阶段不要急着总结，先保证现场信息没有丢失。

### 2. 识别重复模式

并不是所有经验都值得沉淀。值得蒸馏的，通常是那些跨任务反复出现的约束、常见误判、稳定有效的做法，以及总会触发返工的薄弱点。

### 3. 提炼不变量

这一阶段要回答的不是"上次怎么做的"，而是"什么条件下都应该这样做"。只有找到了不变量，经验才有资格从记忆升级为 skill。

### 4. 封装为可调用结构

最后一步，是把不变量写成系统可以重复调用的结构。它可以是一个 meta skill，也可以是一个标准检查表、一个节点模板、一个上下文装配规则。到这一步，经验才真正进入自动化基础设施，而不是继续留在某次任务的边角里。

## 四、什么样的内容值得沉淀成 Meta Skill

最适合沉淀成 meta skill 的，不是具体答案，而是生成答案的方法。也就是说，skill 不应该过度绑定某次任务的主题，而应该描述一类任务如何被拆解、约束、验证和回收。

例如，"如何把零散工作记忆压缩成三层模块结构"，就比"这次 auto 模块应该怎么命名"更值得沉淀；"如何识别对 agent 友好的上下文字段"，就比"这次 prompt 具体写什么"更值得进入 skill 层。

判断一个经验是否值得进入 meta skill，可以看三点：它是否跨任务复用；它是否能降低误判成本；它是否能被参数化，而不是每次重写。

## 五、蒸馏失败的三种常见原因

第一种失败，是把噪音直接升级为规则。一次偶然成功的方法，如果没有跨任务验证，就不该进入长期 skill 层。

第二种失败，是抽象过度。把经验写得太宏大、太空泛，最后听起来都对，但没有任何执行指导价值。

第三种失败，是只做摘要，不做封装。材料虽然被压短了，但仍然只是结论文本，没有进入系统可调用的结构层，结果下一次还得重新阅读。

自动化系统真正成熟时，工作记忆不再只是堆积现场，而会持续向上蒸馏，形成越来越稳的规则骨架。这样，agent 才不是每次从零开始，而是在不断继承自己已经学会的结构。
