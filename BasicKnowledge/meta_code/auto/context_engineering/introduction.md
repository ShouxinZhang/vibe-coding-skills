# Context Engineering：给 Agent 的可执行上下文

## 一、为什么 Auto 需要单独讨论 Context Engineering

在 LLM 自动化系统里，很多人以为真正重要的是 prompt 写得够不够强，workflow 画得够不够完整。但一旦系统进入持续运行阶段，最先暴露出来的问题往往不是能力不够，而是上下文失控：给了太多，agent 读不完；给得太散，agent 抓不到重点；给得太像写给人看的文档，agent 理解成本反而更高。

这说明上下文不是 prompt 的附属品，而是自动化系统中的独立基础设施。所谓 Context Engineering，本质上是在回答三个问题：应该让 agent 读什么；在什么阅读模式下读；读完之后如何把短期信息蒸馏成可复用结构，而不是每次重新消化一次。

如果 Prompts System 解决的是"怎么发指令"，Workflow 解决的是"怎么推进任务"，那么 Context Engineering 解决的就是"系统到底喂给 agent 什么材料，材料如何组织，如何在运行中保持干净"。这三者合起来，自动化才真正具备生产稳定性。

## 二、这个模块解决什么问题

很多自动化系统失败，不是因为模型太弱，而是因为上下文设计仍停留在人类写作直觉里。对人类清晰的文档，不一定对 agent 高效；对人类略显冗长的结构化材料，反而可能让 agent 更快锁定边界、证据与动作。上下文工程的价值，正在于把"能读"升级为"能执行"。

这个模块聚焦三件彼此衔接的事。第一，设计 agent 可高效消费的上下文结构与阅读模式；第二，把运行中的工作记忆蒸馏为可复用的 skill 与规则层；第三，用清理、归档、日志、沙箱、规划、权限这些运行机制保证上下文不会越积越脏。

## 四、从 Prompt 到 Context Engineering：一条隐藏的演化线

早期大家谈 prompt，核心是"怎么问"。后来大家谈 workflow，核心是"怎么让系统自己跑"。但系统一旦开始长期运行，很快就会发现，真正吞掉自动化效率的常常不是提问，也不是流程图，而是上下文本身。

第一阶段是堆料阶段。人们把更多文档、更多规则、更多历史记录塞给模型，期待它因此更聪明，结果却常常得到更慢、更散、更不稳定的输出。

第二阶段是裁剪阶段。人们开始意识到，上下文不是越多越好，而是越贴近当前决策越好。于是出现了摘要、检索、选择性加载、工作记忆等机制。

第三阶段才是真正的 Context Engineering 阶段。此时问题不再是"删掉多少"，而是"如何为不同任务模式设计不同形态的上下文"，以及"如何把一次性工作记忆沉淀为长期可复用结构"。这时，上下文从材料变成系统部件。
