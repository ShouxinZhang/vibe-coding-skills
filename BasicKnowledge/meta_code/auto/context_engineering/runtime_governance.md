# 上下文运行时治理

## 一、为什么上下文也有运行时问题

很多人把上下文理解成静态材料，仿佛它只是任务开始前喂给 agent 的一包信息。但只要系统进入连续执行，上下文就不再是静态文档，而是一种会不断增长、不断污染、不断影响后续决策的运行时资源。

今天新增的一条日志，明天可能变成噪音；一次任务里留下的临时计划，如果没有回收，下一次就可能被误当成正式状态；权限边界如果没有写清楚，agent 甚至无法判断哪些材料可读、哪些动作可做。换句话说，上下文不是只需要设计一次，还需要持续治理。

运行时治理的业务目标很明确：让上下文在长期运行中保持可读、可控、可追踪，而不是随着自动化吞吐上升一起失序。

## 二、五个必须被治理的运行面

### 1. 清理与归档

没有清理机制，工作区里的临时文件、旧计划、过期摘要、历史日志会不断累积。对 agent 来说，这不是简单的存储浪费，而是检索噪音源。清理的重点不是删得快，而是把无效现场信息移出主工作面，把仍有价值但不该参与当前决策的材料归档到次级层。

### 2. 日志记录

日志不是为了留痕好看，而是为了让未来的 agent 和人类都能知道系统为什么走到这里。没有日志，错误只能被描述；有日志，错误才能被分析、复现和蒸馏。真正有价值的日志，不是流水账，而是围绕状态变化、关键判断、失败原因和人工接管点展开。

### 3. 规划状态

规划不是思考的副产品，而是运行时坐标系。agent 如果没有明确的当前步骤、已完成步骤、阻塞点和下一动作，就很容易在长任务里重复劳动或丢失方向。规划的价值，不在于计划写得多漂亮，而在于任何时刻都能知道系统在哪一格。

### 4. 沙箱边界

上下文治理并不只发生在文本层，也发生在环境层。agent 能读哪些目录、能不能联网、能不能写文件、能不能启动命令，这些能力边界会直接影响它如何理解"当前上下文"。边界不清，agent 的决策空间就会变得模糊，最终不是保守失效，就是越界冒险。

### 5. 权限分层

权限不是单纯的安全话题，它也是上下文准确性的条件。只有当系统明确区分哪些节点能看全局、哪些节点只能看局部、哪些动作必须经 Gate 或人工批准时，agent 才能在正确信息面上做正确决策。权限模糊，本质上会把错误信息暴露成错误自由度。

## 三、为什么这些机制应该放在同一个框架里

清理、归档、日志、规划、沙箱、权限，看起来像不同主题，但它们在自动化系统里共同服务于同一个目标：控制上下文质量。

没有清理，旧信息污染新决策；没有日志，系统无法知道哪些信息值得保留；没有规划，状态就无法被压缩成当前焦点；没有沙箱和权限，系统甚至不知道哪些信息本来就不该进入上下文。把这些机制拆开看，容易得到一堆局部规则；把它们放回同一个运行时治理框架里，才会发现它们共同决定了 agent 看到什么、信什么、能做什么。

## 四、运行时治理成熟的标志

一个成熟的自动化系统，不是上下文越来越多，而是越来越分层。当前任务只看到当前需要的材料，历史信息有归档位置，关键判断有日志证据，计划状态随时可恢复，权限边界始终清楚。

当这些条件成立时，自动化系统才不会随着时间推移变成一团自我污染的记忆堆。相反，它会越来越像一个有代谢能力的系统：持续吸收信息，持续清理废料，持续把经验沉淀为更稳定的结构。对业务来说，这意味着自动化不再只是能跑一次，而是能在长期运行中维持质量与秩序。
