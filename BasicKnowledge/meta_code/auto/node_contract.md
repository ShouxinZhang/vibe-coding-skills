# LLM 驱动节点的契约设计

## 一、为什么节点必须先被定义，再被自动化

在 LLM 自动化系统里，节点不是一段随手拼接的 prompt，而是可交付、可审计、可接力的工作单元。系统一旦进入连续执行状态，真正稀缺的不是"让模型先做点什么"，而是确保它每次都在同样的边界内做、产出可判断的结果，并在失败时留下可追溯证据。所谓节点契约，本质上就是把一个节点的责任写清楚：它接收什么输入，允许读取哪些上下文，拥有什么工具权限，要输出什么结构，失败时如何退出，状态如何记录，何时必须把问题交还给人。

没有契约，节点就只是一次性的智能操作；有了契约，节点才变成可验证、可替换、可回放的生产部件。可验证意味着它的输出能被 Gate 判定为通过或失败，而不是靠下游猜测；可替换意味着同一职责可以切换不同模型、不同实现甚至不同供应商，而不重写整条链路；可回放意味着在同样输入、同样权限、同样上下文边界下，可以复现一次执行，解释为什么成功、为什么失败。

## 二、一个节点的契约到底要写什么

契约的第一层是输入输出。输入不只是用户文本，还包括结构化参数、前置产物版本、依赖节点摘要与成功条件；输出也不能只是"给一段自然语言"，而应是明确的结果对象、状态码、证据链接与副作用声明。只有输入输出被固定，节点才有机会被测试、缓存、并行调度和替换。

第二层是上下文边界与工具权限。LLM 最容易失控的地方，不是推理本身，而是上下文无限扩张和工具无限调用。节点必须明确可读上下文的范围：读哪些文件、哪些日志、哪些历史状态，哪些信息一律不可见。工具权限也必须写进契约：可否写文件、可否执行命令、可否联网、可否启动进程。这正是沙箱与权限边界的意义。边界定义在节点之外，节点才能在边界之内高效执行，而不是一边工作一边临时讨权限。

第三层是状态记录。每次执行都要留下日志：输入摘要、环境版本、调用工具、关键中间结论、最终输出、错误原因与耗时。日志不是附属品，而是节点可回放的前提。没有状态记录，失败就只是"模型今天不稳定"；有了状态记录，失败才会变成可分析、可修复、可纳入 Gate 规则的工程事实。

## 三、为什么这对并行化和规模化有直接业务价值

契约的业务意义，不在于文档好看，而在于它把自动化从"单次演示"推进到"可运营系统"。当节点输入输出稳定、上下文边界清晰时，多个节点才能并行运行，因为彼此不再依赖隐含记忆与临场解释；当工具权限被约束时，更多任务可以安全地下放给自动系统，而不必把所有执行都压在少数高权限节点上；当状态被记录并可回放时，问题定位成本下降，SLA 才有现实基础。

规模化时，最大的成本不是模型调用费，而是不可预测性带来的返工、误判和人工兜底。契约把这些隐性成本前置为显式约束。一个有契约的节点可以被排队、复用、缓存、替换，也可以被不同团队共同消费。今天用高能力模型生成方案，明天换成低成本模型做初筛，只要契约不变，系统就不用整体返工。这就是并行化的前提，也是规模化的商业前提：职责清晰，接口稳定，失败可控。

## 四、幂等、重试、组合与人工接管

真正进入生产后，节点还必须具备明确的失败语义。幂等性是第一条底线：同样的输入重复执行，不应制造新的脏状态或重复副作用。能做到幂等，重试策略才成立。重试也不能只是"再跑一次"，而要区分可重试错误与不可重试错误：模型超时、临时资源不足可以退避重试；输入缺失、权限不足、Gate 不通过则必须立即停止。否则系统只是在放大错误，而不是恢复服务。

可组合性是契约的更高价值。节点只有先把自己缩成边界清晰的工作单元，才能被串联、并联、替换，组成更大的流程。上游只关心它是否满足契约，下游只消费它承诺的输出，这样系统才能演化，而不是每次扩展都牵一发动全身。

最后必须保留人工接管点。并不是因为 LLM 不够强，而是因为业务责任最终仍由人承担。当节点遇到高风险改动、证据冲突、重复失败、权限越界需求或 Gate 长期不过时，契约应明确要求暂停自动推进，转入人工审查。自动化最成熟的标志，不是永远不停，而是知道何时必须停。节点契约的价值，正是在这里完成闭环：它让系统跑得更快，但不是靠冒险，而是靠边界、证据与可替代性。
