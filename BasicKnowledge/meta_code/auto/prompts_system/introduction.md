# Prompts System：从一句指令到可运营的提示工程

## 一、来龙去脉：从填词游戏到工程系统

2019 年，OpenAI 发布 GPT-2。那时候"prompt"这个词还没什么重量。你给模型一段开头，它帮你续写。"The weather today is..."——模型接着往下说。本质上是一场填词游戏：你搭语境，模型在语境里随机漫步。没人叫它"提示工程"，大家只是好奇"这玩意儿能生成什么花样"。

2020 年，GPT-3 横空出世。1750 亿参数，第一次让"few-shot learning"变得令人震惊。你在 prompt 里放几个示例，模型就理解了任务规律，甚至不需要微调任何参数。这是一次根本性的跳跃——模型开始看懂你的意图，而不只是在统计上接续你的文本。一时间，研究者们疯狂尝试各种 prompt 格式，发现不同写法可能带来天壤之别的效果。"Prompt engineering"开始密集出现在论文标题里。

2022 年，InstructGPT 登场。OpenAI 用 RLHF 训练了一个专门响应指令的模型：你可以直接说"帮我做 X"，而不必精心设计 few-shot 铺垫。这把模型从"续写机器"变成了"执行助手"。prompt 不再只是上下文铺垫，而是可以直接承载命令的语言接口。

2022 年底，ChatGPT 发布。System prompt 第一次被放到台面上——你可以在对话开始之前，用一段系统级指令告诉模型它是谁、行为边界在哪、应该以什么口吻回应。角色人格、安全护栏、业务约束，全都可以写进 system prompt。从这一刻起，prompt 开始变成有层次的结构，而不是一条扁平文本。

2023 年到 2025 年间，随着 GPT-4、Claude 3、Gemini 1.5 等模型陆续登场，模型基础能力的提升趋于平稳——"够不够聪明"不再是核心矛盾，"能不能稳定运行"才是。企业级 AI 应用的真正瓶颈开始转移：不是模型能力不足，而是输出不稳定、格式漂移、跨任务行为不一致。于是围绕 prompt 的工程体系开始系统化：版本管理、自动评估、prompt 压缩与优化，一整套治理框架逐渐浮现。"Prompts System"这个词开始代替"一条 prompt"，成为描述这套基础设施的新概念。

到 2026 年，这套框架已经成为 LLM 驱动自动化系统中不可缺少的基础层。

## 二、直觉是怎么被逼出来的

大多数人认真研究 prompt 的动机，不是从理论出发，而是被结果逼出来的。

你让模型帮你总结一篇文章。于是你写："帮我总结这篇文章。"把文章贴在后面。模型给了你一段东西——看起来像总结，但太长，废话很多，你真正关心的要点根本没出现。你以为是模型能力问题，但真正的问题是你的表达太模糊，让模型不知道"好的总结"是什么样的。

然后你加了角色：**"你是一位专业编辑，请帮我总结..."**——好一点了。加了格式约束：**"以三个要点列出，每点不超过 20 字"**——更好了。再加了一个示例，告诉模型上一次好的总结长什么样——这次它终于给出了你想要的结果。

这就是提示工程的本质：不是在教模型新知识，而是在搭建一个足够清晰的工作上下文。角色、格式、示例、约束，每一层都在压缩歧义空间，把模型的输出收束到你心里那个答案。只不过，当你同时管理几十个不同任务的 prompt，临场调整就变得代价高昂——你需要记住哪套 prompt 对应哪个任务，哪个关键约束有没有遗漏，不同任务之间的风格是否一致。凭感觉调 prompt 变成了系统性的负担，这时候"一套 prompt"就不够了，你需要的是一个 Prompts System。

## 三、什么是 Prompts System

Prompts System 不是单独的一条指令，而是围绕 LLM 调用设计的多层级工程结构，通常包含以下核心组件：

**System Prompt（系统提示）**：定义模型的角色、行为规范、回答风格和责任边界。在对话开始前就被注入，贯穿整个会话生命周期。它决定了模型的"默认人格"。

**User Prompt（用户提示）**：每次任务的具体请求。可以是静态模板，也可以是由上游节点动态生成的填充结果。

**Few-shot Examples（示例）**：用具体的输入-输出配对告诉模型"好的结果应该长什么样"，是解决格式漂移和风格不稳定最直接的工具。

**Tool Descriptions（工具描述）**：在 agent 工作流中，当模型需要调用外部工具时，工具的功能、参数、使用时机都必须以自然语言描述清楚——这本质上也是 prompt 的一部分，而且往往决定了工具调用的准确性。

**Output Format（输出格式规范）**：告诉模型输出结构是什么：JSON、Markdown、纯文本，是否需要分节，是否包含元数据字段。格式规范让下游解析成为可能。

**Constraints（约束）**：明确禁止什么、鼓励什么、在什么边界条件下切换行为模式。约束是安全护栏，也是行为边界的工程化表达。

这六个组件组合在一起，才构成完整的 Prompts System。它不是一次对话的开场白，而是让模型行为可预期、可验证、可维护的工程基础设施。

## 四、为什么 Prompts System 在 2026 年是 auto 的核心

在 2026 年，模型的底层能力本身已经不是瓶颈了。你可以信任一个前沿大模型能够理解复杂指令、进行多步推理、处理超长上下文。真正的瓶颈转移到了：**如何让这种能力稳定、可复现地为具体任务服务**。

这正是 auto 系统面临的根本挑战。当一个 agent 在无人监督下连续执行几十个任务时，任意一个节点的输出格式漂移都可能让下游工作流崩溃；任意一处 system prompt 设计缺陷都可能让模型在边界场景做出意外决策。当工作流从"手动操作"升级为"自动运营"，对 prompt 质量的要求是成倍上升的——因为你不在场，没有人临场兜底。

Prompts System 就是那把钥匙：它把靠感觉调 prompt 的"艺术"，转化为可以被版本管理、测试验证、团队协作和持续优化的"工程"。一个设计良好的 Prompts System 让 agent 节点的行为变得可预测，让自动化工作流的质量变得可验收，让 AI 系统从"演示成功"走向"可持续运营"。更进一步，当 Prompts System 本身也可以被自动生成和优化时——也就是 meta-prompt 出现的时候——这套工程的自动化闭环才真正合上。
