# Meta-Prompts：让 Prompts 自己生成 Prompts

## 一、定义与类比

有一个编程世界里的老概念叫"元编程"——代码生成代码。Lisp 程序可以在运行时生成新的 Lisp 代码并执行它；C++ 的模板元编程在编译期就完成了代码的生成与特化。程序员写的不是最终运行的代码，而是"生成代码的代码"。这层抽象带来的收益是惊人的：用一套规则覆盖无数具体实现，把重复的模式提升到更高层次的表达。

Meta-prompt 做的事情本质上一样：**它是一段 prompt，其目的是让 LLM 生成或优化另一段 prompt**。

你不是在直接告诉模型"如何总结文章"，你是在告诉模型"如何为'总结文章'任务写一个高质量的 system prompt"。模型扮演的不是任务执行者，而是 prompt 工程师的角色。它观察任务描述、理解输出要求、套用它学到的 prompt 设计原则，为你生成一套可以直接投入使用的提示结构。

这听起来有点绕，但想想另一个类比：你不是自己去劝说某个人，而是让一位有说服力的谈判专家，根据你的目标，替你写好那封邮件。meta-prompt 里的 LLM，就是那位专家；它生成的结果，是直接面向下游任务的 prompt。

## 二、三个具体例子

### 例子 1：让 GPT 基于任务描述自动生成 system prompt

这是 meta-prompt 最直接的用法。你有一个任务，但不确定 system prompt 该怎么写。于是你用下面这段 meta-prompt 来请它帮你完成：

```
你是一位专业的 Prompt 工程师。
我将给你一个任务描述，请为该任务生成一个高质量的 System Prompt。
要求：
- 包含角色定义和行为规范
- 包含输出格式要求
- 包含至少一个输入-输出示例（few-shot）
- 包含关键约束（禁止做什么）

任务描述：
将用户提供的任意一段会议记录整理为结构化的行动项清单，
每项包含负责人、截止时间和具体内容。

请输出完整的 System Prompt：
```

模型的返回可能是这样的 system prompt：

```
你是一位专业的会议助手。你的职责是将用户提供的会议记录，
提取并结构化为行动项清单。

输出格式：每条行动项以 Markdown 列表表示，包含三个字段：
- 负责人：[姓名]
- 截止时间：[YYYY-MM-DD 或"未明确"]
- 具体内容：[一句话描述]

示例输入：
"李明说他会在周五前完成 API 文档，张伟需要下周一前提交测试报告。"

示例输出：
- 负责人：李明 | 截止时间：本周五 | 内容：完成 API 文档
- 负责人：张伟 | 截止时间：下周一 | 内容：提交测试报告

约束：若会议记录中没有明确的负责人或截止时间，填写"未明确"，不得猜测。
```

这个生成的 system prompt 可以直接投入使用，也可以作为人工微调的初稿，大幅降低从零构建的成本。

### 例子 2：APE——自动 Prompt 工程师

APE（Automatic Prompt Engineer）是 2023 年由 Zhou 等人提出的方法，核心思路很优雅：**把 prompt 优化本身变成一个搜索问题，用 LLM 来执行搜索**。

流程分三步：第一步，给模型一批输入-输出示例，让它**反推**"什么样的 prompt 指令可以产出这些输出"，生成若干候选 prompt；第二步，把每个候选 prompt 放到验证集上跑，用准确率或人类偏好评分打分；第三步，选出得分最高的 prompt，可以再用它作为输入，进行下一轮迭代生成。

简化版的实现是这样的：

```python
# Step 1: 让 LLM 生成候选 prompts
meta_prompt = """

{examples}

请生成 5 个不同的 instruction prompt，使得当 LLM 接收到这条指令
及每个输入时，能尽可能接近上述输出。每个 prompt 单独一行。
"""

# Step 2: 对每个候选 prompt 在验证集上打分
# Step 3: 选择得分最高的 prompt 作为最终结果
```

APE 的业务价值很直接：过去你需要手动写几十个 prompt 变体、对比效果，现在这个搜索过程本身由 LLM 自动完成。对于高频、高价值的任务（比如客服分类、合同审阅、报告生成），把 prompt 质量提升哪怕 5%-10%，都可能带来实质性的业务收益。

### 例子 3：agent 工作流中 orchestrator 动态生成 worker prompt

在多 agent 系统里，meta-prompt 的价值最为显著。假设有一个 orchestrator 负责把一个大任务分发给多个 worker agent，每个 worker 负责特定子任务。与其在代码里硬编码每个 worker 的 prompt，更健壮的做法是让 orchestrator 根据子任务描述，动态生成对应的 worker prompt：

```
你是任务分发系统的 Prompt 生成器。
给定一个子任务描述，输出一个适合该任务的 Worker System Prompt。

子任务描述：{sub_task_description}
可用工具列表：{available_tools}
输出格式要求：{output_schema}
上游节点产物摘要：{upstream_summary}

请生成 Worker System Prompt：
```

这样，当业务需求变化、新的子任务类型出现时，orchestrator 不需要代码改动，只需要更新 meta-prompt 里的规则，就能生成适配新任务的 worker prompt。这是 auto 系统保持弹性的关键机制之一。

## 三、演化路径

从最初的 prompt 实践到 meta-prompt 系统，有一条清晰的演化轨迹：

**手写 prompt**：每次任务单独撰写，依靠个人经验，无法复用，质量因人而异。

**模板化 prompt**：把共性结构抽取为模板，通过变量填充适配不同任务。可复用，但模板本身仍需要手动维护，对新任务类型缺乏适应性。

**规则生成 prompt**：用代码逻辑根据任务属性拼接 prompt，适用于输入结构非常规整的场景。覆盖范围有限，面对自然语言边界时失效。

**LLM 生成 prompt**：用 LLM 的语言理解能力取代规则拼接，可以处理模糊的任务描述，生成更具表达力的 prompt。这正是最基础的 meta-prompt 实践。

**Meta-prompt 系统**：不只是单次生成，而是生成、评估、迭代、版本管理形成闭环的完整工程体系。prompt 本身成为被工程化管理的一等公民。

每一步跃迁的驱动力，都是上一步的局限性在规模化时暴露出来的成本——这和软件工程本身的演化逻辑完全一致。

## 四、Meta-Prompt 的工程价值

把 prompt 工程从"艺术"变成"工程"，是 meta-prompt 系统最核心的业务意义。

**可复制**：一套经过验证的 meta-prompt 可以在不同任务域迁移，而不是每次重新从零开始设计。你不是在依赖某个员工的"prompt 天赋"，而是在依赖一套可共享、可传递的生成规则。

**可优化**：通过 APE 类方法，prompt 质量可以被系统性地测量和提升，而不是靠人的直觉反复试错。优化的过程本身也可以被记录和复现。

**可版本控制**：自动生成的 prompt 可以像代码一样被纳入 Git，每次任务场景的变化都对应一个 prompt 版本提交，方便回溯和对比分析。

**可规模化**：当工作流里有几十个不同类型的 agent 节点，手动逐一维护它们的 prompt 是不现实的。meta-prompt 提供了一个统一的生成层，让 prompt 管理的成本不随节点数量线性增长。

简而言之，meta-prompt 让 prompt 工程具备了工程系统的基本属性：可测试、可迭代、可协作、可监控。

## 五、风险与边界

meta-prompt 并不是银弹，它把能力放大的同时，也把风险放大。

**错误的传播性更强**。当 meta-prompt 设计有缺陷，它生成的所有下游 prompt 都会带着这个缺陷。如果 meta-prompt 的角色定义过于宽泛，所有生成出来的 worker prompt 就都缺乏明确的责任边界。错误不再是局部的，而是系统性的。这和软件工程里的"框架级 bug"一样危险。

**偏见的继承**。LLM 本身携带训练数据里的偏见，用它生成 prompt 时，这些偏见有可能被编码进生成出来的指令里。比如，一个用于招聘筛选的 meta-prompt，可能生成带有性别或年龄偏见的评估标准，而这一切发生得比手写更隐蔽。

**可解释性下降**。手写的 prompt 是人写的，出了问题你知道去哪里找。meta-prompt 生成的 prompt 是 LLM 生成的，出了问题你面对的是两层 LLM 逻辑的叠加——调试复杂度显著上升。

**门禁机制是必要的**。一个成熟的 meta-prompt 系统，不能只有生成，还需要评估环节：对生成出来的 prompt 在验证集上做质量检验，在部署前做人工审查，在运行期监控输出漂移。meta-prompt 引入的效率收益，必须以评估与门禁机制为前提——否则你只是在把风险系统化地放大，而不是工程化地控制它。

meta-prompt 是 auto 系统里很强的一张牌，但它不能替代人对系统行为的最终判断责任。自动化最成熟的标志，从来不是永远不停，而是知道什么时候必须让人来看一眼。
