# Workflow：从手动操作到状态驱动的自动流水线

## 一、从"如果这个，就那个"开始

所有的自动化，最开始都非常简单。

2011 年，Zapier 上线了。它的核心理念用一句话就能说清楚：**如果这件事发生了，就触发那件事**。收到新邮件，就在 Trello 创建一张卡。表单被填写，就往 Slack 发一条通知。每一个 "Zap" 都是一个 if-then 触发器，逻辑平铺，没有状态，没有分支，没有等待。

这个模型对于"连接两个工具"极其有效。但是，当任务变得稍微复杂一点——比如"在通知发出之后等待审批，审批通过了再做下一步，审批拒绝了走另一条路"——单一触发器就开始力不从心了。

所以工程师们需要更强的东西。

---

## 二、CI/CD：流水线进化出了骨架

GitHub Actions、Jenkins、GitLab CI——这一代工具给工作流带来了真正的**结构**。

一个典型的 CI 流水线是这样运作的：代码提交触发构建，构建通过后运行测试，测试通过后部署到预发布环境，预发布通过后等待人工审批，审批通过后部署到生产。每一步都有明确的**前置条件**，失败就停下来，成功才继续。

这里有几件事已经悄悄发生了：任务不再是单一触发器，而是**多步骤的链条**；步骤之间存在**条件分支**；系统开始有了某种"当前处于哪一步"的概念——也就是状态。

但这些工具的局限同样明显：每一个步骤都是确定性的程序——编译器要么通过要么失败，测试要么绿要么红。没有人会写一个步骤说"让编译器判断这段代码的设计是否合理"。

然后，LLM 出现了，把这个假设打碎了。

---

## 三、LLM 时代：每个步骤变成了"问模型"

n8n、Dify、LangChain——这些工具在 2023 年之后开始大量涌现。它们的共同特点是：把工作流的某些步骤，替换成"把数据发给 LLM，拿回输出，再继续往下走"。

这听起来像是一个小扩展，实际上是范式级别的转变。

当一个步骤是"调用 LLM 来审查这段代码的安全性"时，这个步骤的输出不再是布尔值，而是一段**自然语言判断**。工作流系统需要能够理解这个判断，并根据它决定下一步走哪条路。更微妙的是：LLM 的判断本身是概率性的、不确定的。同样的输入，今天的模型可能说"通过"，换一个温度参数或者换一个模型版本，结果可能不同。

这带来了一个根本性的问题：**工作流系统怎么知道自己"在哪"？**

如果你只靠 if-else 的链条来组织这些步骤，系统会变成一张复杂的蜘蛛网——每增加一个 LLM 步骤，可能的路径就指数级增长。维护这张网的人，最终会失去对系统行为的掌控。

---

## 四、现在：LLM Agent 既执行 Workflow，也设计 Workflow

2025-2026 年，我们正在进入一个新阶段。

一个 LLM agent 不再只是 workflow 中的一个步骤。它开始**参与设计 workflow 本身**。给定一个目标，agent 可以拆解任务，生成执行计划，调度子 agent，收集结果，判断是否需要重新规划。这是 **meta-workflow**——使用 workflow 来生成和管理 workflow 的能力。

但越是复杂的自动化，越需要一个清晰的回答：**系统现在处于什么状态？**

这就是为什么"状态机"这个已经存在了几十年的概念，在 LLM agent 时代重新变得核心。

---

## 五、Workflow 状态机：用一个例子感受

用"代码 review 流水线"来感受状态机的运作方式。

想象这样一个流程：

1. **提交**：开发者推送代码，流程启动
2. **静态分析**：自动运行 linter 和安全扫描
3. **LLM 审查**：调用大模型，评估代码结构和逻辑
4. **人工评审**：把审查结果发给负责人，等待决策
5. **合并 / 拒绝**：根据评审结果执行最终动作

每一个步骤，就是一个**状态**。系统在任何时刻，都处于且只处于其中一个状态。从一个状态转移到下一个状态，需要满足特定**条件**：静态分析通过了，才能进入 LLM 审查；LLM 给出通过评分，才能进入人工评审。最终的"合并"和"拒绝"都是终止状态——任务的生命周期在这里结束。

这就是状态机的核心思维：**显式地定义"系统在哪里"，而不是让代码的 if-else 分支隐式地决定走向。**

用这种方式组织 workflow，带来了几个立竿见影的好处：哪个步骤卡住了，一眼就能看出来；需要回滚时，可以精确地从某个状态重新开始；每个状态都可以单独测试，不需要跑完整条流水线才能验证某个节点的行为。

---

## 六、为什么状态机是 LLM Workflow 的最佳模型

对于确定性程序（编译器、测试框架），if-else 链能用，因为输出是可预测的。

但 LLM 不是。同一个 prompt，不同时间、不同版本、不同温度参数，输出可能完全不同。这种不确定性在单次调用时可以接受，但在一个多步骤 workflow 里，它像多米诺骨牌一样级联传播——第一步的不确定输出影响第二步的判断，再影响第三步的数据，最终系统的行为变得完全不可解释。

状态机在这里扮演的角色，是一个**确定性的外壳**。

不管 LLM 在某个步骤输出了什么，状态机总能回答这个问题：**系统现在处于哪个状态？** 如果输出不满足进入下一状态的守卫条件，系统就停在当前状态，等待重试或人工介入——而不是带着这个有问题的输出继续往下跑，产生无法回溯的副作用。

显式状态让系统行为**可测试**——针对每个状态写单元测试；**可监控**——用仪表盘展示当前有多少任务卡在哪个状态；**可回滚**——从任意历史状态重新执行，就像游戏存档一样。

这是一种工程上的成熟度：承认不确定性的存在，然后用结构来驯服它。
