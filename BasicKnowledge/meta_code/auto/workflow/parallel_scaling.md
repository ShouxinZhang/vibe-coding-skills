# 并行规模化：让 LLM 工作流从单线程变成流水线

## 一、来龙去脉：从一问一答到流水线工厂

LLM 最初的使用方式极其朴素：你问一句，它答一句，串行到底。这种方式在探索期足够用，但一旦任务复杂起来——比如"帮我整理这 500 篇文档"——单轮对话的天花板立刻暴露：上下文塞不下，质量无法稳定，出错也无从定位。

最早意识到瓶颈的一批人，用的是一个朴素的直觉：把大任务拆开，分别问，再汇总。你不让模型一口气分析 100 篇论文，而是每次喂 5 篇，跑 20 个并行查询，最后汇总结论。这就是 MapReduce 思想在 LLM 时代的翻版：先 map（分发处理），再 reduce（聚合提炼）。早期实验表明，这样做不仅省时，往往还提升了质量——因为每个子任务的上下文更干净，模型不容易因为信息过载而"跑偏"。

2023 年 GPT-4 发布后，token 成本成了绕不开的话题。一个自然的应对方案随之出现：用多个小模型并行处理分块任务，只在最终汇总时调用一次大模型。成本可以砍掉一半以上，而质量损失往往微乎其微。这种"小模型并行 + 大模型收敛"的结构，实际上是并行规模化的早期工程形态。

2024 到 2025 年，LangChain、AutoGen、CrewAI、LlamaIndex 等 agent 框架相继爆发，并行 agent 从实验室方案变成了标准工程配置。进入 2025 到 2026 年，知识工厂、代码工厂、研究工厂这类基于 LLM 的生产系统全面铺开，并行规模化不再是优化项，而是核心基础设施。没有它，这些系统根本跑不起来。

---

## 二、五种核心模式

### Fan-out（分散）

一个任务进来，拆成 N 个并行子任务，每个 agent 独立拿着自己那块去执行，互不干扰。

**例子**：你要对一个代码仓库里的 200 个函数做安全审查。Fan-out 就是把这 200 个函数分成 10 批，每批 20 个，启动 10 个并行 agent 同时执行。没有谁在等谁，速度直接提升 10 倍。

串行方案的问题在于：如果函数之间没有强依赖，前一个审完再审下一个纯属浪费时间。Fan-out 消除了这种无意义的等待。

---

### Fan-in（汇聚）

N 个并行执行的结果流回一个汇聚节点，由它负责去重、整合、校验和最终输出。

**例子**：接上面的例子，10 个 agent 各自跑出了 20 条安全问题列表。Fan-in 节点把 200 条原始记录合并，去掉重复检出的条目，按严重程度分级，生成一份统一的审查报告。

没有 Fan-in，并行只是把混乱放大了。Fan-in 是让并行结果真正可用的最后一道门。

---

### Map-Reduce 变体

先 map（把任务映射分发到各 agent），再 reduce（把所有结果聚合提炼）。LLM 版本的特别之处在于：reduce 步骤本身也可以是一个 LLM 调用，而不是简单的数据拼接。

**例子**：你要从 50 篇行业研报中提炼出"2025 年 AI 投资趋势"。Map 阶段：每个 agent 读一篇，输出结构化的关键判断。Reduce 阶段：把 50 份结构化判断交给一个大模型，让它综合推导出全局趋势。Map 做的是信息压缩，Reduce 做的是认知提炼，两者分工清晰。

相比让一个模型一口气读完 50 篇，这种分层结构的质量更稳定——因为每个 map 节点的上下文是干净的，reduce 节点拿到的是已经初步结构化的输入，而不是巨量原文。

---

### Work-Process-Tree

树形任务分解：父节点定义目标和上下文边界，子节点负责执行。父节点不关心子节点怎么做，只关心子节点是否按契约交付了符合要求的结果。

**例子**：写一本技术手册，父节点定义章节结构和每章的产出标准；子节点各自负责一章，互不干扰。章节内部如果还需要拆分（如小节），就再往下派生一层子节点。

和 DAG 相比，树形结构更保守：中间结果不跨分支共享，每条路径独立，失败了只影响本条路径。DAG 资源利用率更高，但一个共享节点出错，影响面更广。对于知识生产、文档写作这类以可控性为第一优先级的场景，树往往比 DAG 更容易落地。

---

### Streaming Pipeline（流式流水线）

不等所有并行任务全部完成，先完成的结果先进入下一阶段处理，先入先出，全程流动。

**例子**：你在实时分析一个直播的弹幕情感。不是等 1000 条弹幕全部收集完再做分析，而是每来 10 条就立刻跑一次情感分类，结果实时输出给前端展示。分类 agent 不等待，汇总面板随时刷新。

流式流水线适合实时性要求高、任务源源不断的场景。它的代价是汇聚状态更难管理——因为结果是持续流入的，全局一致性需要额外设计。

## 三、并行的前提条件

不是所有任务都适合并行。以下三个条件同时满足时，并行才真正有意义：

1. **弱依赖**：子任务之间没有强先后关系，A 不依赖 B 的中间结果。
2. **输入边界清晰**：每个子任务的输入范围明确，不存在"到底谁负责这块"的歧义。
3. **可独立验收**：每个子任务的完成标准可以单独判断，不需要看整体才知道对不对。

如果任务需要连续推理（后一步依赖前一步的思考过程），或者各 agent 需要频繁协商统一口径，那串行或单线程往往反而更高效——因为协调成本会吞掉并行带来的所有收益。

---

## 四、并行的成本与风险

并行不是免费的午餐，它带来了四类需要正视的成本：

**协调成本**：启动、监控、调度多个 agent，本身就需要基础设施。任务粒度太细时，协调开销甚至会超过执行本身。

**上下文爆炸**：每个 agent 都需要一份上下文，N 个并行 agent 就是 N 份上下文的 token 消耗。如果上下文设计不合理，总成本反而比串行更高。

**收敛节点成为瓶颈**：Fan-in 节点汇聚的信息量随并行度线性增长。当并行度很高时，汇聚节点的输入量可能达到单次 LLM 调用的上限，成为整个流程的瓶颈。

**输出漂移导致汇聚困难**：多个 agent 独立生成内容，即使 prompt 一样，输出格式和用词也可能出现漂移——一个写"高风险"，另一个写"risk level: high"。格式不统一，汇聚时就需要额外的标准化步骤，否则 Fan-in 就会变成混乱拼接。

---

## 五、经典落地场景

**知识库批量更新**：旧文档有几百篇需要按新格式重写。Fan-out 把每篇文档分配给独立 agent 处理，Fan-in 做格式校验和去重，整个流程可以在几小时内完成，相比逐篇手动处理节省数量级的时间。

**代码库批量 Review**：大型仓库的 PR 审查。把每个模块或文件分配给专项 agent，按安全、性能、可读性等维度并行审查，最后汇聚出统一的 review 报告。每个 agent 都在干净的上下文里工作，不互相污染判断。

**多语言文档并行翻译**：一份技术文档需要同时翻译成 8 种语言。串行跑 8 次显然低效，Fan-out 8 个翻译 agent 并行处理，输出一致的术语表由 Fan-in 节点统一校验。总耗时等于单次翻译，而不是 8 倍。

---

并行规模化的本质，不是让系统跑得更快，而是让那些本可以工业化的工作，真正按工业化方式运转。它的价值不在于单个 agent 有多聪明，而在于通过结构设计，把一群普通 agent 组织成一条可预期、可审计、可扩产的流水线。
