# Workflow 状态机：显式状态让自动化系统变得可信

## 一、五个核心概念

开始之前，先把概念说清楚。状态机涉及五个基本元素，每一个都有非常直觉的类比。

**状态（State）**：系统此刻"在哪里"。想象一个交通灯——它要么红，要么黄，要么绿。在任意时刻，它只有一个状态。对于 workflow 来说，状态就是"这个任务现在处于哪个阶段"，比如"等待审批中"或"LLM 审查中"。

**转移（Transition）**：从一个状态跳到另一个状态的动作。交通灯从红灯变绿灯，这就是一次转移。转移总是从一个明确的起点状态出发，到达一个明确的终点状态——没有"不知道从哪来、也不知道去哪"的漂移。

**条件/守卫（Guard）**：转移发生的前提条件。只有满足了守卫，转移才被允许发生。就像门禁——不刷卡，闸机不开。对于 workflow 来说，守卫就是"评分 ≥ 80 分才能进入下一步"或"敏感词检测通过才能继续"。

**动作（Action）**：在转移发生时、或进入某个状态时自动执行的操作。比如"进入等待审批状态时，发送一封邮件给审批人"。动作描述的是**做什么**，状态描述的是**在哪里**，两者分开定义，互不干扰。

**终态（Final State）**：流程结束的状态。可以有多个终态——比如"已发布"和"已拒绝"都是终态，含义截然不同。到达终态意味着这个任务的生命周期结束，不会再发生任何转移。

## 二、例子一：文档撰写 Workflow

用一个真实的文档生产流程来演示这五个概念如何协同工作。

假设一个技术团队需要发布一篇架构决策文档。状态链条是：

**初稿 → 初审 → 修订 → 终审 → 已发布 / 已归档**

**初稿阶段**：作者撰写完成，点击"提交"，触发一个转移动作。守卫条件是"文档字数 ≥ 300 字且内容不为空"。通过后，系统自动转移到**初审**状态，同时触发动作：启动 LLM 审查任务。这个转移是完全自动的，不需要人工干预。

**初审阶段**：LLM 对文档进行结构评分和可读性评估，输出一个 0–100 的分数。守卫条件是"分数 ≥ 60"。如果通过，自动转移到**修订**状态，并把 LLM 的建议列表附带传递过去；如果分数不足，转移回**初稿**状态，标注"需修改"。

这里有个微妙之处：LLM 给出的分数是概率性的，同一篇文档今天跑可能是 63，明天跑可能是 57。状态机的价值正在于此——不管 LLM 输出什么数字，系统总是能判断"通过"还是"不通过"并做出明确的状态转移决策，LLM 的不确定性被守卫条件截住了，不会向整个系统扩散。

**修订阶段**：作者根据建议修改文档，重新提交。这个转移需要**人工操作**（作者主动点击"提交修订"），系统不会自动跳转。人工操作完成后进入**终审**状态。

**终审阶段**：这是最关键的节点。如果文档涉及敏感技术决策或安全相关内容，守卫条件会要求"必须由高级工程师人工审批"——自动化在这里主动让位；对于普通技术文档，LLM 二次评分 ≥ 80 分可直接自动转移到**已发布**。

**终态有两个**：**已发布**（正式生效，进入知识库）和**已归档**（被拒绝或过期，留存记录但不展示）。两者都是终止状态，任务生命周期在这里画上句号。

这个例子展示了一个关键设计原则：**自动转移和人工转移可以共存**。状态机负责判断"什么情况下需要人介入，什么情况下机器自行处理"——而不是把所有决策交给自动化，也不是把所有决策都堆给人工审批员。

## 三、例子二：多模型并行 Research 任务

第二个例子更复杂，引入了**并行状态**的概念——也就是多个状态同时进行（fork-join 模式）。

假设你需要对一个技术问题做全面调研，决定让三个不同的子 agent 同时工作：Agent A 负责搜索学术论文，Agent B 负责分析竞品方案，Agent C 负责整理历史类似项目的经验教训。

**状态链条：**

初始化 → 并行调度 → [等待 A / 等待 B / 等待 C] → 汇总等待 → 冲突检测 → 人工裁决 / 自动合并 → 完成

**初始化**：接收任务描述，验证参数完整，生成三个子任务规范，进入**并行调度**状态。

**并行调度（Fork）**：这是一个分叉动作。系统同时启动三个独立的子状态机实例，每个实例有自己的内部状态（运行中 → 已完成 / 已失败）。主状态机随即进入**汇总等待**状态，不阻塞，不等待任何单个子任务。

**汇总等待（Join）**：这是一个 join（汇合）节点。主状态机在这个状态停留，直到守卫条件满足："三个子 agent 全部到达终态（成功或失败均可）"。超时保护：某个子 agent 超过 10 分钟没有完成，系统强制标记为"超时"并继续往下走，主流程不会因为一个迟到的子任务无限阻塞。

**冲突检测**：自动分析三份报告，用 LLM 检测结论是否存在直接矛盾。守卫判断：无矛盾则自动转移到**自动合并**，输出综合报告；存在核心结论冲突，则转移到**人工裁决**状态，把矛盾点高亮展示给人类决策者。

**人工裁决**：人类做出裁决后，触发转移，进入**完成**（终态）。

这个例子展示的核心是：并行的复杂性被完全封装在了状态机结构里。主状态机不需要感知三个子 agent 内部在做什么，它只关心一件事——"汇总等待的守卫条件是否满足了"。子任务的细节、重试逻辑、超时处理，全部由子状态机负责，对主流程透明。这种**关注点分离**，是状态机在复杂系统里最重要的工程价值之一。

## 四、状态机对 LLM Agent 的特别意义

做程序自动化时，用 if-else 链就够了，因为程序的输出是可预测的。

LLM 不是。同一个 prompt，不同时间、不同版本、不同温度参数，输出可能完全不同。这种不确定性在单次调用时可以接受，但在一个多步骤 workflow 里，它像多米诺骨牌一样级联传播——第一步的不确定输出影响第二步的判断，再影响第三步的数据，最终系统的行为变得完全不可解释。

状态机在这里扮演的角色，是一个**确定性的外壳**。

不管 LLM 在某个步骤输出了什么，状态机总能回答这个问题：**系统现在处于哪个状态？** 如果输出不满足进入下一状态的守卫条件，系统就停在当前状态，等待重试或人工介入——而不是带着这个有问题的输出继续往下跑，产生无法回溯的副作用。

更重要的是**检查点（Checkpoint）机制**。每个状态都是一个天然的存档点。如果 workflow 在第 7 个状态失败了，不需要从头重跑——直接从第 7 个状态恢复，重新执行这一步即可。在一个可能运行几十分钟、调用十几个 LLM 接口的复杂 workflow 里，这意味着巨大的成本节省和调试效率的提升。

这是一种工程上的成熟度：承认 LLM 不确定性的存在，然后用显式状态结构把它约束在可控范围内。

## 五、设计原则

最后，三条实践原则。

**状态数量应当稳定，不要随需求爆炸**。随着业务发展，不要把每一个新的业务变化都变成一个新状态——那只会让状态机变成一幅没人能看懂的地图。好的状态代表的是**核心阶段**，而不是每一个细节的分叉。当状态数量超过 10–15 个，通常意味着某些状态应该被合并，或者某段流程应该被拆出去作为一个独立的子状态机。

**转移条件必须显式，不能隐式**。"当 LLM 说可以的时候继续"是一个隐式条件，无法写进状态机，也无法被测试。转移条件必须是可以被代码判断的明确规则，比如 `score >= 80` 或 `flag == "approved"`。把 LLM 的自然语言输出转换成这样的结构化判断——这是构建 LLM workflow 的工程核心之一。

**门禁要挂在关键转移上，不是每步都挂**。不需要每个转移都有人工审批，但某些转移是系统的"关键路径"——一旦走过去就很难回头（对外发布、删除数据、触发付费操作）。这些转移上，应该始终挂着显式的质量门禁，不管上游的 LLM 有多自信。自动化系统最容易在"看起来没问题"的地方犯不可挽回的错误。
