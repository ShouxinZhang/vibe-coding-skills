# 依赖图：系统可演化性的骨架

## 一、为什么先看依赖

系统是否可持续演进，主要取决于依赖关系是否清晰，而不取决于某个模块是否写得优雅。

当依赖方向稳定，改动影响范围可预估；当依赖关系缠绕，任何小改动都可能引发连锁反应。

## 二、依赖方向是第一原则

建议采用单向依赖：能力层向下依赖基础层，禁止基础层反向调用上层。

在 Agent 工作流中，这意味着规则定义与执行能力分离，避免执行模块反向篡改策略模块。

## 三、三类高风险反模式

1. 循环依赖：A 依赖 B，B 反向依赖 A，导致变更无法局部收敛。
2. 过度横向依赖：同层模块互相调用过密，难以独立测试与发布。
3. 黑箱依赖：调用关系存在但未文档化，维护只能靠口口相传。

## 四、轻量治理规则

1. 明确分层：先定义层，再写依赖。
2. 约束入口：每个模块只暴露少量稳定入口。
3. 可视化最小图：维护一份关键模块依赖图，按迭代更新。
4. 周期巡检：每次大改动后检查是否引入新环。

## 五、对业务的直接价值

依赖图治理的业务回报很直接：

1. 缩短变更评估时间。
2. 降低发布连带风险。
3. 提高并行开发能力。

## 六、结论

依赖图不是架构文档的装饰，而是系统治理的执行工具。方向清晰，自动化才有边界；边界清晰，规模化才可持续。
