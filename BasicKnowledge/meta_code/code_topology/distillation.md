# 提炼：从噪音中提取真实约束

## 一、需求的常态是混杂

业务需求天然混有背景故事、历史包袱与个人偏好。把原始对话直接交给模型，常会得到"语义上正确、业务上偏题"的实现。

提炼的目标不是压缩字数，而是识别哪些信息真正决定系统行为。

## 二、提炼的定义

提炼是一个结构化过程：

1. 区分目标与叙述。
2. 区分硬约束与软偏好。
3. 区分必须一致与可以试错。

当这些边界被写清楚，LLM 才能稳定产出可复用结果。

## 三、常见噪音来源

1. 上下文污染：故事细节被误当功能约束。
2. 隐含常识：说话者默认前提未显式说明。
3. 经验迁移过度：旧方案被不加判断地复制到新问题。

## 四、提炼的四步法

1. 分层拆分：业务目标、功能需求、非功能约束、背景信息分开写。
2. 不可妥协筛选：逐条问"若改变它，业务是否失败"。
3. 反例验证：尝试相反实现，验证约束是否真实必要。
4. 最小约束清单：输出一页以内、可直接执行的约束列表。

## 五、业务价值

提炼完成后，团队获得三项收益：

1. 降低返工：减少因误解造成的二次开发。
2. 提升协同：产品、研发、测试围绕同一约束对齐。
3. 稳定自动化：Agent 在清晰边界下执行，行为更可预测。

## 六、结语

提炼不是文档工作，而是系统设计的入口。约束越清晰，自动化越可靠。
