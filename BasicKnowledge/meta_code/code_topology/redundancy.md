# 冗余：系统的自我保险

## 一、冗余是什么，又不是什么

很多团队把"冗余"等同于"浪费"。但在高不确定性系统里，冗余的本质是用确定成本交换可恢复性。

在 Meta Code 语境中，冗余不是简单的重复代码，而是刻意设计的兜底机制：当 LLM 或自动化流程在边界场景失效时，系统仍然可检测、可回退、可恢复。

## 二、冗余的三种层级

**1. 验证冗余**
同一关键结果由不同机制独立校验。例如规则校验、模拟执行、真实执行前预演三层并行。

**2. 路径冗余**
同一任务准备主路径和降级路径。主路径追求效率，降级路径保障稳定。

**3. 知识冗余**
同一业务约束以不同表达写入系统，避免单一描述失效后约束整体失真。

## 三、设计原则：相关失效最危险

冗余是否有效，关键不在"有几层"，而在"是否独立"。如果多个冗余机制共享同一失效模式，它们会一起失效。

因此应优先覆盖高后果、低可预测的失败场景，尤其是 LLM 在复杂上下文中的误判与幻觉。

## 四、冗余的代价

冗余有明确成本：

1. 延迟成本：更多校验意味着更长链路。
2. 一致性成本：多套规则若口径不一，会制造冲突。
3. 认知成本：机制过多会提高维护门槛。

冗余不是越多越好，而是越关键越要有。

## 五、实践方法

1. 写下失效假设：先定义"哪里可能坏"。
2. 选择正交机制：确保冗余路径不是同源故障。
3. 增强可观测性：每一层冗余都应有日志和信号。
4. 定期演练：主动注入失败验证兜底是否有效。

## 六、结论

Meta Code 追求的不是零错误，而是错误可控。冗余的价值在于把不可预期损害，转化为可管理事件。
