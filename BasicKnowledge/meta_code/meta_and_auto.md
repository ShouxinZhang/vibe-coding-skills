# Meta 与 Auto

## 一、两种力量

在 LLM 时代，有两种力量在共同塑造软件的生产方式。

**Meta** 是上游的力量。它是人类的意图、判断、约束与设计——写进 AGENTS.md 的规则，定义在 SKILL.md 里的工作流，埋在 prompt 中的边界条件。Meta 是慢的，是深思熟虑的，是需要人在场的。

**Auto** 是下游的力量。它是 LLM agent 在无人监督下的连续执行，是 CI 流水线的自动触发，是一个 agent 调用另一个 agent，任务在工具链里流转直到完成。Auto 是快的，是规模化的，是可以在你睡觉时运行的。

它们不是对立的。Meta 定义规则，Auto 执行规则。Meta 是棋局的设计者，Auto 是棋手。

---

## 二、没有 Meta 的 Auto 是危险的

当 Auto 的能力超过 Meta 的覆盖范围，问题就出现了。

一个 agent 在没有明确约束的情况下连续执行任务，它会做出"看起来合理"的决策——但"看起来合理"和"你真正想要的"之间，往往存在一道裂缝。它会删掉你觉得无用但其实关键的文件，会把一个临时的 hack 固化成永久的架构，会在你没有意识到的地方积累技术债。

这不是 agent 的错。这是 Meta 缺失的后果。

没有 Meta 的 Auto 就像一支训练有素、但没有战略目标的军队——执行力极强，方向完全错误。

---

## 三、没有 Auto 的 Meta 是浪费

反过来也成立。

如果你把所有的精力都花在写规则、设计约束、打磨 prompt，却没有让任何东西真正自动运转，那 Meta 只是一堆漂亮的文档。

真正的价值在于：Meta 被 Auto 读取、理解、执行之后产生的结果。一份写得极好的 SKILL.md，如果从未被 agent 调用，它的价值是零。

Meta 是为 Auto 而生的。设计 Meta 的终极目的，是让某种程度的 Auto 成为可能。

---

## 四、Meta 债

软件工程里有"技术债"的概念——为了短期速度而牺牲长期质量。

在 LLM 时代，有一种新的债务叫做 **Meta 债**。

当你的 Auto 系统跑得越来越快，覆盖的场景越来越多，但你的 Meta 层——那些定义行为边界的规则、约束、工作流——没有同步演进，Meta 债就在积累。

它的症状是：agent 的行为越来越难以预测，你不确定它在某个边界场景下会做什么，你开始不敢轻易让它自动运行。最终，你不得不把 Auto 退化回手动，因为 Meta 已经无法覆盖 Auto 的活动范围。

偿还 Meta 债的方式只有一种：停下来，把 Auto 跑出来的边界案例，重新转化为 Meta 的约束。

---

## 五、理想的循环

Meta 与 Auto 的健康关系，是一个自我强化的循环：

1. **人类设计 Meta** — 定义意图、规则、边界
2. **Auto 在 Meta 框架内执行** — 生成结果，也生成反馈
3. **反馈暴露 Meta 的盲区** — 哪些场景没有被覆盖，哪些规则产生了意外结果
4. **人类修订 Meta** — 用真实执行的经验补充和修正规则
5. **Auto 的可信度提升** — 可以在更大的范围内自主运行

每一轮循环，Auto 的自主空间都在扩大，同时 Meta 的质量也在提升。这是一个人机协作的进化过程，不是人被替代的过程。

---

## 六、你坐在哪一侧

这个问题值得每个人问自己。

大多数人自然地停留在 Auto 侧——执行任务，响应需求，产出结果。这很重要，但在 LLM 时代，这一侧的价值正在被自动化压缩。

Meta 侧的稀缺性更持久：设计系统应该如何运作，定义 agent 应该遵守什么规则，判断自动化的边界在哪里。这需要对业务的深度理解、对系统的全局视角、以及对"够用"与"过度"的持续校准。

不是所有人都需要完全站在 Meta 侧。但在一个 Auto 越来越强大的世界里，能够在两侧之间自由切换的人，才是真正不可替代的。
