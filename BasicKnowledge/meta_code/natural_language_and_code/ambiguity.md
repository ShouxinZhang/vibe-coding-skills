# 自然语言的歧义

## 一、语言的模糊是一种特性，不是缺陷

**自然语言的模糊性**不是缺陷，而是特性。它让人类可以在有限的词汇中表达无限的意思，适应了社交沟通中的情感、上下文、隐喻与留白。

但代码世界是另一个宇宙。每一个分支都必须被覆盖，每一个变量都必须有具体的类型，每一个决策都必须有明确的条件。代码不能说"大概""可能""有时候"。

当自然语言作为意图表达工具进入代码生成流程，这种特性变成了**隐形的陷阱**。

---

## 二、歧义的四种类型

**词汇歧义**源于同一个词在不同语境下指代不同的事物。"删除用户"可能是删除账户本身，也可能是从列表中移除。"支持用户"可能是技术支持，也可能是权限赋予。LLM 必须猜测，而它的猜测往往基于训练数据中最频繁的用法。

**结构歧义**出现在复杂的句子中。"支持多用户和管理员登录"——是说两个独立的功能，还是用户和管理员使用同一套登录流程？"在错误时重试三次"——是重试整个操作三次，还是重试三次然后放弃？

**语用歧义**不在词汇或语法，而在隐含的假设。"做个搜索功能"隐含了多少假设：需要模糊匹配吗？大小写敏感吗？需要分页吗？要实时反馈吗？这些问题在人类的日常沟通中会被追问补充，但在 prompt 中通常被跳过了。

**指代歧义**源于代词与指示词的模糊指向。"在它出错时发邮件"——"它"是哪个模块？是"API 调用"还是"数据库操作"？"修改后立即同步"——"修改"指哪一类操作？这些不确定性看似微小，却能导致完全不同的实现。

---

## 三、LLM 如何处理歧义——它会选择，不会询问

当 LLM 面对歧义时，它不会停下来问你。它会从训练数据的**隐形分布**中采样一个答案——通常是概率最高的那个解释。

这个答案往往是正确的。大多数时候，LLM 的"最可能的理解"与你的真实意图是对齐的。但"往往"不是"总是"。而那个不对的时刻，往往恰好是最难被发现的时刻。

因为代码看起来完全正常。测试可能也能通过。它只是在特定场景、特定边界条件下，才会暴露真实意图与实现之间的偏差。

---

## 四、歧义的渗透——从 Prompt 到隐形的决策

模糊的意图经过 LLM 的**理解层**时，会被逐步固化为具体的实现决策。这个过程是单向的："给我一个搜索功能"流经模型变成"用线性扫描，支持前缀匹配，返回前十条结果"。

这种转化没有中间过程可见。代码本身没有留下"这里存在一个假设"的标记。崩溃堆栈也不会在这里出现。调试器无法在这里设断点。

问题只在于：这个转化是 LLM 的理解，而不是你的意图。三个月后，当需求真正落地时，你才发现"我们从来没说过要前缀匹配"。

---

## 五、意图精确是一种新的写作技艺

应对歧义不是要求你写出更多的文字，而是在**歧义的临界点**上写得精确。

首先要识别歧义点。在你的 prompt 中，哪些词有多种合理解释？哪些假设你认为是"显而易见"的，但实际上对于模型可能不显而易见？

消除歧义的方式很多。举反例——"不是 X"能比肯定句更清晰地界定边界。显式声明假设——"我们假设用户已认证"能阻止模型在这里自行决策。给定具体的测试案例——"当输入为 null 时应返回空列表"让意图变得可验证。

这不是在与 LLM 玩文字游戏，而是把你脑中的隐形假设变成可见的约束，让生成的代码更接近真实的意图。
