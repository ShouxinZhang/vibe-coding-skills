# 自然语言与精确代码

## 一、两种语言，两种世界

人类用自然语言思考，用代码控制机器。

这两件事从来都不是一回事。自然语言生长于社会，充满上下文、惯例与隐含前提；代码生长于逻辑，要求消除歧义、穷举边界、声明所有假设。一句"帮我做个登录功能"和一个通过所有边界测试的认证模块之间，横亘着一道深渊。

LLM 被放在了这道深渊的中间。

它的工作是翻译——把人类语言里模糊的意图，翻译成机器语言里精确的指令。这是 LLM 时代最核心的工程挑战，也是理解 Meta Code 的真正起点。

---

## 二、为什么这道鸿沟值得认真对待

表面上，这个问题已经被"解决"了——你说，LLM 写，代码跑起来了。

但这种便利掩盖了一个危险：**翻译看起来成功，并不意味着翻译是正确的**。

自然语言允许模糊，代码不允许。自然语言依赖共识，代码只依赖定义。当你说"快一点"，人类听众会凭经验判断"快多少"；LLM 生成的代码则会做出一个具体的选择——而那个选择，可能和你心里想的完全不同，只是你没有注意到。

错误不发生在运行失败的时候，它发生在代码安静地做了一件错误的事的时候。

这正是 Meta Code 实践者必须直面的核心问题：不是"能不能生成代码"，而是"生成的代码和你的意图之间，差距有多大，你有没有能力发现它"。

## 四、从语言学到工程学：这道鸿沟的历史

**1950年代——图灵测试的误导**
图灵问的是"机器能否模拟人类语言行为"，而不是"机器能否理解人类语言"。这个区分被轻易忽略了几十年。人们以为语言理解是 AI 的终点，实际上它只是起点。

**1970–1990年代——自然语言处理的第一波**
规则系统和早期统计模型尝试把自然语言"编译"成结构化查询。ELIZA、SHRDLU 展示了可能性，也暴露了彻底的局限：领域之外一无所知，稍微换个说法就崩溃。

**2000–2020年代——深度学习接管语言**
Word2Vec、BERT、GPT 系列把语言表示推进到前所未有的深度。但"能表示语言"和"能把语言精确映射到代码逻辑"仍然是两件事。模型越来越擅长生成"看起来合理的代码"，但"合理"不等于"正确"。

**2022年之后——这道鸿沟成为工程问题**
ChatGPT 之后，数百万人开始用自然语言生成代码。鸿沟不再是学术话题，它成了每一个 vibe coding 实践者每天都在踩的坑。歧义、规格缺失、验证困难——这三个问题从理论走进了日常生产。

## 五、阅读建议

如果你是第一次进入这个模块，建议按文件顺序阅读：先读 `ambiguity.md` 建立直觉，理解错误是从哪里进来的；再读 `specification.md` 理解为什么问题无法被"写更详细的需求"彻底解决；最后读 `verification.md`，面对这道鸿沟，找到实际可行的应对策略。

前置背景：阅读本模块前，建议先读 `meta_code/introduction.md`，理解 Meta Code 的整体框架——本模块是其"意图表达能力"维度的深度展开。
