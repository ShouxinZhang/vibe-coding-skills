# 规格的幻觉

## 一、"规格"的幻觉

人人都相信，只要需求写得足够详细、足够精确，语言模型或者任何聪明的开发者就能从中生成出正确的代码。这个信念很强大，以至于我们倾向于认为所有 bug 都来自于"规格不清楚"。

但这其实是一个关于**完备性**的幻觉。自然语言规格描述的，永远只是"正常情况下的主流程"——那些我们觉得"显而易见"的东西。而真实的程序需要的，远不止这些。

---

## 二、未覆盖的角落：省略不是遗忘

每一份自然语言规格都隐含着大量未说出口的假设。当一个产品经理说"做一个文件上传功能"，ta 实际上没有说：文件大小的上限是多少？允许哪些格式？能否同时上传五十个文件？如果用户在上传到一半时网络断开了，怎么恢复？

这不是规格写得不够仔细，这是语言本身的特性——我们用自然语言交流时，总是假设对方知道"显然的事情"。但一个 LLM 对"显然"的理解，可能和你完全不同。

那些被默认为"显然"的东西，正是最容易滋生 bug 的地方。

---

## 三、不完备定理的工程投影

1931 年，哥德尔证明了他的不完备定理：任何足够强的、一致的形式系统，都存在在该系统内无法被证明也无法被证伪的命题。

这个数学真理对规格也成立。任何用自然语言写成的规格，必然存在它无法决定的**边界情况**。这不是因为规格写得不够好，而是语言本身的结构性限制。语言是为了传达主流的、通常的、被广泛接受的观念而设计的——对于那些异常的、不属于主流的、需要特殊处理的情况，语言天然地会显得模糊和无能为力。

Meta 债不是工程懈怠，它是哥德尔定理在工程世界里的投影。承认规格的不完备性，是成熟 Meta Code 实践者的第一步。

---

## 四、LLM 填补空白的方式

面对规格中的空白，LLM 不会拒绝执行。它会用"最可能的实现"来填补那些没有明确说明的地方。

它的填补依据是什么？是训练数据中的**统计规律**——也就是说，大多数人在类似情况下是怎么做的。这意味着：如果你的边界情况符合行业的主流惯例，LLM 的实现大概率没问题。但如果你的系统有任何不寻常的约束、任何偏离常规的特殊需求，那里就是 bug 最可能藏身的地方。

---

## 五、从"完备描述"到"关键约束"

既然完备的规格描述是不可能的，应对策略就不是"写更多"。

而是学会识别：哪些约束是你独有的，不属于行业的默认值？哪些约束隐含在你的商业逻辑里，但不会自动出现在任何开发者的直觉中？把这些**不寻常的约束**显式地写出来。

反常识的地方，才是规格真正需要覆盖的地方。写规格不是为了面面俱到，而是用最小的言语成本，把那些容易被误解的关键约束标记出来，让任何实现者都无法忽视。
