# 验证困境

## 一、验证的前提：你需要知道"正确"是什么

验证代码是否正确，首先需要一个明确的参照——一个被称为"规格"的东西。但当规格是用自然语言写成的时候，"正确"本身就变成了一个模糊的概念。

这就是**验证困境**的根源。你在用一把标尺去量一根绳子，但这把标尺本身就是弯曲的，绳子的两端也没有明确的定义。

自然语言的歧义性是逃不开的。每一个词可能有多种理解，每一个句子可能隐含多层意思。当 LLM 生成代码时，它做出的选择是基于对自然语言规格的某种理解——但这种理解，可能与你的理解完全不同。

---

## 二、"感觉对了"的危险

人类验证 LLM 生成代码的典型方式就是"跑一下看看有没有报错"或"试几个示例能不能跑通"。这个过程中，你会获得一种错误的安心感。

代码没有崩溃、执行完成、输出看起来合理——这些都被当作"正确"的证据。但这只是**最弱的验证**。它能发现明显的语法错误、明显的运行时崩溃，但却无法发现更隐蔽的问题。

"感觉对了"最危险的地方在于：代码完全可能按照 LLM 的某种理解完美地运行，而这个理解与你的真实意图南辕北辙，但你永远不会知道，因为代码没有崩溃。问题没有暴露，不等于问题不存在。

---

## 三、三种验证手段的能力边界

**人工审查**依赖的是审查者对代码意图的理解。但审查者和 LLM 可能会犯同样的模糊性错误——当两个人都从同一份模糊的自然语言规格出发，他们可能会携手走向同一个错误的方向。审查者的审查能力，被整个自然语言鸿沟的宽度所限制。

**测试覆盖**听起来更科学，但本质上面临同样的困境。测试用例本身也是用自然语言意图写出来的，如果规格本身有歧义，测试用例也会承继这个歧义。测试覆盖率度量的是代码路径的覆盖，无法度量业务语义的覆盖——你可能用 100% 的路径覆盖率，却完全没有验证 LLM 是否真正理解了你的意图。

**形式验证**能够给出数学上的正确性保证，但代价是：你必须把自然语言规格转化为形式语言。这个转化过程本身，就是在与自然语言鸿沟做最后的对抗。一旦规格被严格地形式化，此前隐藏的歧义就会被一一暴露出来。

---

## 四、验证的真正目标：消除假设

如果验证不是"确认代码能跑"，那它到底是什么？

答案是：验证的目标是**消除假设**。代码中的每一行都隐含着某个选择，有些选择是显然的，但有些是隐含的。LLM 做出了这个选择——但它为什么做出来？是因为理解正确，还是因为碰巧猜对了？

每当你发现一个 LLM 隐含做出的假设，并显式地确认它——"是的，这就是我想要的"或"不对，我实际上想要的是另一种"——你就完成了一次有效的验证。验证的真正工作，是把代码里每一个非显然的选择翻译回自然语言，再问自己：这是我的意图吗？

---

## 五、实践策略：从意图倒推验证点

在写 prompt 的时候，同时写下你能想到的"我不希望它做什么"。这些**负面约束**往往比正面约束更有力量，因为它们直指那些最容易被误解的地方。

每一个歧义点（见 `ambiguity.md`）都是一个验证点；每一个规格的空白（见 `specification.md`）都暗示着代码中会有某种隐含假设。找到这些点，逐一确认。

Meta Code 实践者的审查能力，本质上是把隐性假设不断显性化的能力。你不是在"检查代码对不对"，而是在把代码上面蒙着的那层迷雾揭开，看清楚 LLM 到底在做什么，然后判断：这是你想要的吗。
