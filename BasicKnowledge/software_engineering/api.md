# API 设计范式

---

## 一、API 是什么

应用程序接口（Application Programming Interface，API）是软件系统之间通信的标准化约定。用插座类比：API 是统一规格的电器接口，调用方只需按规范插入，无需了解内部电路设计。

### 三个层面的 API

| 层面 | 说明 | 示例 |
|------|------|------|
| 操作系统 API | 操作系统向应用提供的接口 | 文件读写、内存分配、进程管理 |
| 库/SDK API | 语言库或工具包的编程接口 | Python `requests`、Java `java.util` |
| Web API | 通过网络（HTTP）提供的服务接口 | GitHub API、微信支付、地图服务 |

本文重点讨论 Web API。

### HTTP 方法语义

| 方法 | 语义 | 幂等性 | 典型使用 |
|------|------|--------|---------|
| GET | 获取资源 | 是 | 查询，不修改状态 |
| POST | 创建资源 | 否 | 提交新数据 |
| PUT | 完全替换资源 | 是 | 整体更新 |
| PATCH | 部分更新 | 否 | 修改部分字段 |
| DELETE | 删除资源 | 是 | 移除指定资源 |

**幂等性**（Idempotency）：多次调用与单次调用结果相同。DELETE 一个不存在的资源仍返回 200/204，不会抛出不同结果。

## 二、REST（表述性状态转移）

REST（Representational State Transfer）是最主流的 Web API 架构风格，基于 HTTP 方法与资源 URL 的组合来操作数据。

### 2.1 六条核心约束

1. **无状态**：每个请求包含处理所需的全部信息，服务端不保存客户端上下文
2. **统一接口**：资源被明确标识（URL），通过标准方法操作
3. **客户端-服务器分离**：双方独立演进，通过接口解耦
4. **可缓存**：响应标注是否可缓存，客户端/代理层可复用
5. **分层系统**：客户端不知道自己是否通过代理连接，支持负载均衡、网关等中间层
6. **按需代码**（可选）：服务端可下发可执行代码（如 JS）给客户端

### 2.2 资源与 URL 设计

REST 将数据建模为资源，通过 URL 路径层次标识：

```
GET    /users            → 获取用户列表
GET    /users/123        → 获取 ID=123 的用户
POST   /users            → 创建新用户
PUT    /users/123        → 整体更新用户 123
DELETE /users/123        → 删除用户 123
GET    /users/123/orders → 获取用户 123 的订单
```

### 2.3 HTTP 状态码语义

| 类别 | 常见状态码 | 含义 |
|------|----------|------|
| 2xx 成功 | 200 OK | 请求成功 |
| | 201 Created | 资源创建成功（POST 后返回） |
| | 204 No Content | 成功但无响应体（DELETE 后返回） |
| 3xx 重定向 | 301 Moved Permanently | 资源永久移动 |
| | 304 Not Modified | 缓存未失效，使用本地缓存 |
| 4xx 客户端错误 | 400 Bad Request | 请求格式或参数错误 |
| | 401 Unauthorized | 未提供有效认证 |
| | 403 Forbidden | 已认证但无权限 |
| | 404 Not Found | 资源不存在 |
| 5xx 服务端错误 | 500 Internal Server Error | 服务端未知错误 |
| | 503 Service Unavailable | 服务暂时不可用 |

### 2.4 REST 的局限

- **过度获取（Over-fetching）**：服务端返回字段多于客户端所需，浪费带宽
- **获取不足（Under-fetching）**：获取完整数据需要多次请求（如先拿用户，再拿订单）

这两个问题催生了 GraphQL。

## 三、GraphQL

GraphQL 是 Facebook 开源的查询语言与运行时，将"决定返回什么数据"的控制权从服务端转移到客户端。

### 3.1 核心思想

客户端精确声明所需字段，服务端按需返回，彻底消除过度获取与获取不足。所有请求通过单一端点（`/graphql`）发送，用查询语言描述数据形状。

### 3.2 三种操作

- **Query（查询）**：读取数据，等价于 GET
- **Mutation（变更）**：修改数据，等价于 POST/PUT/DELETE
- **Subscription（订阅）**：建立长连接，服务端主动推送数据变化

### 3.3 GraphQL vs REST 对比

| 维度 | GraphQL | REST |
|------|---------|------|
| 数据获取灵活性 | 高——客户端定制字段 | 低——服务端固定响应结构 |
| 学习曲线 | 陡——需掌握查询语言与 Schema | 平——HTTP 方法直觉易懂 |
| HTTP 缓存 | 难——所有请求均为 POST | 容易——GET 可直接缓存 |
| 适合场景 | 复杂数据需求、多端适配、微服务聚合 | 简单 CRUD、公开 API、资源导向 |

GraphQL 在移动端（减少流量）、Feed 流（字段组合频繁变化）、数据聚合网关（整合多微服务）场景优势明显。

## 四、gRPC

gRPC 是 Google 开发的高性能远程过程调用（Remote Procedure Call，RPC）框架，基于 Protocol Buffers（protobuf）实现强类型接口定义。

### 4.1 核心特点

- **接口定义**：用 `.proto` 文件定义服务与消息类型，自动生成多语言客户端和服务端代码
- **传输格式**：二进制序列化，相比 JSON 消息更小、解析更快
- **传输协议**：HTTP/2，支持多路复用，一个 TCP 连接并行处理多个请求

### 4.2 四种通信模式

| 模式 | 说明 | 适用场景 |
|------|------|---------|
| Unary | 单请求 + 单响应 | 普通 API 调用 |
| Server Streaming | 单请求 + 持续响应流 | 日志订阅、数据下载 |
| Client Streaming | 连续请求流 + 单响应 | 批量上传、文件传输 |
| Bidirectional Streaming | 双向流式通信 | 实时协作、语音识别 |

### 4.3 gRPC vs REST 对比

| 维度 | gRPC | REST |
|------|------|------|
| 序列化格式 | 二进制（protobuf） | 文本（JSON） |
| 性能 | 高——消息体积小、解析快 | 中——JSON 解析有开销 |
| 类型安全 | 强——protobuf Schema 强约束 | 弱——JSON 弱类型 |
| 浏览器直接调用 | 困难——需要 grpc-web 代理 | 原生支持 |
| 适合场景 | 微服务内部通信、高性能场景 | 公开 API、浏览器调用 |

## 五、WebSocket 与实时通信

传统 HTTP 是请求-响应模型，服务端无法主动推送数据。以下是实时通信方案的演进：

### 5.1 三种方案对比

| 方案 | 原理 | 延迟 | 服务端压力 | 场景 |
|------|------|------|---------|------|
| HTTP 轮询（Polling） | 客户端定时轮询 | 高（受间隔限制） | 高 | 简单通知，低频 |
| 长轮询（Long Polling） | 服务端延迟响应至有数据 | 中 | 中 | 对结构改造要求低的场景 |
| WebSocket | 持久全双工连接 | 低 | 低 | 实时聊天、游戏、协作 |

### 5.2 WebSocket 握手

WebSocket 连接由 HTTP 请求升级而来：客户端发送 `Upgrade: websocket` 头部请求升级，服务端返回 `101 Switching Protocols`，此后双方维持持久连接，可随时双向发送数据。

### 5.3 Server-Sent Events（SSE）

SSE 是服务端向浏览器单向推送事件的轻量方案（基于 HTTP 长连接），浏览器通过 `EventSource` API 监听。相比 WebSocket 更简单，适合只需服务端推送的场景（如通知、日志实时滚动、AI 流式输出）。

## 六、API 设计最佳实践

### 6.1 版本管理策略

| 策略 | 示例 | 优点 | 缺点 |
|------|------|------|------|
| URL 路径版本 | `/v1/users` | 直观，易调试 | URL 存在版本号略显冗余 |
| 请求头版本 | `Accept: application/vnd.api+json;version=1` | URL 保持整洁 | 需要明确约定，不易感知 |

### 6.2 接口文档与契约

OpenAPI（原 Swagger）是 API 契约的行业标准：机器可读的 JSON/YAML 格式，可自动生成交互式文档、客户端 SDK、服务端代码框架。保持文档与代码同步是 API 长期维护的关键纪律。

### 6.3 认证机制对比

| 方案 | 原理 | 安全性 | 适用场景 |
|------|------|--------|---------|
| API Key | 请求头携带密钥 | 中 | 服务间调用、内部工具 |
| OAuth 2.0 | 第三方授权流程，签发访问 Token | 高 | 第三方集成、社交登录 |
| JWT | 无状态含签名的 Token | 高 | 微服务、移动端、分布式 |

### 6.4 限流与幂等性

**限流（Rate Limiting）**：限制单位时间内调用次数，防止滥用。通常在响应头中返回剩余配额（`X-RateLimit-Remaining`）和重置时间，调用方可据此控制频率。

**幂等性**：POST 请求默认不幂等（重复提交会创建多条记录）。通过客户端在请求头提供唯一幂等键（Idempotency Key），服务端对相同键的重复请求只处理一次，防止网络重试导致重复操作。
