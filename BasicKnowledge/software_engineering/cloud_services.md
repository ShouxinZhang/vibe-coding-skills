# 云服务基础

---

## 一、云计算的本质

云计算通过网络提供计算、存储、数据库等资源，用户按需付费，无需前期采购硬件。

### 1.1 传统部署的痛点

- **前期资本投入大**：购买服务器、网络设备、机房租赁，资金回流周期长
- **资源利用率低**：按高峰容量采购，平时大量机器闲置
- **运维负担重**：硬件故障、系统补丁、空调断电——7×24 小时值班
- **扩展受限**：新增服务器从采购到上线需数周，无法应对突发流量

### 1.2 云计算核心价值

| 特性 | 优势 |
|------|------|
| 按需付费 | 用多少付多少，无预付资本 |
| 弹性伸缩 | 分钟级增删资源，应对流量波动 |
| 全球部署 | 就近部署降低延迟，内置多区域灾备 |
| 无硬件采购 | 物理机、机房、网络全部外包 |

### 1.3 三种部署模型

| 模型 | 说明 | 适合场景 |
|------|------|---------|
| 公有云（Public Cloud） | 多租户共享基础设施，AWS/Azure/GCP | 成本敏感、快速扩张的互联网业务 |
| 私有云（Private Cloud） | 企业独占，部署在自有或托管数据中心 | 监管合规要求高、数据主权敏感行业 |
| 混合云（Hybrid Cloud） | 核心/敏感业务用私有云，其他用公有云 | 传统大型企业的渐进式上云路径 |

## 二、服务模型：IaaS / PaaS / SaaS

用餐饮类比理解三层责任边界：

- **IaaS = 租厨房**：平台提供钢筋炉灶（硬件），你负责买食材、做菜（操作系统到应用全部自管）
- **PaaS = 半成品套餐**：平台管好厨房和基础流程，你只需要决定菜谱（专注应用代码）
- **SaaS = 成品外卖**：直接拿来用，平台负责一切（软件即服务）

### 2.1 责任边界对比

| 服务组件 | IaaS | PaaS | SaaS |
|---------|------|------|------|
| 物理硬件 | 服务商 | 服务商 | 服务商 |
| 虚拟化/网络 | 服务商 | 服务商 | 服务商 |
| 操作系统 | 用户 | 服务商 | 服务商 |
| 运行时/中间件 | 用户 | 服务商 | 服务商 |
| 应用代码 | 用户 | 用户 | 服务商 |
| 数据 | 用户 | 用户 | 用户 |

### 2.2 典型产品

- **IaaS**：AWS EC2、Azure 虚拟机、Google Compute Engine——提供虚拟化计算资源，控制粒度最细
- **PaaS**：Heroku、Google App Engine、Vercel、Railway——面向开发者的应用托管，推送代码即部署
- **SaaS**：Gmail、Salesforce、Notion、GitHub——直接在浏览器中使用，零运维

## 三、主流云服务商生态

### 3.1 全球三大巨头

| 厂商 | 市场份额（2025） | 优势领域 | 适合场景 |
|------|--------------|---------|---------|
| AWS | ~33% | 服务最多（300+）、生态最成熟 | 传统企业迁云、大型互联网业务 |
| Azure | ~23% | 与微软软件生态无缝集成 | Office 365 用户、.NET 应用、混合云 |
| GCP | ~12% | 数据分析（BigQuery）、AI/ML 能力强 | 数据密集型、机器学习训练推理 |

### 3.2 国内云

阿里云（市场份额第一）、腾讯云（社交游戏优势）、华为云（2B 企业市场）。相比国际厂商定价更低、国内网络延迟更小，但国际扩展能力较弱。

### 3.3 新兴开发者平台

Cloudflare（边缘计算）、Vercel（前端托管）、Supabase（开源数据库托管）等垂直平台正获得开发者青睐——零运维配置、全球边缘加速、开源友好，适合中小团队快速上线产品。

## 四、核心云服务分类

### 4.1 计算服务（Compute）

| 服务类型 | 代表产品 | 冷启动 | 运维负担 | 弹性 | 适合场景 |
|---------|---------|--------|---------|------|---------|
| 虚拟机（VM） | EC2、Azure VM | 秒级 | 中等 | 分钟级 | 传统应用迁移、7×24 长期运行 |
| 容器编排 | EKS (AWS)、GKE (GCP) | 秒级 | 高 | 秒级 | 微服务架构、DevOps 成熟团队 |
| 无服务器（FaaS） | Lambda、Cloudflare Workers | 毫秒~秒 | 极低 | 毫秒级 | 事件处理、API 端点、定时任务 |

无服务器（Serverless）的计费粒度是函数执行时间，真正实现"不运行不花钱"。代价是冷启动延迟（首次调用需初始化）和对长连接、有状态操作的限制。

### 4.2 存储服务（Storage）

**对象存储（Object Storage）**：S3（AWS）、GCS（GCP）。存储图片、视频、日志、备份，支持高并发读取，通常配合 CDN 分发。按存储量和请求次数计费，无容量上限。

**块存储（Block Storage）**：EBS（AWS）。以数据卷形式挂载到虚拟机，适合数据库数据文件、高频 I/O 场景。性能可选（SSD 或 HDD）。

**文件存储（File Storage）**：EFS（AWS）、Azure Files。多个实例共享挂载，适合 NFS 应用和分布式计算。

**CDN（内容分发网络，Content Delivery Network）**：静态资源复制到全球边缘节点，用户请求就近获取，将端到端延迟降低 20-50%，同时减轻源站压力。

### 4.3 托管数据库（Managed Database）

托管数据库由云服务商代为管理：自动备份与时间点恢复、主从自动切换（高可用）、性能监控与告警、安全补丁更新。企业因此省去大量 DBA 运维工作。

| 数据库类型 | AWS 产品 | GCP 产品 | 适用场景 |
|-----------|---------|---------|---------|
| 关系型 | RDS（PostgreSQL / MySQL） | Cloud SQL | 结构化数据、强事务、复杂查询 |
| NoSQL 键值 | DynamoDB | Firestore | 超高并发读写、实时应用 |
| 缓存 | ElastiCache（Redis / Memcached） | Memorystore | 会话存储、热数据加速 |

## 五、云原生（Cloud Native）理念

### 5.1 十二因素应用（The Twelve-Factor App）

由 Heroku 工程师提出的云原生应用设计法则，核心几条：

- **配置与代码分离**：数据库地址、密钥等通过环境变量注入，不硬编码在源码中
- **无状态进程**：每个进程实例无本地状态，可随时销毁重建，支持横向扩展
- **后端服务当作资源**：数据库、缓存、消息队列通过配置替换，不依赖具体位置
- **日志视为事件流**：进程只输出到标准输出（stdout），由平台聚合与持久化

### 5.2 微服务 vs 单体

微服务并不是银弹：

| 架构 | 优势 | 劣势 | 适合阶段 |
|------|------|------|---------|
| 单体（Monolith） | 开发快、部署简单、调试容易 | 规模增长后难以维护 | 早期创业、小团队 |
| 微服务（Microservices） | 服务独立部署、故障隔离、技术栈灵活 | 分布式复杂性高、运维成本大 | 大团队、成熟产品 |

过早拆分微服务是常见的工程过度设计陷阱。

### 5.3 基础设施即代码（Infrastructure as Code，IaC）

Terraform、Pulumi 等工具用声明式代码描述基础设施状态（网络拓扑、虚拟机规格、数据库配置），提交到 Git 版本控制。好处是：环境可复现（灾备重建）、变更可 Code Review、历史可追溯。

### 5.4 可观测性三支柱（Observability）

现代云原生系统出问题不能靠登录服务器排查，需要完善的可观测性体系：

| 支柱 | 作用 | 工具示例 |
|------|------|---------|
| 日志（Logs） | 记录事件，定位具体错误 | ELK Stack、CloudWatch Logs |
| 指标（Metrics） | 收集 CPU/内存/请求数等数值，发现趋势与异常 | Prometheus + Grafana |
| 追踪（Traces） | 记录分布式调用链，定位性能瓶颈 | Jaeger、AWS X-Ray |

三者缺一不可：指标告警触发排查，日志定位错误，追踪还原调用路径。
