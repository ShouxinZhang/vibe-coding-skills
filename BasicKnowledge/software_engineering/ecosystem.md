# 软件工程生态

软件工程生态是现代开发的基础设施，涵盖工具链、版本控制、依赖管理与自动化流程。这些体系共同确保代码的可维护性、可扩展性与交付可靠性。

---

## 一、技术栈全景

技术栈（Technology Stack）是构建软件系统所需技术组件的集合，包括编程语言、框架、数据库、部署环境等。选择合适的技术栈决定了开发效率、运维成本与系统性能。

| 栈名称 | 语言 | 前端框架 | 后端框架 | 数据库 | 部署 |
|--------|------|---------|---------|--------|------|
| LAMP | PHP | — | Apache + PHP | MySQL | 虚拟主机/服务器 |
| MEAN | JavaScript | Angular | Node.js/Express | MongoDB | 云平台 |
| 现代云原生 | Python/Go | React/Vue | FastAPI/Gin | PostgreSQL | Kubernetes/Docker |
| 企业 Java | Java | React | Spring Boot | Oracle | 企业私有云 |

同一个业务目标可以用不同技术栈实现，选型更多是团队熟悉度与生态成熟度的权衡，而非绝对的优劣之分。

## 二、版本控制系统

版本控制（Version Control）记录代码的完整变更历史，是团队协作的核心基础设施。

### 2.1 Git 核心概念

- **仓库（Repository）**：存储代码与全部变更历史的容器
- **分支（Branch）**：独立的开发线，允许多人并行工作互不干扰
- **提交（Commit）**：一次代码变更的原子快照，附带作者、时间、说明
- **合并（Merge）**：将一个分支的变更整合到另一个分支
- **变基（Rebase）**：改写提交历史，将一组提交重新应用到另一分支，保持线性历史

### 2.2 主流平台定位

- **GitHub**：最大开源社区，强大的 Code Review 与 CI/CD 集成，企业版提供供应链安全功能
- **GitLab**：一体化 DevOps 平台，内置 CI/CD 与制品库，适合自托管的企业场景
- **Bitbucket**：与 Jira 深度集成，专为需要项目管理联动的企业设计

### 2.3 协作流程对比

| 流程 | 分支策略 | 适用场景 | 发布频率 |
|------|---------|---------|---------|
| GitHub Flow | main + feature 分支，PR 驱动 | 高频迭代，小型团队，持续部署 | 每次合并即发布 |
| Git Flow | develop + release + hotfix 多层分支 | 版本化发布，大型项目，有明确版本周期 | 周期性发布 |

## 三、包管理与依赖

包管理器（Package Manager）自动处理第三方库的安装、更新与依赖关系解析，避免手动下载和版本冲突。

### 3.1 各语言生态

- **npm / pnpm / yarn**（Node.js）：最大生态，周边工具完整，node_modules 体积问题突出
- **pip + venv / Poetry**（Python）：简洁易用，虚拟环境隔离不同项目的依赖
- **Maven / Gradle**（Java）：声明式依赖管理，Maven 生态庞大，Gradle 构建脚本更灵活
- **Cargo**（Rust）：内置包管理，依赖安全默认可控

### 3.2 依赖锁定文件

`package-lock.json`、`poetry.lock`、`go.sum` 等锁定文件记录所有依赖的精确版本。其意义是确保不同机器、不同时间安装后的环境完全一致，解决"在我机器上能跑，部署就崩"的根本问题。

### 3.3 语义化版本（Semantic Versioning，SemVer）

版本号遵循 `MAJOR.MINOR.PATCH` 格式：

- **MAJOR**：不兼容的 API 变更，调用方需要修改代码
- **MINOR**：向后兼容的新功能，调用方升级无需改动
- **PATCH**：向后兼容的缺陷修复

例：`2.5.3` 表示主版本 2，添加过 5 批新功能，修复过 3 批 bug。

## 四、CI/CD 流水线

### 4.1 CI vs CD 的区别

**持续集成（Continuous Integration，CI）**：每次代码提交自动触发构建与测试，快速发现代码错误，杜绝"破坏性提交"积累。

**持续交付（Continuous Delivery，CD）**：自动将通过测试的代码部署到 staging 环境，需要人工确认后推送生产。

**持续部署（Continuous Deployment）**：更激进——通过测试后自动部署到生产，无需人工介入。

### 4.2 典型流水线阶段

```
代码提交 → 自动构建 → 单元测试 → 代码扫描（linting/SAST）
         → 部署 staging → 集成测试 → 部署生产
```

每个阶段均可独立失败，失败时立刻通知开发者，阻止问题代码继续流转。

### 4.3 主流工具

| 工具 | 定位 | 适合场景 |
|------|------|---------|
| GitHub Actions | 原生集成 GitHub，YAML 配置，免费额度充足 | GitHub 托管代码的中小团队 |
| GitLab CI | 内置于 GitLab，强大的并行与缓存机制 | 自托管 GitLab 的企业 |
| Jenkins | 开源标杆，高度可定制 | 传统企业，需完全自主控制 |
| CircleCI | 云原生设计，快速反馈，成本透明 | 对构建速度要求高的团队 |

### 4.4 触发机制

- **Push 触发**：任何代码提交即刻运行流水线
- **PR 触发**：拉取请求创建或更新时做预检验证
- **定时触发**：定期跑性能测试、安全扫描、依赖更新检测
- **手动触发**：特殊场景下临时构建或回滚操作

## 五、开发环境与容器化

本地开发直接在操作系统上运行，启动快但存在环境差异问题（操作系统版本、Python/Node 版本、依赖版本各异）。容器化（Docker）将应用及其全部依赖封装成镜像，保证任何环境以完全相同的方式运行。

### 5.1 Docker 核心概念

| 概念 | 含义 |
|------|------|
| 镜像（Image） | 应用的静态模板，包含代码、运行时、库文件、环境变量 |
| 容器（Container） | 镜像的运行实例，一个隔离的进程环境，可启动和销毁 |
| Dockerfile | 描述如何一步步构建镜像的配置文件 |
| Registry | 镜像仓库，如 Docker Hub 或私有仓库，用于存储和分发镜像 |

容器化从根本上解决了"在我机器上能跑"的问题：开发、测试、线上共用同一镜像，消除环境差异。现代云原生工作流中，容器结合 Kubernetes（容器编排平台）已成为大规模服务部署的标准方案。
