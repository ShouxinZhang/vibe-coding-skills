# 前后端分工

---

## 一、什么是前端与后端

将现代应用想象成一家餐厅：**前端是服务员加装修**（用户看得见、摸得着的部分），**后端是厨房**（业务处理的核心），**数据库是食材仓库**（信息的持久存储）。

- **前端**处理渲染、交互、用户体验。代码运行在浏览器中，用户能直接看到界面、点击按钮、输入数据。
- **后端**处理业务逻辑、数据处理、权限验证、第三方集成。代码运行在服务器上，用户无法直接访问。

两者通过 API 通信——前端发起请求，后端处理并返回数据。

## 二、前端技术栈

### 2.1 基础三件套

- **HTML**（超文本标记语言）：定义页面的结构与内容，是文档的骨架
- **CSS**（层叠样式表）：控制外观——颜色、布局、动画
- **JavaScript**：处理交互逻辑，操作页面元素，与后端通信

三者缺一不可：HTML 是肉体，CSS 是衣着，JavaScript 是行为。

### 2.2 前端框架对比

| 框架 | 定位 | 生态规模 | 学习曲线 | 适合场景 |
|------|------|---------|---------|---------|
| React | 视图库 + 生态自由组合 | 极大 | 中等 | 大型复杂应用、需要灵活性 |
| Vue | 渐进式框架，开箱即用 | 中等 | 平缓 | 中小项目、快速原型迭代 |
| Angular | 完整企业级框架 | 中等 | 陡峭 | 大型企业应用、强 TypeScript 场景 |

### 2.3 浏览器运行机制

浏览器解析 HTML 生成 DOM（文档对象模型，Document Object Model）树，解析 CSS 生成样式规则，两者合并为渲染树后完成布局与绘制。JavaScript 通过 DOM API 修改节点，触发重排（Reflow）与重绘（Repaint）。频繁操作 DOM 是前端性能问题的主要来源之一。

### 2.4 构建工具

构建工具负责代码打包（Bundle）、转译（Transpile，将新版 JS 转为兼容旧浏览器的代码）、模块化处理与热更新。

- **Vite**：基于 ES Module 的新一代工具，开发服务器启动近乎即时，热更新极快
- **Webpack**：历史最悠久，生态成熟，配置灵活，适合复杂定制场景

### 2.5 移动端扩展

- **React Native**：用 JavaScript/React 开发原生 iOS/Android 应用，代码复用率高
- **Flutter**：Google 出品，使用 Dart 语言，渲染性能更优，但生态相对较小

## 三、后端技术栈

### 3.1 常见语言与框架

| 语言 | 主流框架 | 性能 | 适用领域 | 学习难度 |
|------|---------|------|---------|---------|
| Node.js | Express / Fastify | 中高 | I/O 密集、实时应用 | 低 |
| Python | Django / FastAPI | 中 | Web 应用、数据处理 | 低 |
| Java | Spring Boot | 中高 | 企业级系统、金融 | 中 |
| Go | Gin / Echo | 高 | 微服务、高并发 | 中 |
| Rust | Axum / Actix | 极高 | 系统级、性能敏感 | 高 |

### 3.2 Web 服务器

应用服务器前通常放一层 Web 服务器（如 Nginx）充当反向代理（Reverse Proxy）：

- 将外部请求分发给后端多个实例（负载均衡）
- 直接处理静态文件，不占用应用服务器资源
- SSL/TLS 终止，统一处理 HTTPS

Nginx 以低资源占用和高并发处理能力著称，Apache 功能更全但性能略低。

### 3.3 数据库选型

| 类型 | 代表产品 | 适合场景 |
|------|---------|---------|
| 关系型（SQL） | PostgreSQL、MySQL | 结构化数据、事务需求强、复杂查询 |
| 文档型（NoSQL） | MongoDB | 半结构化数据、快速迭代、灵活 Schema |
| 键值缓存 | Redis | 会话存储、热数据缓存、排行榜 |

大多数应用混合使用：关系型库存核心业务数据，Redis 加速高频读取。

## 四、架构模式演进

### 4.1 四大模式对比

| 模式 | 说明 | SEO | 首屏性能 | 开发复杂度 | 适合场景 |
|------|------|-----|---------|----------|---------|
| MPA（多页应用） | 服务端渲染，每次导航请求完整 HTML | 优秀 | 中等 | 低 | 内容型网站、SEO 敏感型 |
| SPA（单页应用） | 前端路由，后端仅提供数据 API | 差 | 低（需等待 JS 加载） | 中 | 后台系统、内部工具 |
| SSR（服务端渲染） | 首屏服务端渲染，后续前端接管 | 优秀 | 优秀 | 高 | 需要兼顾 SEO 与交互性 |
| SSG（静态站点生成） | 构建时预生成静态 HTML | 优秀 | 极优 | 中 | 博客、文档、营销页面 |

Next.js（React）与 Nuxt.js（Vue）同时支持 SSR 与 SSG，是当下最流行的全栈框架。

### 4.2 BFF（Backend for Frontend）

微服务架构下，后端被拆分成数十个独立服务。前端若直接调用每个微服务，协调成本极高。BFF 是专为某个前端（如 Web 端或 App 端）定制的聚合层：

- 合并多个微服务的数据，一次返回前端所需的全部内容
- 根据设备适配数据格式（PC 端与移动端需要的字段不同）
- 后端微服务只关心业务，前端只关心 BFF 的接口

## 五、前后端协作机制

### 5.1 API 契约

前后端通过 OpenAPI（Swagger）标准定义接口文档，作为协作契约。文档机器可读，可自动生成客户端 SDK，确保前后端并行开发时接口理解一致。

### 5.2 跨域问题（CORS）

浏览器的同源策略（Same-Origin Policy）禁止 A 域名的页面向 B 域名发送请求（防止跨站攻击）。前后端分离部署时（前端 3000 端口，后端 8000 端口算作不同源），后端需在响应头中设置 CORS（跨域资源共享，Cross-Origin Resource Sharing）头部，允许特定来源。

### 5.3 身份认证方案对比

| 方案 | 状态存储 | 扩展性 | 适用场景 |
|------|---------|--------|---------|
| Cookie / Session | 服务端存 Session，Cookie 存 ID | 差（多台服务器需同步 Session） | 传统 Web 应用，单服务器 |
| JWT Token | 客户端存 Token，无状态 | 优 | 微服务、移动端、分布式系统 |

JWT（JSON Web Token）的工作原理：服务端签发一个包含用户信息的加密 Token 给客户端，客户端每次请求携带此 Token。服务端验证签名即可确认身份，无需查询数据库。

### 5.4 本地开发的代理配置

本地开发时，前端运行在 `localhost:3000`，后端在 `localhost:8000`，属于不同源。解决方法是在前端开发服务器配置代理（Proxy）：将 `/api/*` 的请求转发到后端，对浏览器来说请求始终发向同一域，绕过同源限制。生产环境则由 Nginx 统一处理，原理相同。
