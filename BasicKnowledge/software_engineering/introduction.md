## 为什么需要软件工程学？

传统的"一个人 + 一台电脑 + 一个编辑器"的开发模式在现代已完全不可行。现代软件系统复杂度指数级上升，团队规模不断扩大，产品迭代周期急剧缩短。软件工程学的核心问题从未改变：**如何在多人协作下稳定、高效、可维护地交付软件**。

这不仅是技术问题，更是生产力问题。在工程规范上的微小改进（如减少 Code Review 时间、提高测试覆盖率），在大型团队中可能意味着数百万的成本节省或市场机会的获得。

## 历史脉络与演进阶段

| 时间 | 关键事件 | 工程启示 |
|------|---------|---------|
| 1960s | **软件危机出现** —— 项目严重超期、超预算、交付质量低下 | 意识到"软件也是工程"——需要流程、规范、工具 |
| 1970s | **结构化编程** —— Dijkstra 等人提出模块化、分层设计原则 | 代码不再是艺术，开始成为可管理的工程产物 |
| 1990s | **互联网革命** —— 客户端-服务器架构普遍采用，远程协作成为常态 | 引入分布式系统问题；API 设计变得至关重要 |
| 2000s | **敏捷方法论** —— Scrum、XP 流行；开源运动兴盛（Linux、Apache） | 打破"瀑布流"僵化，拥抱变化；版本控制成基础设施 |
| 2010s | **云原生浪潮** —— 微服务、容器化（Docker）、Kubernetes 横空出世 | 从"买硬件"到"租计算资源"；DevOps 融合开发与运维 |
| 2020s | **AI 辅助开发** —— GitHub Copilot、Claude 等大语言模型编码助手问世 | 代码生成自动化；工程关键从"写代码"转向"架构设计与代码审查" |

## 核心层次

软件工程在宏观上可分为三个层次，逐层深化：

**1. 工具层与流程层**（生产力载体）
- 版本控制系统（Git）、构建工具、测试框架
- 持续集成（CI）/持续交付（CD）流水线自动化
- 参考：[ecosystem.md](./ecosystem.md)

**2. 架构层**（系统稳定性来源）
- 前后端分工、API 通信策略、模块解耦方案
- 从单体应用到微服务的拆分策略
- 参考：[frontend_and_backend.md](./frontend_and_backend.md)、[api.md](./api.md)

**3. 基础设施层**（成本与扩展性平衡）
- 云服务（IaaS、PaaS、SaaS）的选择与权衡
- 参考：[cloud_services.md](./cloud_services.md)

## 推荐学习路线

| 读者群体 | 阅读顺序 | 核心目标 |
|---------|---------|---------|
| 初级开发者/学生 | ecosystem → frontend_and_backend → api → cloud_services | 建立全景认知，了解现代开发流程 |
| 技术主管/架构师 | api → frontend_and_backend → cloud_services → ecosystem | 重点掌握架构模式，再评估工具选型 |
| 创业 CTO | cloud_services → ecosystem → frontend_and_backend → api | 优先关注成本与快速迭代，再做架构规划 |

**最小前置知识**：熟悉至少一种编程语言，理解代码执行的基本概念。随着项目规模增长，软件工程规范会快速成为生产力瓶颈，越早掌握越好。
