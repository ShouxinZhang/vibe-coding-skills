# 术语与层级对照

这份文件不是正式论证，而是给后面所有 DFS 扩写先立一套共用坐标。

如果这里不先把词拆开，后面很容易出现这种混写：

- 一边在讨论抖音这样的内容平台
- 一边又跳到微信 / QQ 这种关系协议
- 然后突然开始谈“数据主权”
- 最后再把 agent、身份、治理、分层塞进同一句话

这样写会让判断看起来很猛，但其实是在跨层说话。

所以这份文件只做一件事：**把后面反复出现的高频词，放回它们各自应该待的位置。**

## 一张总图：先分层，再谈变化

当前这条 DFS 主线，最稳的基础不是更多观点，而是先固定住四个层级，再加一个横切视角。

| 层级 | 这一层在处理什么 | 典型对象 | 真正关心的问题 |
|------|------------------|----------|----------------|
| 内容生产与消费层 | 内容如何被生产、分发、消费、反馈 | 抖音、小红书、B 站、数字人内容流 | 哪些内容职能会先被 agent 工作流化 |
| 社会关系与身份层 | 人与人如何连接、识别、拉群、维持长期关系 | 微信、QQ、通讯录、群关系、好友链 | 平台到底是在卖消息界面，还是在托管关系协议 |
| 数据存储与权属层 | 什么数据归谁、能否导出、迁移、继续调用 | 聊天记录、联系人、行为数据、用户工作区 | 用户到底能带走什么，什么只是平台内可见不可迁移的资产 |
| 治理与合规层 | 哪些能力允许被放开、复制、自动化、跨平台流动 | 实名、备案、风控、审计、平台责任 | 哪些结构变化在现实制度下能成立，哪些只能停留在概念上 |
| 横切视角：agent 角色 | agent 到底只是工具，还是人的社会接口扩展 | OpenClaw、个人 agent、数字分身 | agent 改的是效率层，还是人作为社会节点的存在方式 |

后面凡是写一句判断，都应该先问一句：**这句话在讲哪一层？**

如果一句话同时跨三层，通常就说明这句话还没拆干净。

## 高频术语对照表

| 术语 | 应放在哪一层 | 这里的最小定义 | 最容易混淆成什么 |
|------|--------------|----------------|------------------|
| 内容平台 | 内容生产与消费层 | 帮人生产、分发、消费内容的系统 | 被混成所有互联网平台 |
| 关系平台 | 社会关系与身份层 | 托管身份、联系人、群组、社会连接的系统 | 被当成普通聊天工具 |
| 数据主权 | 数据存储与权属层 | 用户对自身数据的访问、导出、迁移与再使用权 | 被说得过大，最后什么都能往里装 |
| 通信历史主权 | 数据存储与权属层 | 用户对聊天记录、互动脉络、上下文历史的持续调用权 | 被误写成“普通文件导出权” |
| agent 作为工具 | 横切视角：agent 角色 | agent 帮人执行任务，但不替代人的社会身份 | 被误写成 agent 已经拥有独立主体资格 |
| agent 作为社会接口 | 横切视角：agent 角色 + 社会关系层 | agent 开始替人维持联系、筛选消息、参与对外协商 | 被直接跳写成“agent 就是另一个人” |
| 用户工作区 | 数据存储与权属层 | 用户可直接配置、积累、迁移、复用的个人生产与记忆环境 | 被误等同于平台资产库 |
| 平台关系资产 | 社会关系与身份层 + 数据权属层 | 平台内部沉淀的关系图谱、分发入口、行为信号与信任结构 | 被误以为只要导出数据就能完整带走 |
| 分层 | 治理与合规层 | agent 能力和配置权不均造成的社会差距扩张 | 被写成纯情绪判断 |
| 治理 | 治理与合规层 | 对身份、权限、自动化行为、平台责任的制度安排 | 被误写成技术系统自然会完成的事情 |

## 最关键的几组边界

## 1. 内容平台 vs 关系平台

这两个词最容易被混在一起，但它们服务的不是同一种核心需求。

| 对比项 | 内容平台 | 关系平台 |
|--------|----------|----------|
| 核心价值 | 帮你看到内容、发出内容、拿到反馈 | 帮你维持人际连接、身份入口和持续通信 |
| 用户来的主要目的 | 看、发、搜、被推荐 | 找人、说话、建群、维持关系 |
| 更容易被改写的地方 | 生产流程、消费入口、内容编排 | 关系管理、权限入口、代理沟通 |
| 更不容易被替代的地方 | 分发规模、版权、商业撮合、治理 | 社会信任、真实身份、群体组织、制度兼容 |

一句更实用的话是：

- 内容平台更像“流”
- 关系平台更像“网”

agent 对“流”的改写会来得更快，对“网”的改写会更慢，但后果可能更深。

## 2. 数据主权 vs 通信历史主权

“数据主权”太大，容易把讨论写散。

如果这里只是想讨论用户真正想夺回什么，那么更尖锐的词不是“所有数据”，而是“通信历史主权”。

| 词 | 为什么不够用 / 为什么更有用 |
|----|-----------------------------|
| 数据主权 | 太宽。文件、照片、浏览记录、联系人、聊天上下文都能塞进去，结果每次讨论的对象都不一样 |
| 通信历史主权 | 更具体。它直接指向“谁能继续调用我与他人的互动历史、关系脉络和上下文” |

这组区分的业务意义很直接：

- 如果用户只拿回静态数据，很多关系价值并不会跟着走
- 如果用户能带走可继续工作的通信历史，平台锁定力才会被真正挑战

## 3. 用户工作区 vs 平台关系资产

这个边界如果不立住，后面很容易把“我能导出一些数据”误写成“我已经拿回整套能力”。

| 对象 | 属于谁更关键 | 为什么重要 |
|------|--------------|------------|
| 用户工作区 | 用户 | 这是个人 agent 持续工作的底座，包含偏好、任务流、记忆、私有上下文 |
| 平台关系资产 | 平台 | 这是平台长期积累出来的真实社交图谱、信任机制、行为信号与组织接口 |

所以真实问题不是“能不能导出”，而是：

**用户到底能带走哪一部分能力，平台又把哪一部分继续锁在自己的关系层里。**

## 4. agent 作为工具 vs agent 作为社会接口

这是整条 DFS 线最容易被写飞的一条边界。

| 视角 | 最稳的说法 | 风险 |
|------|------------|------|
| agent 作为工具 | agent 帮你写、搜、整理、回复、执行工作流 | 容易低估 agent 对关系层的渗透 |
| agent 作为社会接口 | agent 开始替你筛消息、接触别人、维护群关系、代表你发起互动 | 容易过早跳到“agent 已经是独立主体” |

更稳的写法不是二选一，而是承认这是一个连续谱：

1. 先是执行工具
2. 再变成个人工作台入口
3. 再变成对外沟通接口
4. 最后才可能触碰主体资格问题

这样写，主线会更稳，不会一上来就冲到高假设区。

## 常见混写清单

下面这些句子都不是完全错，而是层级没有拆开。

### 混写 1：数据透明了，所以平台会消亡

问题在于：

- “数据透明”属于数据权属层
- “平台会不会消亡”是平台职能重排问题
- 中间还缺了工作流化、迁移成本、关系锁定、治理约束这些桥梁

更稳的写法应该是：

**即使数据可见性提高，平台也未必消失；真正先发生的，往往是平台部分职能被迁移到用户侧工作流。**

### 混写 2：微信就是聊天工具，所以也会像内容平台一样被替代

问题在于：

- “聊天工具”只描述了表层交互
- 微信 / QQ 的关键价值更接近身份入口、关系托管和社会协议

更稳的写法应该是：

**关系平台不是不能变，而是它被改写的难点不在消息 UI，而在社会关系与制度接口。**

### 混写 3：有了个人 agent，用户就拿回了数据主权

问题在于：

- agent 能力不等于权属变化
- 本地有工具，不代表平台愿意开放迁移接口
- 能保存一些记录，不等于能完整复用通信历史

更稳的写法应该是：

**个人 agent 会先增强用户侧工作区，但是否形成真正的主权提升，还要看平台关系资产能否被迁移或重建。**

### 混写 4：agent 会加剧分层

这句话方向可能对，但还不够。

还必须继续追问：

- 是谁先获得更强的配置权？
- 谁能雇佣更强的 agent 系统？
- 哪些组织能积累更厚的通信历史和工作流资产？
- 现实治理会允许差距扩到什么程度？

也就是说，分层不是结论词，而是机制问题。

## 后面写作时的最低检查标准

以后无论补哪一块，先过这四个问题：

1. 我这段话在讲哪一层？
2. 这段话讨论的是平台类型、资产权属，还是 agent 角色？
3. 我是不是把近中期变化和高假设远景写在了一起？
4. 我有没有把“感觉会这样”写成“机制已经解释清楚”？

如果这四个问题过不去，就先回到这份文件重新拆词，再继续往下写。
