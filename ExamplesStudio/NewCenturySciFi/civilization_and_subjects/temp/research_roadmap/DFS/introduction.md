# DFS

DFS 这条线负责深挖主线。

它不是让你一次性把所有结论都写死，而是把大问题按逻辑顺序拆开，让每一块都能单独长大。你可以顺着读，也可以只挑一块先补。

## 为什么要有 DFS

原来的单文件路线图最大的问题，不是观点不够，而是所有观点挤在一起：

- 内容平台和关系平台混在一起
- 数据主权和通信历史主权混在一起
- 主线推演和远景脑洞混在一起

DFS 的作用，就是把这些层分开，让后面每次扩写都知道自己到底在补哪一层。

## 七个深挖方向

1. `01_scope_and_boundaries`
   先把词讲清楚。不然所有后续讨论都会在概念上打架。
2. `02_content_platform_shift`
   专门讨论内容平台为什么更早被 agent 工作流化。
3. `03_relation_platform_protocol`
   专门讨论微信 / QQ 这类关系平台为什么更像协议层，而不是简单消息 UI。
4. `04_agent_social_interface`
   专门讨论人和 agent 之间的社交接口，人怎么从单点账号变成多层代理组织。
5. `05_communication_history_sovereignty`
   专门讨论“通信历史主权”这个更具体、更尖锐的问题。
6. `06_stratification_and_governance`
   专门讨论分层、配置权、实名、备案和治理边界。
7. `07_high_assumption_extensions`
   专门收纳具身化、意识跳跃、元宇宙等高假设支线，避免它们冲坏主线。

## 怎么补这条线

每个目录里的 `notes.md` 都用同一套结构：

- 这一块在讨论什么
- 为什么值得单独拆出来
- 关键问题有哪些
- 当前直觉是什么
- 后续怎么扩写

这套格式的好处是，你不需要等“想清楚整篇文章”才动笔。哪怕你今天只多想明白一个问题，也能把它塞进对应文件里。

## 推荐顺序

如果你想从稳一点的顺序开始，建议这样走：

1. 先看边界
2. 再分清内容平台和关系平台
3. 再谈 agent 的社交意义
4. 再进入主权和治理
5. 最后才碰高假设

这样写，文章会更像推演，不像跳跃式发散。
