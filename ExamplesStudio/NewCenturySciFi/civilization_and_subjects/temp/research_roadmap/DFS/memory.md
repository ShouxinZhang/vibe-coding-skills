# DFS Memory

这份 `memory.md` 不是正式正文，也不是新的大纲。

它的作用只有一个：**把这条 DFS 线之前已经发生过什么、为什么会拆成现在这样、目前真正想讨论的是什么，先用人话记下来。**

因为现在各个 `notes.md` 都比较短，更像“议题卡片”，适合继续补，但如果不先看上下文，很容易不知道这套结构是怎么来的。

## 先说结论：这条线原本不是这样长出来的

一开始这里不是一个目录，而是一个单文件 `research_roadmap.md`。

那个版本的问题不是没内容，恰恰相反，是内容太挤了：

- 一会儿在说 OpenClaw 是什么
- 一会儿在说抖音会不会被干掉
- 一会儿又跳到微信 / QQ 到底是不是关系协议
- 然后又开始谈 agent 社交、社会分层、治理、机器人替身、意识跳跃、元宇宙

读的时候会感觉脑子很热闹，但层次不稳定。很多想法是好的，只是没有排好队。

所以后来才把它拆成 `research_roadmap/` 目录，分成两条线：

- `BFS.md`：负责“先把好聊的话题摊开”
- `DFS/`：负责“把主线按逻辑顺序拆开，慢慢深挖”

现在你看到的 `DFS`，本质上就是那份旧路线图被拆开后的深挖版本。

## 这条 DFS 线到底想解决什么问题

它不是在研究一个具体产品，也不是在写某个平台的行业报告。

它真正想抓的是这个大问题：

**如果 agent 系统越来越强，人类今天的内容平台、关系平台、数据所有权、社交方式和治理结构，会怎么被重新组织？**

更具体一点，这里其实一直在围绕五个主问题打转：

1. 内容平台会先怎么变  
   例如抖音这类平台，会不会先被工作流化，而不是直接消失？

2. 关系平台会怎么变  
   例如微信 / QQ 这种东西，到底是消息界面，还是一种更深的社会连接协议？

3. 人会怎么变  
   也就是，人是不是会从“单点账号”慢慢变成“带多个 agent 的微型组织”？

4. 数据主权会怎么变  
   真正值钱的，是普通数据，还是“通信历史”这种能继续工作的上下文？

5. 治理和分层会怎么变  
   agent 会不会放大差距？如果会，是通过什么机制放大的？中国语境下又会被怎样限制？

## 为什么现在的 notes 看起来都偏短

因为它们本来就不是“已经写好的文章”，而是“适合继续往下长的议题卡”。

当时拆目录的时候，优先解决的是结构问题，而不是一次把每个主题写满。于是每个 `notes.md` 都先统一成了五段：

- 这一块在讨论什么
- 为什么值得拆出来
- 关键问题
- 当前直觉
- 后续怎么扩写

这套格式的优点是很轻，任何时候都能继续补。

但缺点也很明显：**如果没有这份 memory，当你第一次进来时，容易只看到“接下来要做什么”，却不知道“前面为什么会走到这里”。**

所以现在需要这份文件来补回“前情提要”。

## 这条线背后的原始素材有哪些

目前这条 DFS 线，主要是从下面几份材料里长出来的：

- `temp.md`
  最原始的发散草稿。优点是想象力足、问题密度高；缺点是跳跃很快，层次经常混在一起。

- `chatgpt.md`
  一版中间收敛稿。它最大的贡献是把一些关键判断钉住了，比如：
  - 社交数据不是普通文件，而是关系数据
  - 微信 / QQ 更像关系协议，而不是简单聊天工具
  - agent 更可能先改交互层，而不是立刻改关系归属层
  - 真正值得抓的词，可能不是“数据主权”，而是“通信历史主权”

- `Claude_debug.md`
  这份材料最像结构批注。它没有给很多新观点，但很准确地指出了旧草稿的问题在哪里，比如：
  - “数据透明”直接跳成“平台消亡”，中间逻辑断了
  - 内容平台和关系平台混着讲
  - 强假设来得太早
  - 马太效应只是感觉，没有机制解释

可以简单理解成：

- `temp.md` 给原料
- `chatgpt.md` 给修正
- `Claude_debug.md` 给结构批评

而 `DFS/` 是把这三者压成一个更适合长期维护的工作区。

## 现在这七个目录，分别是在替旧草稿解决什么问题

### 01_scope_and_boundaries

这是在替旧草稿解决“概念混写”的问题。

先把内容平台、关系平台、数据主权、通信历史主权、治理层这些词分开，不然后面每一段都在跨层说话。

### 02_content_platform_shift

这是在替旧草稿解决“为什么老拿抖音举例，但没有把抖音这类平台单独讲清楚”的问题。

核心判断是：内容平台更可能先被工作流化，而不是先消失。

### 03_relation_platform_protocol

这是在替旧草稿解决“微信 / QQ 到底是什么”这个问题。

重点不是它们是不是聊天工具，而是它们是不是一种社会连接协议。

### 04_agent_social_interface

这是在把旧草稿里最有劲的一条线单独拉出来：

**agent 的社交意义，不只是替人做事，而是重写‘人如何作为一个社会接口存在’。**

### 05_communication_history_sovereignty

这是在把“数据主权”这个太大的词，收缩成一个更尖锐、更贴近原问题的词：

**通信历史主权。**

也就是，你真正想拿回的，也许不是所有数据，而是自己那段能继续工作的历史上下文。

### 06_stratification_and_governance

这是在处理旧草稿里最容易黑化、最容易一句话带过的部分。

如果要说 agent 会加剧分层，那就得解释：

- 差距是怎么来的
- 谁掌握更强的系统配置权
- 现实治理会怎么卡住这些差距

### 07_high_assumption_extensions

这是一个“缓冲区”。

旧草稿里关于机器人替身、意识跳跃、元宇宙、多时空存在的内容不是没价值，而是太容易冲坏主线。所以现在把它们先关在这里，等主线稳了再放出来。

## 目前这条线最真实的状态

现在这条 DFS 线，已经不是“完全没结构”的状态了，但也绝对不是“已经写完”的状态。

更准确地说，它现在处于：

**主问题已经拆出来了，但每一块都还只是第一层厚度。**

也就是说：

- 现在已经知道该分成哪几块
- 也知道每块为什么存在
- 也知道每块大概要问什么
- 但每块还没进入真正的长篇论证

所以不要把这些 `notes.md` 当成成品，更应该把它们看成：

**一组已经排好队、但还没长肉的讨论接口。**

## 接下来最合理的维护方式

我建议以后这样维护：

1. `memory.md` 只负责记“上下文”和“前情提要”
   不在这里写太多新论证，重点是让后来的人一进来就知道发生过什么。

2. `introduction.md` 只负责给 DFS 导航
   它像地图，不像仓库日志。

3. 每个 `notes.md` 只负责自己那一块
   不再承担整条主线的背景说明。

4. 如果某个 `notes.md` 逐渐写厚了，再考虑把它升级成正式文章
   这样整条线会比较稳，不会再次退回“所有东西塞进一个文件”的状态。

## 如果你现在要继续写，建议先怎么走

如果你刚回到这个目录，不知道先补哪块，我建议这样：

1. 先看这份 `memory.md`
   先把前情提要补回来。

2. 再看 `introduction.md`
   知道七个块各自是干什么的。

3. 然后只选一个 `notes.md` 往下扩
   不要一上来又想把整条线一起重写。

## 一句话总结

这条 DFS 线不是从“完整研究设计”长出来的，而是从一份**想法很多、跳跃也很多**的旧草稿里，一边 debug 一边拆出来的。

所以现在最需要的，不是再发明一个更复杂的大纲，而是先把这条线的前情提要保存好，让后面的扩写有连续记忆。
