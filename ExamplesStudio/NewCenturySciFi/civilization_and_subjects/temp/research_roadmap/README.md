# research_roadmap

这个目录不再把路线图压成一个大文件，而是拆成两条并行路径：

- `DFS/`：深挖主线，把大问题一块块拆开，适合慢慢补、慢慢吵、慢慢长成正式文章。
- `BFS.md`：口语化问题池，把能直接开聊的小话题先铺开，避免每次都要先读完一整套严密结构。

这样改的目的很简单：原来的 `research_roadmap.md` 虽然完整，但太像一次性报告，不像一个可以长期工作的讨论区。现在把它改成目录后，后续只要补一个局部文件，就能推进整个主题。

## 怎么用

如果你现在想继续搭主线，先看 [DFS/introduction.md](./DFS/introduction.md)。

如果你现在只想快速挑一个话头开聊，先看 [BFS.md](./BFS.md)。

如果你已经知道自己想补哪一块，可以直接进入对应的 DFS 子目录：

- [01_scope_and_boundaries](./DFS/01_scope_and_boundaries/notes.md)
- [02_content_platform_shift](./DFS/02_content_platform_shift/notes.md)
- [03_relation_platform_protocol](./DFS/03_relation_platform_protocol/notes.md)
- [04_agent_social_interface](./DFS/04_agent_social_interface/notes.md)
- [05_communication_history_sovereignty](./DFS/05_communication_history_sovereignty/notes.md)
- [06_stratification_and_governance](./DFS/06_stratification_and_governance/notes.md)
- [07_high_assumption_extensions](./DFS/07_high_assumption_extensions/notes.md)

## 这套结构解决什么问题

- 把“内容平台会怎样”和“关系平台会怎样”拆开，不再混写。
- 把“数据主权”进一步拆成“通信历史主权”，不再一句话包打天下。
- 把高假设支线单独收起来，避免主线一路写着写着就飘到元宇宙和意识跳跃。
- 给后续扩写留足接口，不需要每次都重写总纲。

## 目录关系

- `DFS` 建骨架，负责把论证顺序、边界和主问题立住。
- `BFS` 提供话头，负责让讨论更像聊天，而不是像提交论文。
- 两者不是重复，而是互补。
